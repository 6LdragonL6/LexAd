# LexAd v0.7.0 演示与评测指南

本文用于现场演示、仓库复现和书面评审。演示重点是完整能力链和企业协作价值，不依赖某一条固定广告案例。

## 1. 演示目标

评估人员应在一次短演示中理解：

1. LexAd 解决的是企业广告发布前的跨部门合规协作问题；
2. 系统不是关键词直接判定工具；
3. AI 优先使用法规、平台规则和案例资料库；
4. 已确认风险、资料核验和人工复核具有不同含义；
5. 结果能够进入法务决定、修改重提和版本追溯闭环。

## 2. 演示前检查

### 环境

- 后端和前端已启动；
- <http://localhost:5173> 可以访问；
- <http://localhost:8000/docs> 可以访问；
- 数据库迁移位于最新版本；
- 管理员 AI 配置验证成功；
- 所演示平台具有当前生效的结构化规则版本；
- 至少存在一个已发布舆情资料库版本。

### 账号

准备市场、法务和管理员三种本地演示账号。不要在公开录屏或对外材料中使用正式环境账号和密码。

### 输入材料

建议准备经过脱敏的通用演示文案，覆盖以下能力即可：

- 语境正常、应排除机械候选的表达；
- 包含数据、来源、认证或资质，需要核验材料的表达；
- 能够由现有法规或平台资料明确支持的风险表达；
- 适合说明舆情判断与法律判断差异的完整文案。

不需要把任何单一案例包装成系统全部能力，也不要使用无法公开的企业材料。

## 3. 推荐演示节奏

### 第一段：问题与定位（约 1 分钟）

说明企业广告审查的实际困难：规则分散、人工检索耗时、关键词误报、多平台重复工作和跨部门记录断裂。

一句话定位：

> LexAd 让规则负责找候选，让 AI 在资料库约束下结合语境判断，再由系统验证证据和依据，最终把任务交给合适的人处理。

### 第二段：市场提交（约 2 分钟）

1. 使用市场账号登录；
2. 进入“提交广告物料”；
3. 粘贴完整文案或上传文件；
4. 展示文本预览、行业多选、投放平台和品牌关联；
5. 发起审查。

讲解重点：行业和平台不是装饰字段，它们决定资料检索与平台规则适用范围；文件提取后仍需要用户核对原文。

### 第三段：解释审查结果（约 3 分钟）

按以下顺序讲解：

1. 法律合规和舆情是两条独立风险轴；
2. 已确认风险具有完整原文、中文等级、判断理由、资料依据和建议；
3. 资料核验事项用于处理数据、来源、认证、资质和证明材料；
4. 页面不展示内部关键词、数字切片和相似度作为结论理由；
5. 资料不足或 AI 不可用时会明确要求人工复核。

讲解重点：法规合规分和舆情安全分均为 0–100 且越高越好；无可靠舆情自动结论时显示待人工复核，不用 0 分代替。

### 第四段：法务闭环（约 2 分钟）

1. 切换法务账号；
2. 打开待审队列和详情；
3. 结合原文、依据、核验事项和舆情提示作出决定；
4. 展示通过、有条件通过或退回所代表的流程；
5. 说明退回后重新提交会生成新版本并保留历史记录。

### 第五段：资料治理（约 2 分钟）

使用管理员账号展示：

- AI 配置只显示密钥掩码；
- 舆情事件从草稿、AI 整理、人工复核到发布；
- 平台规则按版本、来源和有效期管理；
- 导入记录、操作日志和回收站；
- 缺少生效平台规则时，系统不会假装审查通过。

## 4. 结果阅读检查表

演示任何一条物料时，都可以用以下问题检查系统输出：

| 检查问题 | 合格表现 |
| --- | --- |
| 是否把内部词条当成结论 | 不展示原始关键词、数字切片或匹配分 |
| 风险证据是否完整 | 能在原文中逐字定位，且具有语义 |
| 是否具有资料依据 | 引用当前输入资料库中真实存在的标题与版本 |
| 事实待证是否被误判 | 单独进入资料核验事项 |
| 平台规则是否有效 | 使用当前生效版本并记录版本标识 |
| 舆情是否机械匹配 | 由 AI 阅读完整语境，案例只作已验证参考 |
| 异常是否被隐藏 | 明确显示人工复核或不可用原因 |

## 5. 降级能力演示

安全降级是可信设计的一部分，可以选择展示其中一种：

- 临时清除本地演示环境的 AI 配置，再提交测试物料；
- 选择没有生效规则版本的演示平台；
- 使用仅有草稿规则而没有启用版本的测试数据。

预期表现是提示人工复核或资料待补充，而不是直接输出违规，也不是无条件显示低风险。演示结束后恢复原配置。

不要在正式环境故意中断服务，也不要删除无法恢复的资料。

## 6. 工程复现

首次安装与启动见 [本地开发指南](../guides/local-development.md)。核心验证命令：

```bash
cd backend
python -m pytest app/tests -q
```

```bash
cd frontend
npm run build
```

运行后可通过 OpenAPI 页面检查接口契约，通过前端分别验证市场、法务和管理员流程。

## 7. 评测维度

### 法律方法

- 规则与结论是否分离；
- AI 是否受到资料库约束；
- 风险、核验事项和人工复核是否边界清楚；
- 是否明确不替代最终法律判断。

### 技术可信度

- 模型输出是否结构化；
- 原文和引用是否经过程序校验；
- 失败是否安全降级；
- 历史输入和规则版本是否可追溯；
- 自动化测试能否复现。

### 企业价值

- 是否减少无效人工筛查；
- 是否缩短法规与平台规则检索路径；
- 是否支持营销、法务和运营协同；
- 是否便于组织资料持续维护和审计复盘。

### 产品完整度

- 是否覆盖提交、审查、决定、退回和重提闭环；
- 是否具有角色权限、资料管理和异常提示；
- 文档、演示和实际代码是否一致。

## 8. 效率评测方法

如需量化企业效率，建议在同一批脱敏物料、相同资料范围和相同人员条件下记录：

- 从提交到结构化结果的时间；
- 规则候选数量与最终确认问题数量；
- 法务需要人工查看的项目数；
- 形成完整依据说明所需时间；
- 多平台审查覆盖数量；
- 需要补充材料的事项是否被正确分流。

应同时记录样本量、资料版本、模型配置、网络条件和人工判断标准。没有实际采集结果时，只说明评测方法，不把估算写成效率结论。

## 9. 演示边界

- 演示结果不代表对所有行业、平台和地区规则的完全覆盖；
- 模型或资料缺失造成的人工复核是预期安全行为；
- 不使用真实客户、隐私数据、内部合同或未公开争议材料；
- 不宣称零误判、完全自动合规或替代律师；
- 评估人员可从 [审查方法论](../architecture/review-methodology.md) 继续核对技术边界。
