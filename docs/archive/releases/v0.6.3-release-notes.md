# LexAd v0.6.3 发布说明

> 历史版本文档：本文仅记录 v0.6.3 当时的实现，不代表当前审查逻辑。请查看 [v0.7.0 文档中心](../../README.md) 和 [v0.7.0 发布说明](../../releases/v0.7.0-release-notes.md)。

发布日期：2026-07-14

## 重点修复

### 法律第二层模型请求

- 为语义合规审查、舆情案例结构化和舆情风险审查补充明确 JSON 输出契约与示例。
- 校验模型完成状态，对空响应、长度截断和结构异常执行有界重试。
- HTTP 400 不再统一显示成“固定模型不可用”，后台保留脱敏后的任务类别和请求标识。
- 管理员 API 测试仍只负责验证连接；业务稳定性由业务代码和自动化测试保证。

### 真实舆情案例

- 将团队整理的 34 条真实案例纳入 `data/sentiment_cases.json`，包含来源、事实核验与严重度校准。
- 规范化后形成 33 个事件；两条桃李面包月饼记录合并为一个规范事件，保留全部来源和核验信息。
- 启动同步、管理员导入与命令行脚本共用同一规范化服务。
- 停止把 L3 法律执法案例自动当作舆情基线；旧记录保留用于历史追溯，但不进入新资料库快照。

### 本地与 AI 融合

- 触发词只作为弱候选线索，不再通过单词数量直接决定中高风险。
- 本地检索综合原始文案、触发摘要、风险维度、行业、平台、案例核验状态和文本相似性。
- AI 无论本地是否命中都可独立审查完整文案，并返回风险分数、主题、原文证据、反向证据、建议和可信度。
- 系统验证 AI 引用是否存在于原文、案例 ID 是否来自输入候选，再计算 AI 有效可信度。
- 本地证据基础优先级为 60%，AI 为 40%，再按各自有效可信度动态归一化；单侧无证据时不使用零分稀释另一侧。
- 高可信冲突会保留审慎风险提示并要求人工复核。

## 目标场景

“被生活毒打过，才懂这盒月饼的甜……”一类文案能够识别为至少中风险，标记价值观争议和苦难营销，并关联规范化后的桃李面包真实案例。游戏语境中单独出现“毒打”不会据此自动判为中高风险。

## 兼容性

- 无数据库结构变更；现有 SQLite 与 PostgreSQL 模式继续兼容。
- 旧审核记录缺少新舆情字段时，结果页使用兼容空状态。
- 管理员维护的案例不会被内置同步覆盖或删除。
- 历史 v0.5.0 发布说明曾记录“已接入 34 条案例”，但仓库当时未包含案例文件；v0.6.3 完成该数据的实际入库和运行时接入。
