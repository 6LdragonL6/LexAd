# LexAd v0.5.0 稳定发布设计

日期：2026-07-08

## 1. 背景

v0.4.2 已完成管理员资料中心、平台规则版本管理、法律与舆情双轴结果页和多平台提交入口。但测试人员使用市场、法务和管理员账号后暴露出三个实际问题：

1. 平台规则状态始终提示“以下平台暂无生效规则”，多平台选择接入后中文平台名与后台规则标识没有稳定映射。
2. 市场部查看被退回物料时，看不到法务退回原因和内部备注，无法按法务意见修改。
3. L1 用语检测存在系统性缺口。知识源中有“最（含‘最’字的绝对化用语）”这类概括性规则，但运行时只保留有限枚举词，导致“最耐用”漏检；“国家级品质”虽命中“国家级”，报告却没有解释命中逻辑。

同时，0.5.0 需要支持退回物料多次循环提交，并接入团队提供的真实舆情数据。

## 2. 目标

- 修复 v0.4.2 的三项实测 bug，优先保证测试人员能完整走通业务。
- 将退回物料做成可循环流程：修改、递增版本、重新提交、再次法务审核，直到归档或删除结束。
- 让审查报告对市场和法务人员更可读，避免“未命中规则”等偏技术表达造成误解。
- 接入 `D:\LDragonl\桌面\team` 中的舆情案例和触发词，使用现有 `public_opinion` 双轴架构，不新增 L2 `sentiment` 旁路。
- 所有前端业务展示优先使用中文名称，避免直接暴露 `douyin`、`xiaohongshu`、`jd` 等内部标识。

## 3. 非目标

- 不重做现有前后端架构。
- 不新增第二套舆情结果字段。
- 不做实时舆情抓取或平台规则自动爬取。
- 不用单词补丁掩盖 L1 用语检测问题；bug3 必须作为规则编译和检测体系缺口处理。
- 不把舆情风险计入法律合规分。

## 4. 总体方案

0.5.0 采用“修复优先版”：

1. 后端修复平台规则别名匹配、退回循环版本、归档/删除边界、L1 模式规则和舆情资源导入。
2. 前端补齐市场部退回信息展示、二次提交入口、版本标识、归档/删除操作和中文化平台展示。
3. 报告层把技术状态翻译成业务语言，并解释每条高风险命中的依据。
4. 测试层增加 L1 规则覆盖审计、平台规则中文名回归、多轮退回流程和舆情导入验证。

## 5. 数据与状态

### 5.1 物料状态

`MaterialStatus` 增加 `archived`。

- `draft`：未进入审查，可编辑、可删除。
- `ai_reviewing`：AI 审查中，不允许编辑。
- `pending_legal` / `in_legal_review`：等待或正在法务审核，不允许市场编辑。
- `returned`：法务退回，市场可编辑并再次提交。
- `approved`：流程通过，保留历史。
- `archived`：市场部主动结束退回或草稿流程，不再参与待办。

物理删除只允许没有审查历史的草稿。已产生 `Review` 的物料只能归档，避免审查记录失去归属。

### 5.2 版本递增

编辑 `returned` 物料时，如果 `raw_text`、`industry` 或 `platforms` 发生变化，`Material.current_version += 1`。再次发起 AI 审查时，新 `Review.version` 使用当前版本号。

版本列表返回：

- 提交版本号。
- AI 审查状态和风险分。
- 法务决定。
- 退回原因。
- 法务备注。
- 审核时间。

前端显示“第 1 次提交 / 第 2 次提交”，不让用户只看到数据库版本号。

## 6. 平台规则修复

### 6.1 根因

提交页保存中文平台名，如“抖音”“小红书”。管理员平台规则示例和测试数据使用内部标识，如 `douyin`。当前 L4 匹配只比较 `platform_name` 和 `display_name` 的归一化等值，缺少别名层，因此平台规则集存在时仍会被判定为不可用。

### 6.2 设计

新增平台别名归一化：

| 展示名称 | 内部标识候选 |
|---|---|
| 抖音 | `douyin`, `douyin_ec`, `巨量引擎` |
| 小红书 | `xiaohongshu`, `red`, `xhs` |
| 微信 / 视频号 | `wechat`, `weixin`, `shipinhao` |
| 微博 | `weibo` |
| 京东 | `jd`, `jingdong` |
| 淘宝 | `taobao`, `tmall` |
| 拼多多 | `pdd`, `pinduoduo` |

匹配顺序：

1. 将用户选择、规则集 `platform_name`、规则集 `display_name` 都归一成别名集合。
2. 任一别名相交即视为同平台。
3. 结果中保存内部版本 ID，但前端主展示使用中文 `display_name` 和版本标签。

平台规则状态文案改为：

- 有版本：`抖音 / 2026-07 已用于本次审核`
- 无规则集：`小红书：暂无规则集，请管理员补充`
- 无生效版本：`京东：已有规则集，但暂无生效版本`

## 7. L1 用语检测根因修复

### 7.1 问题定位

知识源 `knowledge/L5_templates/forbidden/001_最类绝对化用语.txt` 表达的是概括性规则：“最（含‘最’字的绝对化用语）”。运行时 `data/forbidden_words.json` 只包含“最好、最大、最高、最佳”等枚举词。当前 `layer1_hard_rules.py` 使用 Aho-Corasick 精确词匹配，无法覆盖“最耐用”“最省电”等合理泛化形态。

### 7.2 规则类型

L1 拆成三类规则：

- 精确词规则：现有 `forbidden_words.json` 词条，如“国家级”“最低价”。
- 模式规则：概括性表达，如“最 + 性能/品质/效果/体验描述词”。
- 审计规则：检查知识源概括性规则是否在运行时有对应精确词或模式规则承接。

### 7.3 模式检测

新增“最类绝对化”模式检测：

- 命中结构：`最` + 1 至 6 个中文描述字符。
- 重点描述域：品质、性能、耐用、安全、舒适、省电、省油、先进、专业、强、快、稳、优、纯、全、美、火等。
- 输出解释：`命中模式：最 + 性能/品质描述`。
- 建议：使用可证明、可限定的表达，例如“耐用性表现较好”“经测试具备较好耐用性（注明测试依据）”。

边界控制：

- 不把法规标题、知识库说明、引用文本作为广告主张误判。
- 对“最近”“最终”“最多可申请额度”这类需要上下文判断的表达，优先给出较低风险或要求证明材料，而不是一律高风险。
- 0.5.0 暂不做白名单管理界面；边界词先以代码清单维护，并覆盖测试集中最常见的漏检形态。

### 7.4 报告解释

`MatchedRule` 增加可选解释字段，或通过现有字段提供清晰说明：

- `rule_text`: `最耐用`
- `match_type`: `绝对化用语`
- `source_law`: `广告绝对化用语规则 / 命中模式：最 + 性能品质描述`

“国家级品质”展示为：

- 命中词：`国家级`
- 风险类型：`权威背书/绝对化用语`
- 说明：容易被理解为国家机关、国家级资质或最高等级背书；如确有认证，应注明认证名称、主体、编号和适用范围。

## 8. 退回闭环

### 8.1 市场部

市场部工作台中，对 `returned` 物料展示：

- 最新法务决定。
- 退回原因。
- 法务内部备注。
- 被退回的版本号。
- “修改并重新提交”按钮。
- “归档”按钮。
- 安全可删除时显示“删除草稿”按钮。

修改后重新提交时，页面明确显示“将作为第 N 次提交重新进入审查”。

### 8.2 法务部

法务详情页显示物料版本和历史版本入口。法务只能对当前最新且未决策的 Review 提交决定，不能覆盖旧版本决定。

### 8.3 历史追溯

结果页和版本列表都能查看每次提交的 AI 结果、平台规则版本、舆情资料版本、法务决定、退回原因和备注。

## 9. 报告人性化

将偏技术表达替换为业务表达：

- `未命中硬规则` -> `本层未发现明确违禁词或硬性风险点`
- `未命中平台规则` -> `本次未发现与该平台生效规则直接冲突的表达`
- `unavailable_platforms` -> `需要管理员补充规则的平台`
- `资料库待补充` -> `当前舆情案例库不足，暂不能给出案例化判断`

报告按“结论、原因、建议、依据”组织。默认展示业务信息，技术 ID 放进次级区域。

## 10. 舆情数据接入

团队资源路径：

- `D:\LDragonl\桌面\team\sentiment_cases.json`
- `D:\LDragonl\桌面\team\trigger_words.json`

接入方式：

1. 编写导入脚本或管理员导入 API，将 `events` 转为 `PublicOpinionEvent` 和 `PublicOpinionEventVersion`。
2. 案例默认进入已发布状态，并生成新的 `PublicOpinionLibraryVersion`。
3. `external_id` 用于幂等导入；重复时更新草稿或跳过已发布记录，不能静默覆盖历史。
4. `trigger_words.json` 作为确定性触发词补充，用于 `public_opinion` 引擎的 fast path。
5. 结果页继续使用 `public_opinion_result`，不新增 L2 `sentiment` 字段。

## 11. 前端展示约束

- 业务页面展示中文平台名，不展示拼音或英文内部标识。
- 内部标识只用于管理员调试、导入字段、审计记录或技术详情。
- 版本快照主展示格式为 `抖音 / 2026-07`，不是裸 UUID。
- 页面中的按钮、标签和状态都避免编程化术语。
- 结果页保留“技术详情”折叠区，用于复制版本 ID 和排障。

## 12. API 调整

- `PUT /materials/{id}`：编辑退回物料时触发版本递增。
- `POST /materials/{id}/archive`：归档退回或草稿物料。
- `DELETE /materials/{id}`：仅允许无审查历史的草稿。
- `GET /materials/{id}/versions`：补充每版的法务决定、退回原因、备注和版本状态。
- `GET /reviews/queue`：市场部返回最新 review 的退回信息，法务部仍返回待审队列。
- 管理脚本或管理员 API：导入团队舆情资源并生成资料库版本。

## 13. 测试策略

### 13.1 后端单元测试

- 中文平台名能命中 `douyin` 规则集。
- 有规则集但无 active version 时只提示无生效版本。
- `最耐用`、`最安全`、`最省电`、`国家级品质`、`世界级工艺`、`独家配方` 能给出可解释命中。
- L1 不把明显非广告主张的普通文本误判为高风险。
- `returned` 物料编辑后版本递增，新 Review 使用新版本。
- 已产生 Review 的物料不能物理删除，只能归档。

### 13.2 集成测试

- 市场提交 -> AI 完成 -> 法务退回 -> 市场看到原因和备注 -> 修改 -> 第 2 次提交 -> 法务再次审核。
- 平台规则版本快照展示中文平台名和版本标签。
- 舆情案例导入后，新审核不再显示“资料库待补充”。
- 重复导入同一 `external_id` 不破坏已发布历史。

### 13.3 前端验证

- 市场工作台直接显示退回原因和操作入口。
- 结果页平台状态不出现 `douyin`、`xiaohongshu` 等内部标识。
- 法务详情页非 completed review 不白屏。
- 报告中的“未命中”文案改成业务语言。

## 14. 验收标准

- 测试人员提交中文平台后，不再全部显示“暂无生效规则”。
- 市场部能看到法务退回原因和备注，并完成至少两轮重新提交。
- “最耐用”一类未枚举的最类绝对化表达能够被检测并解释。
- “国家级品质”命中有明确依据和修改建议。
- 前端业务页面不出现拼音式平台内部标识。
- 团队舆情案例导入后，舆情结果能基于真实案例给出判断。
- 自动化测试覆盖平台别名、L1 规则审计、退回循环和舆情导入。
