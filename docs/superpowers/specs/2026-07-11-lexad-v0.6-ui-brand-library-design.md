# LexAd v0.6 UI 与品牌库设计规格

日期：2026-07-11  
状态：已完成讨论，待书面规格复核  
基线：`main` @ `85940b3`（v0.5.1）

## 1. 背景与范围

v0.6 有两个目标：参考 `D:\LDragonl\桌面\0.6target\New_UI.html` 的美术语言升级全部页面并适配手机端；参考品牌库演示与 P0 任务书，实现品牌选择、档案和品牌记忆闭环。

参考 HTML 只约束视觉语言。项目吸收其深色侧边栏、蓝色品牌色、渐变、卡片、圆角、阴影、表单和状态控件风格，但不复制登录页与业务页粘合、桌面比例失衡等结构问题。

### 目标

- 保留现有页面、路由、请求和业务逻辑，进行增量视觉升级。
- 登录界面与登录后应用壳层严格分离。
- 桌面端固定深色侧边栏；窄屏使用不占内容宽度的移动导航。
- 全部页面支持浅色、深色和手机布局。
- 品牌作为可选维度接入提交、结果和法务审核。
- 品牌档案实时聚合现有记录，不复制统计状态。
- 模块保持开放，后续添加菜单、红线和分析时无须重写现有边界。

### 非目标

- 不复制参考 HTML 的 DOM 或页面切换方式，不重写前端。
- 不引入大型 UI 框架。
- 不修改 `backend/app/engine/pipeline.py`，品牌不影响 AI 风险分数。
- P0 不建品牌红线表、统计表、品牌版本表或缓存表。
- v0.6 不实现品牌合并、恢复、永久删除、主数据对接或系统建议红线。
- 不为未来需求创建未使用的空表、空接口或抽象层。

## 2. 方案选择

采用“现有 UI 增量换肤 + 品牌库纵向新增”。先扩展主题令牌与应用壳层，再逐页调整，最后完成品牌库从数据库到页面的闭环。依赖默认不新增；只有轻量、可 tree-shaking 且有明确收益的依赖才可加入，当前设计不依赖新增运行时库。

## 3. 前端架构

### 3.1 登录与业务壳层

```text
/login
└─ AuthLayout
   └─ LoginPage（不加载业务导航和登录后用户控件）

受保护路由
└─ AppShell
   ├─ DesktopSidebar
   ├─ MobileNavigation
   ├─ AppTopBar
   └─ router-view
```

路由守卫认证后才进入 `AppShell`；退出后立即回到独立登录布局。登录页不是业务壳层里的可切换面板。

### 3.2 导航与响应式

- `>= 1024px`：固定深色侧边栏，稳定宽度，不抽屉、不折叠；主内容独立滚动。
- `< 1024px`：侧边栏退出布局，使用顶部品牌栏和覆盖式导航；实现可选侧滑或全屏菜单，但交互契约一致。
- 三栏审核页在窄屏按业务顺序纵向排列。
- 移动导航锁定背景滚动，支持菜单、遮罩和 Escape 关闭；路由变化后关闭并恢复焦点。
- 桌面和移动导航读取同一配置，字段为 `route`、`label`、`icon`、`roles`、`group`、`order`。

### 3.3 主题与组件

`main.css` 集中管理品牌色、渐变、页面/卡片/侧栏背景、文字、边框、状态色、圆角、阴影、间距、控件高度、侧栏宽度和断点。页面不得散落硬编码主题颜色和导航尺寸。

```text
frontend/src/
├─ components/layout/
│  ├─ TopBar.vue
│  ├─ AppSidebar.vue
│  ├─ MobileNavigation.vue
│  └─ ThreeColLayout.vue
├─ components/common/
```

不建立完整的 Base 组件体系。普通控件继续使用 UnoCSS 和少量语义类；只有存在重复业务状态、逻辑或无障碍行为时才抽组件。

### 3.4 页面范围

- `LoginPage`：桌面为渐变品牌区与登录卡片双栏；手机为单列卡片。
- `HomePage`：渐变欢迎区、摘要、最近提交和合规提示。
- `SubmitPage`：卡片化表单，加入品牌选择和记忆卡。
- `ResultPage`：强化摘要、风险、命中与依据，加入不参与评分的品牌提示。
- `LegalDashboard`：桌面高密度列表；手机记录卡片。
- `LegalDetail`：桌面三栏；手机按物料、AI、品牌、法务决定排列。
- `KnowledgePage`：调整导航、搜索、卡片和阅读区，不改知识层级。
- `AdminKnowledgePage`：保留管理逻辑，统一表单、标签页、表格和弹窗；复杂表格可局部横向滚动。
- `BrandArchivePage`：列表、基础信息、指标、高频违规、最近审核和通过物料；法务与管理员可编辑归档。

触控目标接近或大于 44px。手机端必须能完成登录、提交、查看结果和法务审核。

## 4. 品牌数据结构

### 4.1 `brands`

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | `String(36)` | PK | UUID |
| `name` | `String(100)` | NOT NULL, UNIQUE, INDEX | 展示名称 |
| `aliases` | JSON | NOT NULL, default `[]` | 别名列表 |
| `industry` | `String(50)` | NOT NULL, default `""` | 默认行业建议 |
| `description` | Text | NOT NULL, default `""` | 品牌说明 |
| `status` | Enum | NOT NULL, INDEX | `active` / `archived` |
| `created_by_id` | `String(36)` | FK `users.id`, NOT NULL | 创建人 |
| `created_at` | timezone datetime | NOT NULL | 创建时间 |
| `updated_at` | timezone datetime | NOT NULL | 更新时间 |

`materials` 增加 `brand_id: String(36) | null`，外键到 `brands.id` 并建立索引。品牌只归档、不物理删除，旧物料无须回填。

### 4.2 迁移与归一化

迁移从当前实际 Alembic head `a4b7d1c6e825` 接续，而不是来源任务书中过期的 `d8a1f0e3b5c6`。SQLite 必须用 `batch_alter_table` 增加 `brand_id`、索引和外键，并验证 SQLite/PostgreSQL 升降级。

服务层以“去首尾空白、移除内部空白、统一小写”比较名称。名称或别名命中时返回现有档案。数据库 `name` 唯一索引提供最终并发保护；冲突时回滚、重查并返回已有品牌，改名冲突返回 `409`。

## 5. Schema 与 API

```text
BrandCreate: name(1..100), industry(0..50)
BrandUpdate: name?, aliases?, industry?, description?, status?
BrandOut: id, name, aliases, industry, description, status, created_at
BrandProfile:
  brand, total_materials, total_reviews, decided_reviews,
  approved_count, pass_rate, avg_versions,
  top_violations[], recent_reviews[], approved_materials[]
```

`MaterialSubmit` 与 `MaterialOut` 增加 `brand_id: str | None`。通过物料只返回短文本预览。

| 方法 | 路径 | 用途 | 权限 |
|---|---|---|---|
| GET | `/api/v1/brands?search=` | 搜索有效品牌及别名 | 登录用户 |
| POST | `/api/v1/brands` | 输入即建；重复返回已有品牌 | 登录用户 |
| GET | `/api/v1/brands/{id}/profile` | 查看实时档案 | 登录用户 |
| PATCH | `/api/v1/brands/{id}` | 编辑或归档 | 法务、管理员 |

创建响应为 `{ brand, created }`。错误语义：`403` 无权编辑，`404` 不存在，`409` 名称冲突，`422` 输入无效或新物料引用不存在/已归档品牌。

## 6. 聚合口径

- 只统计 `task_status == completed` 的审核。
- `pass_rate = approved / decided_reviews * 100`；`conditional` 不算通过。
- 无已决定审核时 `pass_rate = null`；无物料时 `avg_versions = null`。
- 高频违规解析 `ai_result.layer1` 至 `layer4` 的 `matched_rules`。
- 普通命中权重 1，法务退回中的命中权重 2，取前 10 条。
- 最近审核倒序取 5 条；最近通过物料取 10 条短预览。
- Python 服务层解析 JSON，不使用数据库专属 JSON 运算。
- 品牌数据不传入审核引擎，不改变评分或决定语义。

## 7. 数据流与状态

```text
提交页加载有效品牌
→ 恢复重提交品牌或浏览器上次选择
→ 搜索已有品牌或输入即建
→ 选择后按需获取 Profile 并显示记忆卡
→ 提交可空 brand_id
→ 原 AI 审查照常运行
```

- 品牌加载或 Profile 失败不阻塞提交。
- 品牌行业只作建议，不覆盖用户选择。
- 归档品牌清除浏览器记忆且不能用于新提交。
- 重提交默认继承，用户可更换或清除。
- 结果和法务页按 `brand_id` 获取同一 Profile。
- 法务决定仍只写现有 Review，后续 Profile 实时反映。
- 品牌记忆卡标注“历史参考，不参与本次 AI 风险评分”。
- 所有角色可查看；法务/管理员可编辑归档；归档档案仍可直接查看。

统一状态为局部 `loading`、可行动的 `empty`、可重试 `error`、防重复 `submitting` 和非阻塞 `success`。Profile 按需请求；P0 候选品牌一次加载后前端过滤。不建品牌 Pinia Store、不缓存 Profile、不建统计快照。

## 8. 开放性约束

- 导航使用单一配置，桌面与移动端不维护两套菜单。
- 壳层通过 `router-view` 和插槽承载页面，不枚举业务模块。
- 视觉数值集中在主题令牌，状态徽章集中映射。
- 品牌组件依赖标准类型和 API，不依赖演示品牌名。
- 演示数据只存在于独立、幂等 seed。
- P0 档案聚合与未来红线执行保持独立服务边界。
- 扩展点来自清晰接口，不来自预建空表、空端点或未使用抽象。

## 12. 回滚原则

- UI 按壳层和页面组分批提交，可逐组回退。
- 品牌字段全程可空，回退前端不阻塞既有提交。
- 数据库使用对应 Alembic downgrade；生产降级前备份。
- 品牌功能异常时可隐藏入口，核心审核流程继续运行。
