# LexAd v0.6.1 体验与审查质量加固设计

## 背景

v0.6.0 已完成新版应用壳层、深浅主题、品牌库和本地启动流程，但实际联调仍暴露出六类问题：窗口在中间尺寸连续变化时，侧栏和页面控件可能遮挡或越界；知识目录与运行时资料库脱节，舆情相似点和平台规则命中不足；法务工作台出现属于市场端的“需修改”统计；多次提交后无法直观回看每一轮提交、审查报告和法务结论；工作台合规提醒内容单一；版本号紧跟导航选项，视觉上过于突出。

当前仓库已经包含 `knowledge/L4_platforms/拼多多` 等平台资料，但本地数据库中的平台规则集、平台规则版本、舆情案例和舆情资料库版本均可能为空。运行时 L4 引擎只读取数据库中已经启用的结构化平台规则版本，不直接读取知识目录。因此，本版不仅修正界面表现，还要打通知识文件到运行时审核的接入链路，并改进检索与报告的信息层级。

## 目标

1. 浏览器窗口从 320px 到 1920px 连续改变宽度时，导航、内容和操作控件能够自然重排，不出现侧栏遮挡、控件越界或整页横向滚动。
2. 在不损害内容观感的前提下尽可能保留桌面侧栏，形成完整侧栏、图标侧栏和导航抽屉三级模式。
3. 首次初始化时将内置知识资料幂等接入运行时资料库，同时保护管理员已经维护和启用的版本。
4. 使用确定性规则、中文文本相似性和可选模型解释组成混合检索，提高改写和近义表达的召回率。
5. 将审查报告改成“命中优先”：突出检测出的风险内容，压缩已执行但未命中的模块，同时明确展示资料不可用和执行失败。
6. 按市场、法务等角色展示准确的工作台统计，法务端不再出现“需修改”。
7. 支持从当前页面打开历史版本抽屉，回看每次提交对应的内容快照、审查报告和法务审核结果。
8. 使用可审阅的结构化提醒池轮换工作台合规提示，并将版本号放到导航区域左下角的低干扰位置。

## 非目标

- 不引入向量数据库或以外部大模型作为唯一检索路径。
- 不重建认证、品牌库、审批状态机或用户权限体系。
- 不建设合规提醒的后台编辑页面。
- 不在每次启动时用知识目录覆盖管理员维护的规则。
- 不为缺少提交快照或报告快照的旧记录伪造历史内容。
- 不进行与上述六项问题无关的大规模前端重构。

## 总体方案

v0.6.1 使用一份发布设计、三个可独立实施和回归的轨道：

1. **布局与角色体验**：统一应用壳层、连续响应式重排、角色化统计卡片和低干扰版本显示。
2. **知识与审核价值**：内置知识基线导入、混合检索、平台接入诊断和命中优先报告。
3. **版本与内容复用**：审核历史详情、只读抽屉和结构化合规提醒池。

三条轨道继续使用现有 `Material → Review → Result` 主流程。数据库和接口采用增量兼容方式；审查模块相互隔离，单个模块失败不应清空其他模块的有效结果。

## 1. 应用壳层与连续响应式布局

### 1.1 三级导航模式

应用壳层统一负责导航宽度和主内容占位，避免固定侧栏与手工 `margin-left` 分别维护导致失配。

| 可视区域宽度 | 导航模式 | 行为 |
| --- | --- | --- |
| `>= 1280px` | 完整侧栏 | 显示图标、文字、分组和左下角版本号 |
| `900px–1279px` | 72px 图标侧栏 | 保留桌面导航；悬停或键盘聚焦时显示菜单提示 |
| `< 900px` | 顶部栏与抽屉 | 不占用正文横向空间；版本号位于抽屉底部 |

导航模式由应用壳层统一决定。详情页、表格页或报告页不能自行切换导航模式，以免路由切换时侧栏忽隐忽现。

### 1.2 内容重排规则

- 页面容器、卡片、弹性列和网格子项统一允许收缩，并设置等价于 `min-width: 0` 的约束。
- 工具栏和按钮组空间不足时自动换行；主要操作保持可见，次要说明随容器换行。
- 多栏页面按照可用内容宽度降为双栏或单栏，不用固定宽度把右侧面板推出视口。
- 七列表格、长标签和代码式内容只允许在自身容器内横向滚动，不能扩大页面根容器。
- 页面采用纵向主滚动。不能依赖根级 `overflow-x: hidden/clip` 掩盖实际溢出。
- 可视高度低至 568px 时，导航、抽屉和内容仍可通过纵向滚动完整访问。

验收既检查明确断点，也连续拖动窗口边缘观察断点之间的状态。重点宽度为 320、768、899、900、1024、1279、1280、1440 和 1920px。

### 1.3 版本显示

完整侧栏中的 `v0.6.1` 固定在左下角，与主导航之间使用自动空白分隔，并降低字号和对比度。图标侧栏使用可辨认但不抢眼的紧凑版本文本。抽屉模式把版本号放在抽屉底部，不在主导航选项之后紧邻展示。

## 2. 角色化工作台

统计卡片由角色配置生成，避免同一套固定卡片在不同职责之间复用。

| 角色 | 统计卡片 |
| --- | --- |
| 市场端 | 待处理、需修改、已通过 |
| 法务端 | 待审核、已通过、已退回 |
| 管理员 | 待审核、已通过、已退回（按其可见的全部物料统计） |

每个统计值必须有明确、可测试的状态集合。市场端“需修改”只统计退回并等待市场修改的物料；法务端“已退回”表示法务已经给出退回结论，不代表法务仍需修改。快捷入口也应受现有角色能力约束，避免展示用户无权进入的操作。

## 3. 内置知识基线接入

### 3.1 数据来源与导入边界

v0.6.1 的启动初始化流程执行一次幂等基线检查，使已经存在但知识表为空的 v0.6.0 数据库也能完成升级。受支持的内置来源明确包括 `knowledge/L4_platforms`、`knowledge/L3_cases`、`data/platform_rules.json` 和 `data/case_library.json`；其他静态法规和行业规则继续沿用现有加载路径。导入单元记录：

- 来源类型与相对路径；
- 来源文件哈希；
- 生成的规则或案例数量；
- 导入版本和时间；
- 成功、跳过或失败状态及原因。

平台名称在接入层规范化。至少保证 `拼多多`、`pdd` 和 `pinduoduo` 指向同一平台标识；其他已支持平台沿用同一别名机制。

### 3.2 幂等与管理员优先

- 某平台没有规则集或没有启用版本时，系统创建并启用内置基线版本。
- 某平台已有管理员启用版本时，内置基线导入必须跳过该平台，不覆盖、不回滚、不改变其生效窗口。
- 同一来源哈希已经成功导入时，重复初始化不创建重复版本。
- 已完成基线检查的数据库在后续普通启动中只做轻量幂等判断，不重新解析或改写已接入资料。
- 单个来源解析失败使用独立事务记录，不撤销其他来源的成功导入。
- 初始化结束输出汇总：发现文件数、生成规则数、已启用平台、跳过平台和失败来源。

拼多多是本版强制回归样例：空库初始化后必须存在可解析、已启用且能被 L4 引擎找到的拼多多基线版本。

## 4. 混合检索

### 4.1 三层检索链路

1. **确定性层**：执行法规条款、禁用词、明确平台规则和资料中配置的触发条件。该层结果必须可复现并带精确来源。
2. **相似性层**：对广告文本和知识条目进行中文文本规范化、分词或字符短语处理，召回改写、近义表达和相似案例要点。结果设置最低阈值和 Top-K 上限，避免低相关内容填满报告。
3. **解释层**：现有模型服务只对候选结果排序、归纳风险和生成建议。模型不可用时，前两层仍返回可审计的命中结果。

相似性不能替代强规则。精确命中的法规或平台禁止项始终保留；低于阈值的相似候选不进入读者报告，但可在诊断信息中统计。

### 4.2 命中记录

每个进入报告的命中项至少包含：

- 所属审查层和规则标识；
- 广告中的命中片段或相似依据；
- 风险要点与来源标题；
- 来源版本；
- 命中方式（精确、关键词、相似性或模型归纳）；
- 可解释的分数或置信等级；
- 修改建议。

## 5. 命中优先的执行报告

报告首屏先回答“检测到了什么、为什么有风险、应当如何修改”，而不是平铺所有执行步骤。

- 顶部摘要显示命中数量、主要风险主题和建议优先处理的内容。
- 有命中的模块默认展开；命中卡片突出原文片段、要点、来源和建议。
- 已成功执行但没有命中的模块压缩为单行状态，读者可主动展开查看执行版本和范围。
- 资料未接入、资料库为空和执行失败分别使用独立状态；这些状态不能显示成“未发现风险”或计入安全结论。
- 报告保留规则版本、平台规则版本和资料库版本快照，使历史结果可以按当时数据解释。
- 详细的执行过程继续可访问，但其视觉权重低于实际命中内容。

## 6. 审核历史与只读抽屉

### 6.1 版本摘要

材料版本接口在兼容现有字段的基础上补充审核记录标识、法务状态和历史报告入口。列表标签使用明确语义：

- `第 N 次提交`；
- 已完成法务处理时显示 `第 N 次法务审核`；
- 尚未处理时显示 `尚未法务审核`。

当前版本和历史版本必须有明显区分。版本编号以提交轮次为主；现有一轮提交对应一条审核记录的业务约束不改变。

### 6.2 抽屉内容与交互

市场结果页和法务详情页复用同一个历史抽屉组件。点击历史版本后，主页面仍保持当前审核对象，抽屉只读展示：

- 当次提交名称、正文、行业、平台、类型、优先级和截止时间；
- 当次审查摘要、命中项、执行状态和资料版本；
- 法务结论、退回原因、备注和审核时间；
- 提交、审查完成和法务处理等关键时间。

旧版本抽屉不展示审批、退回、重新提交或修改按钮。关闭按钮、Esc 键、焦点返回和窄屏全宽展示应符合现有可访问性习惯。

旧记录缺少提交或报告快照时，抽屉展示仍存在的字段，并明确提示“该历史记录未保存完整快照”，不使用当前材料或新规则补算。

## 7. 合规提醒资源池

新增可审阅的结构化提醒资源，例如 `data/compliance_tips.json`。每条提醒包括：

- 稳定唯一编号；
- 标题和短正文；
- 来源资料标题或相对路径；
- 适用角色；
- 标签；
- 启用状态。

进入工作台时按当前角色筛选启用提醒并选择一条展示。前端保存上一次展示编号；候选多于一条时避免连续重复。资源为空、格式错误或没有适用角色条目时，显示稳定的内置兜底提醒，不能阻塞工作台其他数据加载。

提醒正文由现有资料整理成完整、短小的提示，不在运行时随机截取法规原文，避免断句或脱离语境。

## 8. 数据流与异常处理

整体数据流为：

`知识文件 → 基线导入/管理员版本 → 启用版本 → 混合检索 → 审查结果快照 → 命中优先报告 → 历史抽屉`

每次提交先保存不可变的提交快照，再执行各审查模块。每个模块独立保存：

- 执行状态：成功且有命中、成功但无命中、资料不可用或执行失败；
- 命中候选、来源和相似依据；
- 实际使用的规则或资料库版本；
- 面向读者的摘要和建议。

一个模块失败不能清空其他模块结果。前端根据显式状态决定展开、折叠或异常提示，不能通过结果数组是否为空猜测模块状态。

接口和数据库变更保持增量兼容。已有管理员权限、审批状态和启用规则语义保持不变；已有记录缺少新字段时使用可解释的空状态。

## 9. 测试与验收

### 9.1 前端布局

- 检查 320、768、899、900、1024、1279、1280、1440 和 1920px 宽度以及最低 568px 可视高度。
- 自动抽查断点之间的多个宽度，并手动连续拖动窗口。
- 验证完整侧栏、图标侧栏和抽屉切换；所有主页面不得被导航遮挡。
- 验证工具栏、报告、多栏详情和七列表格不存在控件越界或整页横向滚动。

### 9.2 角色体验

- 市场端显示待处理、需修改和已通过。
- 法务端显示待审核、已通过和已退回，不出现需修改。
- 管理员统计和快捷入口符合现有权限。

### 9.3 知识与检索

- 空库首次导入成功；重复运行不产生重复版本。
- 管理员已有启用版本不被基线覆盖。
- 单个坏文件被记录，其他平台仍可导入。
- `拼多多`、`pdd` 和 `pinduoduo` 均找到同一启用规则集。
- 拼多多典型违规表达能够命中 L4 并在报告中显示来源版本。
- 混合检索覆盖精确命中、近义改写、低相关噪声和模型不可用。

### 9.4 报告与历史

- 有命中的模块默认展开，无命中模块默认压缩。
- 资料不可用和执行失败不会显示成安全或无命中。
- 模拟两次提交和两次法务审核，第二次审核页面可以在抽屉中读取第一次的提交、报告和法务结论。
- 历史抽屉没有会改变旧记录的操作。

### 9.5 提醒与发布一致性

- 提醒按角色筛选，候选多于一条时不连续重复。
- 提醒资源异常时显示兜底内容，工作台其他内容仍正常加载。
- 前端包、后端配置与项目元数据、锁文件、README、技术文档和发布说明统一显示 `0.6.1`。

## 10. 完成标准

只有在三个轨道的自动化测试和人工验收均通过、拼多多 L4 接入得到真实运行结果、历史第一版报告可以从第二版审核页面回看、法务工作台不再出现“需修改”、且连续窗口调整没有遮挡或越界时，v0.6.1 才视为完成。
