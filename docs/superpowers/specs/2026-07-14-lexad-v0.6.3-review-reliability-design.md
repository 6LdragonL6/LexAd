# LexAd v0.6.3 审查可靠性与舆情融合设计

## 1. 背景

v0.6.2 将语义合规审查、舆情案例结构化和舆情风险解释统一接入 DeepSeek 网关，并允许管理员配置单一 API Key。实际使用暴露出两个直接问题：

1. 管理员连接测试成功，但法律工作台第二层持续显示“模型请求格式或固定模型不可用”。
2. 舆情判断覆盖过窄。文案“被生活毒打过，才懂这盒月饼的甜。成年人的世界没有容易二字，咬咬牙，把苦咽下去，把甜留给自己。”未能识别价值观偏差、苦难营销及其与桃李面包月饼争议的相似性。

本次设计还审查了项目已有方案、实现和用户提供的数据，确认当前问题不是管理员连接测试不足，而是业务请求、数据接入和舆情融合链路未完整落实既有设计。

## 2. 既有方案与数据审查结论

### 2.1 设计与实现偏差

- v0.6.1 已提出“确定性规则、中文文本相似性和可选模型解释”的混合检索，但当前舆情模型只收到确定性案例命中和相似事件，没有收到已扫描出的触发线索，也不会在本地无命中时独立发现开放风险。
- v0.6.2 统一网关启用了 JSON Output，但第二层和舆情业务提示词没有明确提供 JSON 输出指令及字段示例。健康检查包含最小 JSON 示例，因此连接测试成功不能证明业务提示词正确。
- 当前代码把 DeepSeek HTTP 400 统一翻译成“模型请求格式或固定模型不可用”，掩盖了业务请求错误和模型可用性错误的区别。
- 早期“舆情库为空时停止判断”的约束适用于尚无真实数据的基础设施阶段，已经不适合 v0.6.3 的“本地证据优先、AI 可独立发现”能力。本设计明确取代该约束。

### 2.2 三套不兼容的数据接入路径

项目当前存在三条舆情数据路径：

1. 管理员导入接口要求 `source_text`、`event_process.trigger` 和 `event_process.outcome`。
2. 命令行脚本支持团队数据中的 `ad_copy_original`、`trigger_short`、`verification` 等字段，但默认指向已经失效的桌面路径。
3. 启动初始化从 `knowledge/L3_cases` 读取法律执法案例并将其写成舆情事件。

用户提供的 34 条真实案例无法通过当前管理员导入接口；命令行脚本没有进入正常启动链路；L3 法律案例一旦生成非空舆情库，又会阻止后续真实舆情基线初始化。

### 2.3 重复与失效数据源

- `docs/v0.5.0-release-notes.md` 声称已经接入 34 条 `sentiment_cases.json`，但仓库中没有该文件。
- `data/case_library.json` 仍为空，不应继续作为第二个内置舆情数据源。
- 用户桌面与仓库内的 `trigger_words.json` 内容和 SHA-256 完全一致，无需重复保存。
- 34 条案例中 `sent-case-018` 与 `sent-case-033` 是同一桃李面包事件，原始广告文案相同，但日期和严重度不同。核验说明支持将规范事件日期定为 2025-09-01，严重度采用较审慎的 `medium`，同时保留所有来源和核验意见。

### 2.4 触发词误用风险

当前 70 个触发词中包含“穷”“毒打”“绝对”等短词。它们适合快速召回候选，不适合直接决定风险等级。脱离上下文的单词命中必须被视为弱证据，由相似案例和 AI 完整语义判断进一步确认。

## 3. 目标与非目标

### 3.1 目标

1. 修复第二层及舆情业务请求，使连接成功的 DeepSeek 配置能够稳定完成受支持的结构化审查。
2. 将用户提供的 34 条真实案例纳入项目内置舆情基线，并统一所有导入路径。
3. 以本地可信证据为优先，同时允许 AI 独立发现本地规则和案例未覆盖的新型舆情风险。
4. 将触发词降级为候选召回线索，避免单词命中直接产生中高风险结论。
5. 对本地与 AI 结果进行可解释、可测试的动态可信度融合。
6. 清理错误的 L3 舆情基线和重复桃李事件，不触碰管理员手工维护的数据。

### 3.2 非目标

- 不改变管理员“测试配置”的语义；该功能仍只验证 API Key、固定地址和固定模型连通性。
- 不增加模型切换、自定义 Base URL 或多供应商能力。
- 不抓取实时微博、抖音、小红书等外部舆情。
- 不让舆情风险改变法律合规分；两条风险轴继续独立展示。
- 不用持续增加固定组合短语来替代语义判断。
- 不删除历史设计文档。新设计只明确哪些旧约束已被取代。

## 4. 总体架构

v0.6.3 保留现有统一审核任务和法律/舆情双分支，调整两个内部链路。

### 4.1 法律第二层

1. 加载经过长度限制的法规和相似案例。
2. 通过统一 DeepSeek 网关发送带明确 JSON 契约和示例的语义审查请求。
3. 校验完成原因、JSON 结构、风险等级和引用字段。
4. 成功时生成第二层命中；失败时将第二层标记为不可用，并保留第一、第三、第四层确定性结果。

### 4.2 舆情分支

1. 规范化物料文本。
2. 扫描触发词，仅形成候选线索。
3. 从已发布真实案例中执行文本、风险维度、行业和平台混合召回。
4. 形成带分数、可信度和来源的本地判断。
5. 无论本地是否命中、资料库是否为空，都让可用的 AI 独立审查完整物料。
6. 校验 AI 原文证据和案例引用，折算 AI 有效可信度。
7. 融合本地与 AI 分数，产生最终等级、解释、建议和冲突标记。
8. AI 不可用时返回本地结果；本地无证据但 AI 有可靠证据时返回 AI 语义结果。

## 5. DeepSeek 业务请求可靠性

### 5.1 管理员连接测试保持不变

管理员测试继续发送最小健康请求，只证明 Key、固定 Base URL 和 `deepseek-v4-flash` 可连接。业务正确性由业务请求代码和自动化测试保证，不扩大管理员测试职责。

### 5.2 明确结构化输出契约

三个业务请求分别提供与 Pydantic 输出模型一致的中文说明、JSON 字样、完整字段列表和最小合法 JSON 示例：

- 语义合规审查；
- 舆情案例结构化；
- 舆情风险审查。

提示词不得只依赖 `response_format={"type":"json_object"}` 猜测字段。非关键列表字段可以使用空数组；必需判断字段不得缺失或由客户端静默编造。

### 5.3 响应处理

- `finish_reason=stop` 或等价正常完成状态才进入结构校验。
- 空内容、长度截断、非法 JSON 和结构校验失败可进行有界重试，总尝试次数最多三次。
- 长度截断重试时收紧低优先级案例上下文，不删除物料原文或核心法规依据。
- 认证、权限、余额和不可重试的 HTTP 400 不进行盲目重试。
- 服务端日志记录任务类型、HTTP 状态、上游请求标识、错误类别和脱敏后的简短原因，不记录 API Key 或完整物料。
- 用户界面区分“配置未完成”“认证失败”“请求不兼容”“响应结构异常”“服务暂不可用”，不再把所有 400 归因为固定模型不可用。

## 6. 统一真实案例接入

### 6.1 唯一内置基线

- 用户提供的原始 `sentiment_cases.json` 已原样保存到仓库 `data/sentiment_cases.json`，保留 34 条事件、来源与核验信息作为审计依据。项目副本 SHA-256 为 `4C4C7138E20B2B30DFF6515C9517213B59472B2DC90FB0C83493A3C89C6A5795`，与用户源文件一致。
- 继续使用现有 `data/trigger_words.json`，不复制桌面同内容文件。
- 删除空的 `data/case_library.json` 并移除文档中的有效数据源引用，避免形成第二个内置舆情来源。
- `knowledge/L3_cases` 只服务法律案例，不再自动转换为舆情案例。

### 6.2 单一规范化入口

新增一个领域级案例规范化与同步服务，供以下入口共同调用：

- 启动时的内置基线同步；
- 管理员 JSON 预检与确认导入；
- 命令行导入脚本。

规范化服务同时接受当前管理员模板和团队真实案例格式，转换为同一个内部事件 DTO。命令行脚本不得再直接实现一套数据库写入逻辑，也不得依赖桌面默认路径。

内部 DTO 保留：外部 ID、标题、品牌、日期、行业、平台、原始广告文案、触发摘要、风险维度、时间线、回应、后果、严重度、来源、核验结论、核验修正和原始元数据哈希。

### 6.3 去重与规范事件

去重分两级：

1. `external_id` 完全相同视为同一来源记录。
2. 品牌、规范化原始广告文案和事件时间高度一致时视为语义重复，即使外部 ID 不同。

桃李面包两条记录合并为一个规范事件：

- 规范外部 ID 使用 `sent-case-018`；
- `sent-case-033` 记录为来源别名；
- 日期使用 2025-09-01；
- 严重度使用 `medium`；
- 合并双方不重复的来源、核验修正和严重度说明；
- 原始两条记录仍保留在版本控制文件中，数据库只发布一个规范事件。

其他冲突不得自动取更严重结论。无法依据核验字段确定规范值时，预检必须给出冲突并要求管理员处理。

### 6.4 已有数据库升级

- 新同步记录在 `source_meta` 中写入 `import_source=builtin_sentiment_cases` 和源哈希；同步以规范外部 ID 和源哈希幂等执行，重复启动不增加事件或资料库版本。
- 已存在同一真实案例时不重复创建；系统管理的基线源发生变化时创建新事件版本和新资料库快照。
- 已有同外部 ID 记录若不带系统同步标记，视为管理员或旧脚本管理的数据：同步跳过并记录冲突，不自动覆盖。
- 只有带 `builtin:knowledge/L3_cases/` 前缀且来源元数据确认由旧启动初始化生成的记录才从新舆情快照中排除。记录保留用于历史审核引用，不物理删除。
- 管理员录入、导入、编辑或发布的案例不被启动同步覆盖、归档或删除。
- 只有基线内容或有效事件集合发生变化时才创建新资料库版本。

## 7. 本地舆情判断

### 7.1 触发线索

触发词命中只包含类别、命中词和原文位置，不直接映射最终风险等级。单个短词命中只能提供低可信度线索。多个同主题线索可以提高召回强度，但仍需案例相似性或 AI 语义证据确认。

### 7.2 案例召回

每个已发布案例使用以下字段构建规范检索文本：

- 原始广告文案；
- 触发摘要；
- 风险维度；
- 标题与品牌；
- 已核验的核心事实；
- 行业和平台。

本地召回综合字符短语相似度、文本词元相似度、风险维度候选、行业和平台加权。触发词可以扩大同风险维度的候选集合，但不能让缺乏文本相似性的案例直接成为高相似命中。

每个候选返回事件 ID、标题、相似度、匹配依据、历史后果、严重度、核验状态和来源摘要。只向 AI 提供 Top-K 真实候选。

### 7.3 本地分数与可信度

本地输出包括 `risk_score`、`confidence`、`risk_topics`、`evidence`、`similar_events` 和 `suggestions`。

- 单一触发词只形成弱证据，本地可信度上限为 0.25。
- 多个上下文一致的线索可以提高可信度，但没有案例相似性时不得仅凭词数进入高风险。
- 经核验案例的高文本相似命中是主要证据，`confirmed` 和经过修正的 `corrected` 案例均可使用；存在未解决冲突的案例降低可信度。
- 行业和平台只做加权，不得在缺乏文本证据时单独产生案例命中。

## 8. AI 开放式舆情审查

AI 不以本地命中为启动条件，必须阅读完整物料，并覆盖但不限于：价值观偏差、苦难营销、焦虑营销、群体冒犯、性别与地域歧视、低俗擦边、灾难营销、社会伦理和情绪操纵。

结构化输出包括：

- `risk_score`：0 至 100；
- `risk_topics`；
- `evidence_quotes`：来自物料原文的证据片段；
- `reasoning`；
- `counter_signals`：支持低风险或语境豁免的反向证据；
- `suggestions`；
- `confidence`：0 至 100；
- `matched_case_ids`：只能引用输入候选中的事件 ID。

系统逐条验证 `evidence_quotes` 是否能够在规范化原文中定位，并验证 `matched_case_ids` 是否属于输入候选。不存在的原文引用和越权案例引用不进入最终证据，并降低 AI 有效可信度。

## 9. 动态可信度融合

### 9.1 有效权重

本地和 AI 都产生 0 至 100 的风险分数以及 0 至 1 的有效可信度。基础优先级为本地 60%、AI 40%，再乘以各自可信度：

```text
local_weight = 0.60 × local_confidence
ai_weight    = 0.40 × ai_effective_confidence
final_score  = (local_score × local_weight + ai_score × ai_weight)
               / (local_weight + ai_weight)
```

AI 有效可信度由模型自报可信度、原文证据有效率、结构完整性和案例引用合法性共同折算，不能直接使用模型自报值。

### 9.2 单侧证据

- 本地没有有效证据时，不用本地零分稀释 AI；由通过证据校验的 AI 独立判断。
- AI 不可用或有效可信度为零时，由本地独立判断。
- 两侧都没有有效证据时返回 `uncertain`，不得伪装成低风险。

### 9.3 强证据和冲突

- 经核验案例高相似命中且本地可信度不低于 0.70 时形成风险下限，AI 不能在没有有效反向证据时将其降级。
- 两侧可信度均不低于 0.65 且风险分数相差至少 30 分时标记 `requires_manual_review=true`。
- 高可信冲突采用审慎等级，保留较高风险提示，同时展示双方证据和不一致原因，不用简单平均掩盖风险。

### 9.4 等级映射

- `0–29`：low；
- `30–59`：medium；
- `60–79`：high；
- `80–100`：severe；
- 无有效证据：uncertain。

等级映射只用于舆情轴，不改变法律合规分。

## 10. 目标文案的预期结果

对于文案：

> 被生活毒打过，才懂这盒月饼的甜。成年人的世界没有容易二字，咬咬牙，把苦咽下去，把甜留给自己。

预期数据流为：

1. “毒打”只形成价值观候选线索。
2. 本地案例检索通过“生活毒打”、成年人困境叙事、月饼产品和价值观维度召回规范化后的桃李面包事件。
3. AI 独立识别“将成年人困境、吞咽苦难与商品甜味绑定”的苦难营销和情绪操纵结构。
4. 双方证据融合后至少返回 `medium`。
5. 风险主题包含“价值观争议”和“苦难营销”，展示桃李面包真实相似案例、原文证据、可信度和避免消费苦难/贩卖焦虑的改写建议。

单独包含“毒打”的游戏战报、影视评论等语境不得仅凭该词自动进入中高风险。

## 11. 结果与界面

在兼容现有舆情结果字段的基础上增加：

- `risk_score`；
- `confidence`；
- `assessment_source`：`local`、`ai` 或 `hybrid`；
- `local_assessment` 摘要；
- `ai_assessment` 摘要；
- `evidence_quotes`；
- `counter_signals`；
- `requires_manual_review`；
- `disagreement_reason`。

结果页优先展示最终等级、主题、证据、相似真实案例和建议。仅在双方冲突时突出人工复核提示。模型错误和内部诊断继续放在技术详情中，不用错误文案覆盖可用的本地结论。

## 12. 失败与降级

- 舆情资料库为空：继续运行触发线索和 AI；结果标记无本地案例证据，不提前退出。
- AI 不可用：返回本地案例与线索结果，标记 AI 暂不可用。
- 本地案例不可用：AI 可独立判断，但降低总体可审计性提示。
- AI 响应非法：有界重试后按不可用降级，不接受未经校验的部分字段。
- 单个案例数据损坏：跳过该案例并记录导入错误，不阻断其他合格案例。
- 法律第二层失败：第二层显示不可用，其他法律层与舆情分支继续完成。
- 舆情分支内部意外失败：法律结果正常保存，舆情可单独重试。

## 13. 测试设计

### 13.1 DeepSeek 网关

- 业务提示词包含 JSON 指令、字段要求和合法示例。
- 健康检查代码和语义保持不变。
- 正常 JSON、Markdown 围栏 JSON、空内容、长度截断、非法 JSON、字段缺失和错误风险等级。
- 空响应、截断和可恢复结构异常执行有限重试。
- 认证、权限、余额、HTTP 400、超时、限流和上游 5xx 分类正确。
- 日志和接口错误不包含 API Key 或完整物料。

### 13.2 第二层

- 合法模型结果转换为语义违规命中、依据和建议。
- 模型不可用时第二层为不可用，其他确定性层继续返回。
- 管理员连接成功与业务请求测试相互独立，业务正确性不依赖管理员端点。

### 13.3 案例数据

- 原始文件包含 34 条事件，外部 ID、来源和核验字段完整。
- 两种输入 schema 都能通过同一规范化入口。
- 桃李面包两条记录只产生一个规范事件，别名、来源和核验信息保留。
- 重复同步不增加事件、版本或资料库快照。
- 基线内容变化时只更新系统管理事件。
- 旧 `builtin:knowledge/L3_cases/` 记录不进入新快照，管理员案例不受影响。
- 管理员预检能接受真实案例文件，不再出现 34 条全部不兼容。

### 13.4 舆情判断

- 目标月饼文案至少为中风险，包含价值观争议、苦难营销和桃李相似案例。
- AI 不可用时，目标文案仍能通过本地真实案例得到可解释结果。
- 本地无候选时，高可信且证据有效的 AI 可以发现新型风险。
- 单一短触发词不直接产生中高风险。
- 游戏语境中的“毒打”可以被反向证据降为低风险。
- AI 引用不存在的原文或未提供的案例时降低有效可信度。
- 强本地证据与低风险 AI 冲突时保留风险下限。
- 双方高可信度冲突时标记人工复核并展示不一致原因。
- 本地和 AI 都无有效证据时返回 uncertain，而不是低风险。

### 13.5 回归

- 法律四层审查、舆情任务聚合、资料中心导入、模型配置、回收站和现有权限测试全部通过。
- 前端构建通过，结果页可兼容旧审核记录缺少新字段的情况。
- 新安装和已有数据库各执行一次启动同步测试。

### 13.6 脚本兼容性

- 命令行入口、`--input/-i` 和 `--admin-id` 参数保持不变；无参数运行改为读取项目内置案例，避免依赖可删除的桌面文件。
- `import_sentiment_cases(input_path, admin_id)` 保持原签名和 `created/updated/skipped` 三字段返回结构。
- 舆情模型接口新增的触发词证据参数必须可选，旧 Python 调用无需传入即可照常执行。
- 文件缺失或管理员不存在时继续以错误码 1 退出，并输出可读错误信息。

## 15. 验收标准

1. 管理员连接测试通过后，受支持的第二层业务请求不再因缺少 JSON 契约而持续返回请求格式错误。
2. 用户提供的真实案例成为唯一内置舆情基线，所有导入入口共用同一规范化与同步逻辑。
3. 桃李重复事件在运行时只有一个规范记录，已有错误 L3 舆情基线不再参与新审核。
4. 目标月饼文案至少得到中风险、价值观争议/苦难营销主题和桃李面包相似案例。
5. 触发词只作候选召回；单个短词不能独立决定最终中高风险。
6. 本地证据优先，AI 按经过校验的有效可信度参与融合，并能在本地无证据时独立发现风险。
7. AI、案例库或单个审核分支不可用时保留其他可解释结果，不输出虚假低风险。
8. 数据同步幂等，不覆盖管理员数据；后端测试和前端构建全部通过。
9. 旧命令行参数、导入函数返回结构及不含触发词参数的模型调用均通过兼容回归测试。
