# 舆情审查结果结构化展示设计

## 背景与问题

审查结果页的“相似舆情事件”目前直接使用 `JSON.stringify` 展示案例的 `historical_consequence` 对象。用户会看到字段名、花括号和引号等 JSON 代码，不符合面向业务人员的结果阅读体验。

本次改动仅优化前端展示，不改变舆情审查算法、后端响应结构或既有接口。

## 目标

- 相似事件的历史影响信息默认折叠，避免结果页过长。
- 用户展开后看到中文标签和可读内容，不看到原始 JSON。
- 支持当前案例库中的固定字段，也兼容未来新增字段及嵌套值。
- 保持桌面端、移动端和深色模式下的可用性。

## 非目标

- 不修改 `public_opinion_result` 或 `similar_events` 的接口结构。
- 不调整风险评分、案例匹配、AI 融合或置信度逻辑。
- 不改造审查结果页其他模块。

## 交互与布局

每个相似舆情事件卡片继续默认展示：

- 事件标题；
- 匹配分和核验状态；
- 匹配依据（存在时）。

卡片底部增加“查看历史影响”按钮。历史影响区域默认折叠，各事件独立控制展开状态。展开后按钮文案变为“收起历史影响”，并通过 `aria-expanded` 表达当前状态。

展开内容使用紧凑的信息项布局，字段标签和内容分层显示；小屏单列排列，较宽屏幕允许两列排列。

## 组件设计

新增 `HistoricalConsequencePanel` 组件，职责限定为：

1. 接收一个未知结构的历史影响值；
2. 管理本组件自己的折叠状态；
3. 将有效字段规范化为可渲染的信息项；
4. 以中文业务标签和可读文本展示信息。

结果页只负责把 `event.historical_consequence` 传入组件，不承担 JSON 解析或字段格式化细节。

## 字段与格式化规则

当前已知字段使用固定中文标签：

| 接口字段 | 展示标签 | 格式 |
| --- | --- | --- |
| `reputation` | 舆论影响 | 文本 |
| `business` | 商业影响 | 文本 |
| `regulatory` | 监管结果 | 文本 |
| `duration_days` | 影响持续时间 | `N 天` |
| `severity_hint` | 历史严重程度 | 低 / 中 / 高 / 严重 / 不确定 |

兼容规则：

- 未知字段将下划线或连字符拆分为空格后作为兜底标签，避免丢失新数据。
- 字符串和数字直接转为文本；布尔值展示为“是”或“否”。
- 数组按可读列表展示；嵌套对象按子项递归转成文本，不调用 `JSON.stringify` 输出原始代码。
- `null`、`undefined`、空字符串、空数组和空对象不生成信息项。
- 展开后若没有有效内容，显示“暂无可展示的历史影响信息”。

## 数据流与兼容性

数据流保持为：后端 `similar_events[].historical_consequence` → `ResultPage.vue` → `HistoricalConsequencePanel`。

后端数据、API 路径、请求参数和响应字段完全不变。已有脚本和其他前端调用方不受影响。

## 错误与异常处理

- 输入不是对象时仍尝试格式化为单条可读内容。
- 格式化异常时组件显示空状态，不向用户暴露运行时对象或 JSON 代码。
- 超长文本允许自动换行，避免撑破卡片宽度。

## 验证标准

- 初次进入结果页时，所有历史影响区域均为折叠状态。
- 点击某个案例的按钮只展开或收起该案例。
- 五个已知字段均以正确中文标签和业务格式显示。
- 未知字段、数组、嵌套对象、空对象均有稳定且无 JSON 代码的展示结果。
- 桌面端、移动端和深色模式的文字、边框与交互状态清晰。
- 前端类型检查和生产构建通过。
