# LexAd v0.5.1 Theme Polish Design

Date: 2026-07-08

## Goal

v0.5.1 is a small quality release focused on dark/light mode support, frontend style consolidation, and low-risk project polish. It should make the existing v0.5.0 workflow feel more mature without changing review engine behavior or expanding the product surface.

## Release Positioning

This release should stay closer to a patch/minor polish update than a frontend rewrite.

In scope:

- Add light and dark theme support.
- Default to the user's system color scheme on first visit.
- Allow manual theme selection and persist it locally.
- Replace repeated hard-coded light UI colors with theme-aware tokens and shortcuts.
- Update frontend-visible version references to v0.5.1.
- Add small structure improvements that directly support theming.
- Update README and release notes for v0.5.1.

Out of scope:

- Changing backend review logic.
- Redesigning the whole UI.
- Splitting every large page component.
- Replacing UnoCSS or adding a UI component framework.
- Reworking browser `confirm` / `alert` flows in this release.
- Adding new user-facing workflow features beyond theme selection.

## Recommended Approach

Use CSS custom properties as the theme foundation and let UnoCSS shortcuts reference semantic classes or CSS variables. This fits the current codebase because most UI is already built from utility classes and a small set of shortcuts such as `card`, `input`, `btn-primary`, and `page-heading`.

Avoid a large component rewrite. The first pass should make the shared primitives theme-aware, then fix hard-coded colors on key pages where dark mode would otherwise break readability.

## Theme Model

The app should support three modes:

- `system`: follow `prefers-color-scheme`.
- `light`: force light theme.
- `dark`: force dark theme.

The persisted value should live in `localStorage` under a stable key such as `lexad-theme`. The active resolved theme should be applied to the document root with a class or data attribute, for example `data-theme="dark"`.

Initial load should apply the theme before or during Vue startup so the app avoids a visible flash from the wrong theme where practical.

## Frontend Architecture

Add a small theme module, for example `frontend/src/composables/useTheme.ts`, responsible for:

- Reading and validating the stored preference.
- Resolving `system` to `light` or `dark`.
- Applying the active theme to `document.documentElement`.
- Listening to system theme changes when preference is `system`.
- Exposing `themePreference`, `resolvedTheme`, and `setThemePreference`.

Add a version constant, for example `frontend/src/constants/app.ts`, responsible for:

- `APP_VERSION = '0.5.1'`.
- `APP_NAME = 'LexAd'` if useful.

Update `TopBar.vue` to:

- Display `v0.5.1` from the constant.
- Add a compact theme control with icon-style buttons or a small segmented control.
- Keep the control usable on mobile drawer/header layouts.

## Style Token Design

Define semantic CSS variables in `frontend/src/styles/main.css`:

- Surface tokens: page background, card background, muted background, overlay.
- Text tokens: primary, secondary, muted, inverse.
- Border tokens: default, strong, focus.
- Brand tokens: primary, hover, active, soft.
- State tokens: success, warning, danger, info and their soft backgrounds.

Light mode should preserve the current sky-blue professional style. Dark mode should use neutral dark surfaces with sky-blue accents, not a purple or overly saturated palette.

Update UnoCSS shortcuts in `frontend/uno.config.ts` so shared primitives become theme-aware:

- `btn-primary`
- `btn-outline`
- `btn-ghost`
- `btn-danger`
- `card`
- `input`
- `label`
- `page-heading`

Where page templates still use hard-coded classes for large surfaces or important text, replace them with semantic classes or variable-backed utility values.

## Page Coverage

The first implementation pass should cover:

- `LoginPage.vue`
- `DefaultLayout.vue`
- `ReviewLayout.vue`
- `TopBar.vue`
- `HomePage.vue`
- `SubmitPage.vue`
- `ResultPage.vue`
- `LegalDashboard.vue`
- `LegalDetail.vue`
- `KnowledgePage.vue`
- `AdminKnowledgePage.vue`
- review components under `frontend/src/components/review/`

`AdminKnowledgePage.vue` is the largest frontend file and should be treated conservatively in v0.5.1. Theme-related class cleanup is in scope; broad component extraction should be deferred to a later version.

## Error Handling

Theme preference parsing should be defensive. Unknown stored values should fall back to `system`.

If `localStorage` is unavailable, the app should still run and use `system` for the current session.

The theme control should not block navigation or review workflows. Theme changes should be immediate and should not require a page reload.

## Testing And Verification

Run:

```bash
cd frontend
npm.cmd run build
```

If backend code is untouched, backend tests are optional for this release. If any backend or shared API behavior changes, run:

```bash
cd backend
python -m pytest app/tests/ -q
```

Manual smoke checks:

- First visit follows system theme.
- User can switch to light, dark, and system.
- Theme persists after refresh.
- Login, workbench, submit, result, legal review, knowledge, and admin pages remain readable.
- Cards, tables, forms, status tags, links, overlays, and drawer navigation have sufficient contrast in both modes.
- No core business workflow changes from v0.5.0.

## Documentation

Update:

- `README.md`: title/version, v0.5.1 highlights, version history.
- `docs/v0.5.1-release-notes.md`: theme support, structure polish, verification notes, known limitations.

Keep v0.5.0 release notes unchanged as historical documentation.

## Deferred Work

These are good follow-up candidates but should not block v0.5.1:

- Split `AdminKnowledgePage.vue` into smaller tabs and form/list/detail components.
- Replace scattered `confirm` and `alert` calls with app-level modal/toast components.
- Tighten frontend API types and reduce `any` in admin knowledge and result payloads.
- Add frontend unit tests for theme preference resolution.
- Add visual regression checks once the UI stabilizes.

## Acceptance Criteria

- The app supports light, dark, and system theme modes.
- Theme preference persists locally and applies consistently after refresh.
- Core pages remain readable in both themes.
- Shared UI shortcuts use theme-aware tokens instead of fixed light colors.
- Frontend build passes.
- Documentation and visible version references show v0.5.1.
- Backend review behavior remains unchanged.
