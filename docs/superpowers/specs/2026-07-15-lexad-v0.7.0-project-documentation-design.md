# LexAd v0.7.0 项目文档体系设计

- 日期：2026-07-15
- 状态：已批准，待实施
- 适用版本：LexAd v0.7.0
- 项目定位：面向企业广告合规协作的法律 AI 应用

## 1. 背景

LexAd v0.7.0 已完成核心审查逻辑升级：硬规则只负责召回候选，AI 在资料库约束下结合语境裁决，系统再验证原文证据与引用依据；事实真实性、证明材料等无法仅凭文本确认的问题进入核验事项；模型或资料不可用时转人工复核。

现有文档仍分散保留 v0.4.x 至 v0.6.x 的版本表述，README、技术参考和管理员指南之间存在重复、过期和叙事不一致的问题。第二阶段需要在不夸大能力的前提下，形成同时服务仓库阅读、现场演示、项目评估和长期维护的文档体系。

## 2. 目标与非目标

### 2.1 目标

1. 让项目评估人员能够快速理解 LexAd 解决的企业广告合规问题、技术创新和应用价值。
2. 清楚区分 LexAd 与普通关键词检测工具：关键词和规则是内部候选，不是直接结论。
3. 说明 AI、法规资料库、平台规则、案例资源、依据验证和人工复核如何协同。
4. 突出系统对企业营销、法务、合规和平台运营协作效率的提升。
5. 为普通用户、管理员和开发者提供可执行、可维护的当前版本说明。
6. 收敛旧文档，同时保留历史版本事实和工程演进记录。
7. 保证关键能力、版本状态和质量结论能够由代码、测试或操作流程验证。

### 2.2 非目标

1. 本阶段不继续修改 v0.7.0 审查业务逻辑。
2. 不以某个实际广告案例作为整套文档的叙事中心。
3. 不编造效率百分比、准确率、获奖情况、企业客户或监管认可。
4. 不把系统描述为律师意见、行政认定或完全自动化的最终决策者。
5. 不在当前文档中保留多套互相冲突的审查架构说明。

## 3. 方案选择

采用“项目价值 + 工程文档”的双层体系。

- 项目价值层回答为什么做、创新在哪里、对企业有什么价值、为什么可信。
- 工程文档层回答如何使用、部署、管理、验证和继续开发。
- README 作为统一入口，只保留高价值概览和最短可行路径，不重复所有细节。

未采用的方案：

- 仅刷新现有文档：成本较低，但无法形成完整项目价值链，过期内容仍容易干扰阅读。
- 单一技术白皮书：适合书面阅读，但不利于仓库导航、现场演示和后续持续维护。

`docs/superpowers/` 保留为主要工程设计与实施记录，不并入普通用户文档，也不作为项目评估人员的首要阅读入口。

## 5. 读者与阅读路径

### 5.1 项目评估人员

`README.md` → `project/project-overview.md` → `project/demo-evaluation-guide.md` → `architecture/review-methodology.md`

项目评估人员应先看到行业问题、企业价值与创新点，再按兴趣深入演示和技术机制。

### 5.2 企业用户与审查人员

`README.md` → `guides/user-guide.md` → `architecture/review-methodology.md`

重点是提交审查、理解风险与核验项、处理人工复核以及正确认识输出边界。

### 5.3 系统管理员

`docs/README.md` → `guides/admin-guide.md` → `architecture/technical-reference.md`

重点是管理法规、平台规则、案例、模型配置和资料有效性。

### 5.4 开发者

`docs/README.md` → `guides/local-development.md` → `architecture/technical-reference.md` → `docs/superpowers/`

重点是本地启动、测试、模块边界、接口契约和历史设计依据。

## 6. 统一叙事主线

所有当前文档围绕以下价值链组织：

> 企业广告合规成本高、规则分散且平台差异明显；LexAd 汇聚法规、平台规则与案例资源，让 AI 在资料库约束下进行语境判断，再通过依据验证和人工复核边界输出可解释、可追溯的结果，从而提升营销、法务与运营协作效率。

叙事顺序统一为：

1. 企业问题：人工检索耗时、机械匹配误报、多平台重复审查、审查尺度难统一。
2. 解决方案：资料库优先、候选召回、AI 语境裁决、引用验证、核验分流和安全降级。
3. 企业作用：发布前初筛、减少无效复核、统一依据、跨部门协作和审计追溯。
4. 法律价值：让结论与资料依据保持联系，并明确事实核验和人工责任边界。
5. 工程可信度：以当前实现、自动化测试、可执行命令和清晰限制提供支撑。

实际案例只在确有助于解释操作时作为简短示例，不承担主要价值叙事，也不进行大篇幅新旧版本案例对比。

## 7. 企业效率与实际作用

效率提升主要通过流程变化论证，不以未经测量的数字宣传。

### 7.1 业务环节

- 营销：在文案进入正式发布流程前获得结构化风险提示。
- 法务与合规：从全面逐字筛查转向重点处理已确认风险、核验事项和人工复核任务。
- 平台运营：在同一任务中理解不同平台规则要求，减少重复查阅。
- 管理人员：通过资料库维护统一组织内的审查依据和知识沉淀。
- 审计与复盘：保留原文证据、判断理由、依据引用和状态信息。

### 7.2 可陈述的效率价值

- 减少由孤立数字、词段和机械规则触发的无效展示。
- 缩短法规、平台规则和案例资料的人工检索与整理路径。
- 将风险判断、事实核验和人工复核分流，降低职责混淆。
- 提高不同人员、部门和平台场景之间的审查一致性。
- 使资料更新集中在资源库，而不是分散在难以维护的硬编码中。

### 7.3 可验证指标

如后续需要量化，只使用实际采集数据，包括单条审查耗时、人工复核占比、有效依据完整率、多平台覆盖和核验事项识别情况。没有可复现数据时只描述机制和评测方法，不给出估算结论。

## 8. 各文档职责

### 8.1 根 README

包含项目定位、企业痛点、核心能力、创新机制、应用价值、架构概览、快速开始、当前状态、文档导航和责任边界。目标是让首次访问者在短时间内建立准确认知。

### 8.2 文档中心 README

按项目评估人员、用户、管理员和开发者提供阅读路线，并明确当前文档、历史归档和工程记录的区别。

### 8.3 项目概览

包含项目摘要、背景与需求、总体方案、创新点、企业应用价值、法律与社会价值、技术可信机制、安全伦理边界和未来方向。其内容可作为项目介绍与评估材料的基础，但不依赖特定表格格式。

### 8.4 演示与评测指南

说明环境准备、演示顺序、界面讲解重点、通用输入方式、结果阅读、异常降级和评测维度。演示流程突出能力链路，不绑定单一实际案例。

### 8.5 用户指南

说明任务创建、参数选择、结果状态、风险依据、核验事项、舆情结果、历史记录和人工复核建议。

### 8.6 管理员指南

合并旧版管理文档中的仍有效内容，说明资料库、平台规则、案例资源、模型配置、版本有效性、数据维护和安全注意事项。

### 8.7 审查方法论

作为审查逻辑的唯一事实来源，说明候选召回、分领域 AI 裁决、引用验证、统一输出、安全降级和责任边界；明确硬规则与关键词不会直接成为用户可见的违规依据。

### 8.8 技术参考

记录系统架构、模块职责、数据流、接口、数据模型、配置、错误处理、测试和安全机制。只描述 v0.7.0 当前实现。

### 8.9 发布说明

记录 v0.7.0 相对上一版本的变化、兼容影响、验证结果、升级注意事项和已知限制，不重复完整使用指南。

## 9. 内容一致性与事实来源

当前文档按以下来源优先级维护：

1. 实际代码、配置和接口模式。
2. 自动化测试及可重复执行结果。
3. 已批准的 v0.7.0 设计与实施记录。
4. 当前用户界面和运行行为。
5. 历史发布说明仅用于说明当时版本，不作为当前事实来源。

`review-methodology.md` 是审查行为的单一事实来源；其他文档只摘要并链接，避免复制后产生分叉。README 与项目概览强调价值，技术参考解释实现，发布说明解释变化。

## 10. 历史文档收敛

1. v0.5.x 至 v0.6.x 发布说明移动到 `docs/archive/releases/`，保留原始版本事实。
2. v0.3.0 至 v0.7.0 的关键变化整合到 `major-version-evolution.md`。
3. 过期管理员指南、迭代报告和单点过程文档先提取仍有效内容，再清理重复文件。
4. `docs/superpowers/` 仅保留每个大版本的主设计与关键实施计划，并建立精选索引；其他过程记录仍可通过 Git 历史追溯。
5. 每份保留的归档文档在开头增加历史版本提示和当前文档入口，但不重写其历史结论。
6. 使用版本控制移动文件并更新仓库内全部引用，不保留内容重复的旧入口文件。
7. 当前指南文件名不带版本号，后续版本在原路径持续维护。

## 11. 异常与边界表达

文档必须明确：

- 模型或资料库不可用时，系统进入人工复核状态，而不是给出虚假通过结论。
- 法规、平台规则和案例资料有时效性，管理员需要维护有效版本。
- 事实真实性不能仅凭广告文本确认，应通过核验事项补充证明材料。
- AI 解释不是独立法律依据，用户应查看系统关联的资料库依据。
- LexAd 提供合规辅助，不替代律师、企业责任人员或监管机关的最终判断。
- 不公开 API 密钥、真实账号、隐私数据或不适合公开仓库披露的配置。

## 12. 文档质量验证

实施完成后执行以下检查：

1. 全仓搜索旧版本号和过期架构表述，区分当前文档与明确标记的历史归档。
2. 检查用户可见风险等级是否统一使用中文，不出现 `high`、`medium`、`low` 作为展示标题。
3. 检查安装、启动、构建和测试命令是否与当前仓库一致，并实际执行关键命令。
4. 检查 Markdown 相对链接、目录和文件名是否有效。
5. 核对版本号、测试结果、功能状态和限制说明，删除无法验证的陈述。
6. 检查是否残留密钥、真实账号或敏感信息。
7. 确认 README、项目概览、审查方法论、技术参考和发布说明之间不存在冲突。
8. 确认归档材料有历史提示，主导航不会把过期文档作为当前说明。

## 13. 实施顺序

1. 建立目标目录并整理历史文档。
2. 先编写审查方法论和技术参考，确定当前事实基础。
3. 编写用户、管理员和本地开发指南。
4. 编写 v0.7.0 发布说明。
5. 基于已核验事实编写项目概览和演示评测指南。
6. 最后重写根 README 和文档中心导航。
7. 统一执行内容、命令、链接、版本和敏感信息检查。

这个顺序保证上层项目叙事来自已核验的工程事实，而不是先写宣传再反向拼接技术说明。

## 14. 验收标准

- 项目评估人员能够从 README 快速理解产品解决的企业问题、核心创新和可信边界。
- 文档能够说明系统如何提升营销、法务、合规和运营协作效率。
- 读者能够明确区分 LexAd 与关键词直接匹配工具。
- AI、资料库、依据验证、核验事项和人工复核形成完整且一致的说明链。
- 当前文档不再使用旧审查架构或过期版本作为现状说明。
- 历史材料完整保留并明确归档，不干扰当前使用。
- 所有关键命令、链接、版本号、状态和质量结论可核验。
- 文档不以特定实际案例为中心，不包含未经证实的效率或准确性宣传。
- 技术创新、企业价值、法律价值和安全边界形成闭环。
