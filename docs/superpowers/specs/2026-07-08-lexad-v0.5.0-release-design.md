# LexAd v0.5.0 发布完善设计

## 背景

v0.4.2 引入管理员资料中心、平台规则版本与双轴审查后，实际测试暴露出平台别名、退回流程、规则识别、历史版本和页面表达上的问题。v0.5.0 的目标是把这些能力收敛成可稳定演示和继续迭代的版本。

本文合并原先同日维护的“稳定性设计”和“项目打磨设计”，保留核心决策，减少重复的发布与验收说明。

## 目标

- 修复中文平台名与内部标识不匹配。
- 支持退回物料修改、版本递增、重新提交和再次审核。
- 明确归档与物理删除边界。
- 扩展 L1 绝对化表达检测并接入真实舆情案例资源。
- 让结果页、队列和版本历史使用一致、面向业务的中文表达。
- 完善空状态、错误状态、加载状态和操作反馈。
- 统一 v0.5.0 版本信息、README、发布说明和测试入口。

## 平台规则归一化

前端继续展示中文平台名，后端通过统一别名表转换为内部标识。规则匹配、版本查询和结果快照都使用规范标识；展示层根据快照返回中文名和版本标签。

存在规则集但没有 active 版本时，审查结果明确提示“已有规则集，但暂无生效版本”，不把它误报为未配置平台。

## 退回与版本闭环

法务退回后，市场人员可以查看退回原因和备注，修改物料并重新提交。影响审核的字段变化时，`Material.current_version` 递增，新 Review 使用新版本号；历史 Review 保留原始提交快照、AI 结果、平台规则版本和法务决定。

只有当前最新、尚未决策的 Review 可以提交法务决定。已有审核记录的物料只允许归档；没有审核记录的草稿可以物理删除。

## 规则与舆情

L1 增加对“最耐用”“最安全”等“最”类绝对化模式的检测，并提供规则依据与修改建议。舆情模块导入团队案例与触发词，输出风险议题、受影响群体、传播诱因和相似历史事件。

法律合规和舆情风险保持独立结果轴。舆情结果不计入法律合规分，模块不可用时提供降级状态而不是让整个审查失败。

## 前端体验

- 首页、提交页、队列、结果页和管理员资料中心统一中文平台显示。
- 市场人员可直接从退回状态进入修改与重提。
- 结果页展示提交次数、法务决定、退回原因和资料版本标签。
- 加载、空数据、失败、禁用和成功状态提供可理解的反馈。
- 技术 ID 仅放在可折叠的技术详情中，不作为主要业务文案。

## 工程与发布

- 清理临时脚本、运行产物和重复说明，但保留迁移、数据模板和正式验证工具。
- README 保持面向使用者，技术细节进入技术参考。
- 新增 v0.5.0 发布说明并统一前后端版本信息。
- 前端构建、后端测试、迁移、种子、完整业务流和真实数据验证必须通过。

## 验收

1. 中文平台名能命中对应 active 规则版本。
2. 退回物料可以多轮修改重提，历史版本不被覆盖。
3. 归档和删除遵守审核记录边界。
4. L1 新模式和舆情真实案例能产生可解释结果。
5. 页面主要区域不显示裸内部标识。
6. README、发布说明和运行时版本统一为 v0.5.0。
