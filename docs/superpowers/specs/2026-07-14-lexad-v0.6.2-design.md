# LexAd v0.6.2 管理能力与登录稳定性设计

## 背景

LexAd v0.6.1 已具备管理员资料中心、品牌档案、物料审核和基于 DeepSeek 的语义审查能力，但实际投入使用后暴露出三类管理与稳定性问题：

1. DeepSeek API Key 只能通过服务器环境变量配置，管理员不能在系统内完成配置或替换。
2. 物料、品牌、舆情案例和平台规则等运行后新增的数据缺少统一删除入口。现有物料物理删除仅允许提交人删除无审核记录的草稿，品牌只有归档能力，难以清理误录或过期数据。
3. 登录接口返回 `401` 时，全局 Axios 拦截器会强制重载登录页，与表单加载状态和路由跳转竞争，造成按钮卡住、页面闪烁或偶发崩溃。

此外，当前三处模型调用分别自行创建 OpenAI 兼容客户端和解析 JSON，输入组织、输出验证、超时、重试与错误分类不统一，需要在接入管理员配置时一并收敛。

## 目标

1. 为管理员提供单一 DeepSeek API Key 配置入口，无需编辑服务器文件或重启应用即可更新运行时配置。
2. 固定使用 DeepSeek 官方 Base URL 和 `deepseek-v4-flash` 模型，不提供多渠道、模型切换或自定义地址。
3. 将语义合规审查、舆情案例结构化和舆情风险解释统一接入一个模型网关，规范输入输出并增强失败隔离。
4. 为物料及关联审核记录、品牌、舆情案例和平台规则提供统一回收站，保留 15 天后自动永久清理。
5. 永久保留导入记录与审计日志，完整记录删除、恢复、手动清理和自动清理行为。
6. 修复错误账户、错误密码、网络异常和会话过期场景下的登录卡顿、整页重载与偶发崩溃。
7. 确保仓库、日志、接口响应和浏览器状态中不出现真实 API Key。

## 非目标

- 不支持多个模型渠道、多个 API Key、负载均衡或渠道故障切换。
- 不允许管理员修改 DeepSeek Base URL、模型名、温度等模型参数。
- 不建设完整密钥管理平台，也不向前端返回已保存 Key 的明文。
- 不允许删除导入记录或审计日志。
- 不引入事件溯源或将全部业务状态重构为操作事件流。
- 不把回收站扩展到用户账户、内置法规文件或系统种子数据。
- 不使用真实 API Key 执行自动化测试。

## 总体方案

v0.6.2 采用增量安全改造，保留现有 Vue 3、FastAPI、SQLAlchemy 和 Alembic 架构：

- 新增管理员“系统管理”页面，集中承载 AI 配置和回收站。
- 新增安全配置服务，加密保存单一 DeepSeek API Key。
- 新增统一 DeepSeek 模型网关，取代散落在业务模块中的直接 SDK 调用。
- 在可删除的业务根对象上增加软删除元数据，并用统一回收站索引管理恢复与清理。
- 修正认证客户端的 `401` 分类和登录状态事务，消除错误登录触发的整页重载。

通用 JSON 快照回收站未被采用，因为模型结构演进后恢复旧快照容易失配；完整事件流也未被采用，因为对一次小版本更新过重。

## 1. 管理员系统管理入口

新增仅管理员可访问的 `/admin/settings` 页面和“系统管理”导航项。前端路由元数据限制角色为 `admin`，后端所有接口继续使用 `require_admin` 强制鉴权，前端隐藏不作为安全边界。

页面包含两个标签页：

### 1.1 AI 配置

展示以下信息：

- 配置状态：已配置或未配置；
- 固定服务：DeepSeek；
- 固定 Base URL：`https://api.deepseek.com`；
- 固定模型：`deepseek-v4-flash`；
- Key 掩码，仅显示少量尾号；
- 最近更新时间与更新管理员。

可执行以下操作：

- 输入新的 API Key；
- 保存并验证；
- 在不修改当前 Key 的情况下重新测试当前配置；
- 清除配置，清除前需要二次确认。

前端不保存 API Key 到 `localStorage`、`sessionStorage`、URL、Pinia 持久层或错误日志。输入框提交成功或失败后均清空。

### 1.2 回收站

统一展示物料、品牌、舆情案例和平台规则，支持：

- 按对象类型筛选；
- 查看名称、删除人、删除时间、预计清理时间和剩余天数；
- 恢复；
- 二次确认后立即永久删除；
- 分页和刷新。

普通业务列表不显示已删除数据。回收站列表不加载已永久清理对象的完整敏感内容，只展示管理所需摘要。

## 2. API Key 安全配置

### 2.1 存储

数据库新增单例式安全配置记录，保存：

- 配置键；
- 加密后的值；
- 可安全展示的尾号指纹；
- 创建、更新时间；
- 最后更新管理员；
- 最近验证状态、时间和脱敏错误类别。

API Key 使用由服务器 `SECRET_KEY` 派生的对称密钥加密后入库。生产环境若仍使用默认 `SECRET_KEY`，后端拒绝保存 API Key，并返回明确的部署配置错误。数据库中不保存明文，也不在审计日志中保存请求体。

密钥轮换不在 v0.6.2 范围内；部署方变更 `SECRET_KEY` 前必须先清除或迁移已加密配置。

### 2.2 保存与验证

保存新 Key 时执行以下原子流程：

1. 校验输入非空、长度合理且不含控制字符。
2. 使用固定 Base URL 和固定模型发送最小化结构化请求。
3. 只有认证、模型可用性和结构化响应验证通过后，才加密并替换当前 Key。
4. 网络超时、认证失败、模型不可用或响应非法时保留旧 Key，不产生半配置状态。
5. 保存成功后使运行时模型客户端缓存失效，下一次调用立即使用新 Key，无需重启。

配置读取接口只返回状态、掩码、更新时间和验证摘要。任何接口都不提供读取明文 Key 的能力。

### 2.3 Git 与日志保护

- `.env`、数据库文件、运行时密钥文件和敏感日志保持在 Git 忽略范围内。
- 示例配置只使用显式占位符，不使用形似真实凭据的字符串。
- 自动化测试使用伪造 Key 和 Mock 传输层。
- 提交前扫描暂存区中的疑似 DeepSeek/OpenAI Key、Bearer 令牌、数据库文件和 `.env` 内容。
- 模型日志不记录 Key、Authorization 头、完整提示词、完整物料或模型原始响应。

## 3. 统一 DeepSeek 模型网关

### 3.1 固定运行参数

- Base URL：`https://api.deepseek.com`
- 模型：`deepseek-v4-flash`
- 输出模式：JSON object
- 客户端密钥来源：安全配置服务

业务代码不能直接创建 OpenAI 客户端或自行读取 API Key。语义合规审查、舆情案例结构化和舆情风险解释均调用统一网关。

### 3.2 输入规范化

每个模型任务定义独立、类型化的输入对象和提示模板：

- 去除无意义首尾空白并规范换行；
- 对物料、法规上下文、相似案例和命中结果分别限制长度与数量；
- 使用 JSON 或明确分区传递不可信内容，避免把用户文案与系统指令混为一体；
- 为截断行为记录可诊断元数据，但不记录被截断的原文；
- 空输入在调用模型前直接拒绝。

### 3.3 输出验证

三类任务分别使用 Pydantic 输出模型验证：

- 必需字段存在；
- 对象、数组和文本类型正确；
- 风险等级等枚举值合法；
- 数值范围和列表长度合理；
- 模型没有返回空内容或非对象 JSON。

网关使用 DeepSeek JSON 输出模式。为兼容服务端偶发的 Markdown 包裹，可进行一次保守的代码围栏剥离；除此之外不猜测或静默补造缺失字段。非法响应转换为统一的结构化错误。

### 3.4 超时、重试与失败隔离

- 设置明确的连接超时和响应超时，防止请求无限悬挂。
- 仅对连接失败、超时、`429` 和可恢复的 `5xx` 最多重试两次，并使用短暂指数退避和抖动。
- `401`、`403`、请求参数错误和结构验证错误不重试。
- SDK 自带重试与应用重试不能叠加，避免重复调用失控。
- 错误对外只暴露类别和可操作提示，不暴露上游响应中的敏感内容。

单项模型调用失败时，确定性规则与其他成功模块继续保留；审查报告明确标记该模块不可用或执行失败，不能把失败伪装成“无风险”。

### 3.5 可观测性

每次调用只记录：

- 调用类型；
- 内部请求关联 ID；
- 耗时；
- 尝试次数；
- 成功或错误类别；
- 输入是否被截断。

不记录 API Key、Authorization 头、完整提示词、用户原始物料或未经清洗的模型响应。

## 4. 回收站数据设计

### 4.1 通用元数据

物料、品牌、舆情案例和平台规则集增加统一语义的软删除字段：

- `deleted_at`；
- `deleted_by_id`；
- `purge_after`。

新增回收站索引记录对象类型、对象 ID、安全摘要、删除时间、清理时间、删除人和当前状态。该索引用于统一列表和调度，不承担业务对象快照恢复。

删除、恢复和永久清理必须在数据库事务内完成；失败时整体回滚。服务层提供统一接口，端点不能自行拼接删除逻辑。

### 4.2 物料

- 物料是删除根对象，其提交快照和审核记录作为同一关系图保留在数据库中。
- 删除后物料及关联记录从列表、队列、统计、详情和审查入口隐藏。
- 若正在执行审查，任务在下一个安全检查点终止；相关执行状态标记为已中止，不继续写入结果。
- 恢复普通物料时恢复删除前业务状态；删除时正在审查的物料恢复为草稿，需要管理员或提交人重新发起审查。
- 永久删除时按审核记录、提交快照、物料的顺序清理，避免外键孤儿。

### 4.3 品牌

- 删除后品牌不能用于新提交或普通品牌检索。
- 历史物料保留品牌关联和显示能力，不随品牌进入回收站。
- 恢复后重新可用于提交。
- 永久删除时先将历史物料的 `brand_id` 置空，再删除品牌；物料本身不删除。

### 4.4 舆情案例

- 舆情事件及其版本作为整体进入回收站。
- 删除后不参与新审查或资料中心普通检索。
- 恢复时恢复删除前发布状态。
- 永久删除时清理事件版本和事件，但永久审计日志保留对象摘要和操作记录。

### 4.5 平台规则

- 平台规则集及其版本作为整体进入回收站。
- 删除后立即退出新审查的规则选择，不影响已经保存的历史审查结果。
- 恢复时若该平台没有其他生效规则，可恢复删除前状态；若已有新的生效规则，则恢复为未启用状态，不能静默替换当前规则。
- 永久删除时清理规则版本和规则集；历史审查中已序列化的命中结果继续可读，无法读取的旧规则详情明确显示为“规则版本已清理”。

### 4.6 保留期与清理

所有回收站对象的 `purge_after` 为删除时间后 15 天。清理机制：

- 应用启动时执行一次到期扫描；
- 应用运行期间每 24 小时执行一次；
- 管理员可在保留期内手动永久删除；
- 清理任务使用幂等操作，重复执行不会产生错误或重复审计；
- 单个对象清理失败不阻断其他对象，并记录脱敏失败原因供管理员检查。

导入记录和审计日志不受保留期影响，永久保留。

## 5. 审计与权限

所有系统管理接口均要求管理员角色。审计记录至少包含：

- 操作类型：配置更新、配置清除、配置测试、删除、恢复、手动清理、自动清理；
- 操作者 ID；自动任务使用系统主体；
- 目标类型、目标 ID 和安全摘要；
- 时间、结果和脱敏错误类别。

配置审计不记录 Key、Key 哈希全文、请求体或上游认证响应。删除审计不记录完整物料文案。

非管理员访问管理页面时由路由重定向，直接调用后端接口时返回 `403`。

## 6. 登录稳定性

### 6.1 根因

当前全局响应拦截器对所有 `401` 执行清除令牌并设置 `window.location.href = '/login'`。错误账户或密码本应是登录表单的正常业务错误，却触发了整页重载。重载、表单 `finally` 和路由导航并发执行，造成按钮停留在加载态、页面闪烁和偶发崩溃。

当前登录流程还会先保存令牌、再调用 `/auth/me`。若第二步失败，`fetchUser` 会清除令牌但吞掉异常，登录页仍尝试跳转首页，形成半登录状态和额外重定向。

### 6.2 修复

- 登录成功响应直接包含令牌和用户信息，前端不再追加 `/auth/me` 请求。
- 登录状态以事务方式更新：成功时同时写入令牌和用户；失败时两者均为空。
- `/auth/login` 的 `401` 只返回给表单，不触发会话过期处理。
- 只有请求携带会话令牌、请求不是登录接口且返回 `401` 时，才清除会话并进入登录页。
- 已位于登录页时不重复导航或整页刷新。
- 登录提交函数增加同步重复提交保护，并等待路由替换完成。
- 所有成功、业务错误、网络错误和超时路径都在 `finally` 中恢复按钮。
- 错误提示区分账户或密码错误、请求超时和服务不可用，但不透露具体账户是否存在。
- Axios 客户端设置合理的全局超时，防止断网时无限等待。
- 后端认证对不存在用户和错误密码返回一致的 `401` 与提示，并隔离异常密码哈希造成的服务器错误。

## 7. 数据库迁移与兼容性

使用 Alembic 执行增量迁移，覆盖本地 SQLite 和 Neon PostgreSQL：

- 新增安全配置表；
- 新增回收站索引表；
- 为四类业务根对象新增软删除字段和必要索引；
- 新增或复用永久审计表所需字段。

迁移不改变现有对象的业务状态；新增删除字段默认为空，现有数据继续可见。应用只有在迁移完整后才启用新接口，避免半迁移运行。

普通查询的“排除已删除”条件集中在服务层，避免端点遗漏。需要读取回收站数据时使用显式的管理员专用查询。

## 8. 测试策略

### 8.1 后端

- 安全配置：管理员权限、加密落库、掩码响应、默认 `SECRET_KEY` 拒绝、旧 Key 保留、缓存失效。
- 模型网关：成功、空响应、非法 JSON、字段缺失、非法枚举、超时、连接错误、限流、`5xx`、认证失败、重试次数和输入截断。
- 模型接入：扫描生产代码，保证不存在网关之外的 OpenAI 客户端创建或直接 API Key 读取。
- 回收站：四类对象删除、隐藏、恢复、手动清理、15 天到期清理、重复清理和事务回滚。
- 关系完整性：物料审核与快照、品牌历史关联、舆情版本和平台规则版本没有孤儿记录。
- 审查并发：正在审查的物料进入回收站后安全中止，恢复后为草稿。
- 登录：错误账户、错误密码、异常哈希、成功登录响应和权限边界。
- 数据库：SQLite 与 PostgreSQL 兼容的迁移和查询行为。

模型测试全部使用伪造 Key 和 Mock 客户端，不产生真实外部请求。

### 8.2 前端

- 错误账户和错误密码不会刷新页面，按钮恢复可点击。
- 快速重复提交只产生一个登录请求。
- 断网、超时、服务端错误和成功登录均显示正确状态。
- 会话过期仅触发一次清理与导航。
- 非管理员不显示系统管理入口，且无法进入路由。
- API 配置页面不回显明文，提交后清空输入。
- 回收站筛选、恢复、手动清理和倒计时状态正确。
- 生产构建通过，关键页面在窄屏与桌面宽度下可操作。

### 8.3 发布前安全检查

- 扫描 Git 暂存区与待提交文件中的疑似 API Key、Bearer 令牌和真实 `.env` 值。
- 确认数据库文件、上传文件和敏感日志未被跟踪。
- 确认示例与测试只包含明显的虚假占位符。
- 不在发布说明、截图或测试夹具中展示真实 Key。

## 9. 验收标准

1. 管理员可在系统管理页保存、替换、测试和清除单一 DeepSeek API Key；普通用户无权访问。
2. Base URL 固定为 `https://api.deepseek.com`，模型固定为 `deepseek-v4-flash`，前端与接口均不能修改。
3. Key 在数据库中加密，接口、浏览器状态、日志、审计和 Git 中均无明文。
4. 三类模型任务全部通过统一网关，输入输出经过统一格式化和结构验证。
5. 模型错误有界重试且不会清空其他有效审查结果，也不会被误报为无风险。
6. 物料及审核、品牌、舆情案例和平台规则均可进入回收站、恢复和永久删除。
7. 回收站对象在 15 天后自动清理，导入记录与审计日志永久保留。
8. 删除与恢复不产生孤立记录，不破坏历史物料和已保存审查结果。
9. 连续输入错误账户或密码不会刷新整页、卡死按钮或导致未处理异常。
10. 后端测试、数据库迁移测试、前端生产构建和关键交互回归通过。
