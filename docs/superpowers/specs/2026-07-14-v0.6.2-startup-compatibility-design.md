# LexAd v0.6.2 启动兼容与退出脚本加固设计

## 背景

v0.6.2 将 DeepSeek 地址和模型固定在统一网关中，只保留管理员填写 API Key 的入口。已有本地部署的 `backend/.env` 可能仍包含 `DEEPSEEK_BASE_URL` 与 `DEEPSEEK_MODEL`。当前 `Settings` 禁止未知输入，因此 Alembic 在读取数据库配置前就会因这两个旧字段失败。

同时审查 Windows 退出脚本后发现，PID 记录的兼容与失败状态处理仍有三处风险：缺少启动时间的记录会被默认信任；合法 JSON 中的非法 PID 缺少受控错误；停止失败后 PowerShell/批处理入口不一定向调用者返回非零状态。

## 配置兼容设计

`Settings` 显式声明两个仅用于读取旧部署的废弃字段：`DEEPSEEK_BASE_URL` 与 `DEEPSEEK_MODEL`。字段不参与序列化，应用和网关不读取它们，其他未知配置仍保持拒绝。

无论旧 `.env` 中写入什么值，实际调用始终使用代码中的固定地址 `https://api.deepseek.com` 和固定模型 `deepseek-v4-flash`。管理员配置和旧环境变量都不能覆盖这两个常量。

该方案兼容已有环境文件，同时继续发现真正的配置拼写错误，避免使用 `extra="ignore"` 掩盖所有未知字段。

## 退出脚本加固设计

退出脚本仅操作 `scripts/.dev-pids.json` 中通过完整校验的记录。每条记录必须满足：

- 服务名只能是 `backend` 或 `frontend`，且不可重复；
- PID 必须是正整数；
- 必须存在可解析的 `rootProcessStartTime`；
- 当前进程的启动时间必须与记录匹配。

缺字段、非法类型、重复服务或启动时间不匹配时，不终止任何可疑进程；脚本保留 PID 文件、给出可操作的诊断并返回非零状态。已经退出的已验证记录可视为停止成功。

`taskkill /T /F` 仍只针对通过校验的根进程树，不按进程名批量结束 Node、Python 或 PowerShell。端口 8000/5173 的状态用于确认停止结果和诊断残留，不作为终止未知监听进程的授权。

`stop-dev.ps1` 在完全成功时显式返回 0，在任一失败时显式返回 1。`stop-dev.bat` 保存 PowerShell 退出码，成功和失败显示不同提示，并以相同退出码结束，避免 `pause` 覆盖执行结果。

启动脚本复用同样的严格记录校验原则。遇到旧版或损坏的 PID 文件时不自动删除、不尝试终止其中的进程，而是要求用户根据诊断确认后处理，避免 PID 被系统复用时误伤其他程序。

## 错误处理

- 旧 DeepSeek 字段只被接受并忽略，不记录其值，避免意外输出潜在敏感配置。
- PID 文件解析或结构校验失败时，错误消息说明具体记录和字段，但不输出环境变量或密钥。
- 停止命令失败时保留 PID 文件，便于再次执行或人工排查。
- 端口仍被占用时报告监听状态，不自动接管或结束未验证进程。

## 验证

1. 设置两个旧 DeepSeek 环境变量后实例化 `Settings`，确认成功且网关常量不变。
2. 设置任意其他未知变量，确认配置仍拒绝加载。
3. 使用临时 SQLite 执行 `alembic upgrade head`，确认旧 `.env` 不再阻塞迁移。
4. 对退出脚本执行静态语法检查，并用隔离的临时 PID 记录覆盖缺字段、非法 PID、重复记录、启动时间不匹配和已退出进程。
5. 对一个受控测试进程验证成功停止、PID 文件删除和退出码 0；验证失败场景保留 PID 文件并返回 1。
6. 扫描 Git 变更，确保 `.env`、真实 API Key 和运行时 PID 文件均未提交。

## 非目标

- 不修改用户现有 `backend/.env`，也不提交其中内容。
- 不允许管理员或环境变量修改 DeepSeek 地址与模型。
- 不自动终止未被 PID 记录严格验证的端口监听进程。
