# LexAd v0.3.0 — 广告合规审查平台设计文档

**日期**: 2026-05-15
**状态**: 已确认
**范围**: 两部门审查交互闭环（提交→AI审查→结果展示→法务审核→决策→退回/版本管理）

## 一、概述

**目标用户**: 企业内部市场部 + 法务部
**第一期范围**: 提交→AI审查→结果展示→法务审核→决策（不含通知/推送/注册、品牌档案）
**账户策略**: 市场部10个测试账户 + 法务部10个测试账户 + 管理员1个（预置）

## 二、技术栈

| 层     | 技术                              |
|--------|-----------------------------------|
| 前端   | Vue 3 + Vite + TypeScript + Pinia + Vue Router + Axios + UnoCSS |
| 后端   | FastAPI + Python 3.11+ + SQLAlchemy + Alembic + Pydantic v2 |
| 数据库 | Neon PostgreSQL（云端轻量级，仅存用户特征数据） |
| AI引擎 | DeepSeek API (deepseek-chat) |
| OCR    | Tesseract + Pillow |
| 向量库 | ChromaDB（本地文件级，案例语义检索） |
| 部署   | Render（后端 Web Service + 前端 Static Site）+ Docker Compose（本地开发） |

## 七、审查引擎设计（四层流水线）

引擎层与知识库层的对应关系（KB-Lx 命名空间隔离）：

| 知识库层级 | 内容           | 引擎层级         |
|-----------|---------------|-----------------|
| KB-L1     | 法律法规全文   | 第一层（硬规则）+ 第二层（法条注入） |
| KB-L2     | 行业专项规则   | 第二层（Prompt 注入） |
| KB-L3     | 行政案例库     | 第二层（ChromaDB Top5 检索） |
| KB-L4     | 平台规则库     | 第四层（平台差异对照） |
| KB-L5     | 改写模板库     | 第一层（违禁词提取）+ 最终建议 |

### 流水线

**预处理**: 文本清洗 → OCR（如有图片）→ 行业分类确认

**第一层 — 硬规则匹配**（确定性计算，~0.5s）
- 数据来源：KB-L1 提取的硬性禁止条款 + KB-L5 违禁词映射
- 算法：AC 自动机 / Trie 多模式匹配
- 违禁词分类：绝对化用语 / 效果保证 / 涉医用语 / 功效宣称 / 虚假表述
- 输出：matched_rules[] (rule_id, rule_text, source_law, match_type)

**第二层 — 语义推理**（DeepSeek API，~4-6s）
- 数据来源：KB-L1 法条原文 + KB-L2 行业规则 + KB-L3 ChromaDB Top5 相似案例
- 处理第一层无法覆盖的"无形违规"
- 输出：违规项 / 法律依据 / 案例参考 / 推理链路 / 风险分 / 修改建议

**第三层 — 证明材料检查**（确定性规则 ~1s）
- 识别需证明材料的表述（"经XX验证""临床证明""数据显示"等）
- 不直接判违规，标记需提供证明材料
- 参考《广告法》第28条

**第四层 — 平台差异适配**（确定性规则，~0.5s）
- 数据来源：KB-L4 平台规则
- 逐项对照违规项 × 投放平台 → 每平台判定等级（禁止/限制/需资质/允许）

### 综合评分
- 基础分 100 - Σ(各项扣分)
- 第一层：-15~35 分/项，第二层：-10~30 分/项，第三层：-5~10 分/项
- 第四层不扣分，仅标记平台风险

## 九、核心交互流程

1. **登录** → 角色识别，差异化首页
2. **提交物料** → 表单（物料名/行业/平台/类型/原文/图片(可选)/优先级）
3. **AI自动审查** → 提交后自动触发四层引擎，约15-20s，状态 ai_reviewing → pending_legal
4. **查看结果** → 三栏布局，交互联动，支持采纳建议/重新编辑/提交法务/导出PDF
5. **法务审核台** → 队列排序（极速红色置顶 > 加急橙色 > 普通按时间）
6. **法务决定** → 通过/退回/有条件通过，退回后生成新版本号，版本对比视图
