# LexAd v0.4.2 / v0.5.0 设计

日期：2026-07-07

## 1. 背景

LexAd v0.4.1 已具备广告物料提交、文件文本提取、法律与行业规则审核、异步审核任务、法务复核和五级知识库浏览能力。本次设计解决三个问题：

1. 物料提交页字段集中，首次使用者难以快速理解，重复提交也存在不必要操作。
2. L4 平台规则虽然已有本地资料，但审核流水线仍是占位实现；系统没有版本、生效时间、差异和回滚机制。
3. 当前审核只衡量法律合规风险，不能提示广告发布后可能引发的品牌舆情风险。

## 2. 设计目标

- 将物料提交改为单页渐进式流程，降低首次使用和重复提交成本。
- 建立仅管理员可访问的“资料中心”，统一管理舆情事件和平台规则。
- 让不具备计算机基础的管理员通过自然语言录入舆情事件，不要求日常使用 JSON。
- 支持结构化 JSON 批量导入，作为高级能力和后续数据交付格式。
- 将现有 L4 平台规则接入正式审核，并支持版本差异、生效、回滚和历史复现。
- 在统一审核任务中并行生成法律风险和舆情风险，采用双风险轴展示。
- 在真实舆情数据尚未提供时完整交付基础设施，并明确显示空资料库状态。

## 3. 非目标

- v0.4.2 不抓取微博、抖音、小红书等平台的实时舆情。
- v0.4.2 不监测广告发布后的舆情变化。
- v0.4.2 不自动爬取或自动发布平台规则。
- 舆情风险不计入现有法律合规分。
- 系统输出是发布前风险提示，不代替品牌、公关或法务人员的最终判断。

## 4. 版本边界

### 4.1 v0.4.2

v0.4.2 交付可运行的完整框架：

- 单页渐进式物料提交。
- 管理员资料中心及权限控制。
- 舆情事件自然语言录入、AI 结构化、人工复核、发布、归档、恢复和版本记录。
- 舆情 JSON 批量导入、模板下载、逐条校验和重复数据处理。
- 舆情混合引擎的接口、检索流程、结果模型和空库降级状态。
- 平台规则导入、版本差异、生效时间、启用、回滚和 L4 正式审核。
- 统一审核 API、法律与舆情双轴结果、独立失败和独立重试。
- 数据库迁移、权限与审计、自动化测试和管理员操作说明。

v0.4.2 的舆情库为空时不得输出“低风险”，必须输出“资料库待补充，暂无法完成基于案例的舆情判断”。

### 4.2 v0.5.0

用户提供真实舆情事件 JSON 后完成：

- 导入、校验和发布真实舆情事件。
- 抽检系统生成的议题、群体、触发因素、后果等级和摘要。
- 校准规则权重、相似度阈值、风险等级和模型提示词。
- 建立舆情回归测试集，统计误报和漏报。
- 完成发布验证并将版本号更新为 v0.5.0。

## 5. 角色与权限

### 5.1 市场人员

- 提交物料。
- 查看本人的法律与舆情审核结果。
- 对失败的分析模块发起重试。

### 5.2 法务人员

- 使用现有法务待审流程。
- 查看法律与舆情双轴结果及其版本依据。
- 不维护平台规则或舆情资料。

### 5.3 管理员

- 拥有市场和法务现有能力。
- 访问资料中心。
- 录入、导入、复核、发布、归档和恢复舆情事件。
- 导入、比较、启用和回滚平台规则版本。
- 查看导入记录和审计记录。

所有资料中心写操作必须在后端校验管理员角色，不能只依赖前端隐藏入口。

## 6. 物料提交设计

提交页采用“单页渐进式”布局。

首屏只显示：

- 物料内容：拖入文件或粘贴文本。
- 行业。
- 投放平台。
- “开始风险审查”按钮。

“更多设置”折叠区包含：

- 物料名称；未填写时由文件名或文案摘要自动生成。
- 物料类型；上传文件后自动识别。
- 优先级。
- 截止时间；仅加急任务显示。

文件提取结果仍允许预览和编辑。提交后沿用现有异步审核任务和结果轮询，不要求用户停留在页面。

## 7. 管理员资料中心

资料中心包含三个一级页签：

1. 舆情事件。
2. 平台规则。
3. 导入记录。

### 7.1 舆情事件日常录入

默认入口不显示 JSON 或技术字段。管理员按以下步骤操作：

1. 在“这件事发生了什么”中粘贴新闻、内部记录或自然语言说明。
2. 在“造成了什么后果”中描述声誉、业务、监管或其他影响。
3. 可选填写来源名称或链接、大致发生时间。
4. 点击“AI 帮我整理事件”。
5. 系统生成标题、行业、平台、触发原因、时间线、企业回应、后果分类、敏感群体、风险议题和后果等级。
6. 管理员在自然语言核对页修改结果；不确定字段以黄色提示。
7. 管理员发布后，事件才进入新审核任务使用的资料库。

草稿自动保存。状态流固定为：

`草稿 → AI 处理中 → 待复核 → 已发布 → 已归档`

AI 处理失败时回到草稿并保留原始输入，允许重试。已用于审核的事件不能物理删除，只能归档；未发布草稿可以彻底删除。

### 7.2 JSON 批量导入

批量导入放在“更多工具”中，并提供可下载模板和示例。管理员选择文件后先看到校验结果，不直接写入已发布资料库。

导入处理规则：

- 结构错误定位到事件编号和字段。
- 合格事件保存为草稿。
- 不合格事件不保存，并给出普通语言修复说明。
- `external_id` 重复时提供“新增、更新、跳过”选择，不自动覆盖。
- 整批导入中断时，已解析数据仍保持草稿或未发布状态。
- 导入完成后逐条或批量发起 AI 整理，但仍须人工复核后发布。

## 8. 舆情事件输入格式

```json
{
  "schema_version": "1.0",
  "events": [
    {
      "external_id": "case-001",
      "title": "事件标题",
      "industry": ["食品"],
      "platforms": ["微博", "抖音"],
      "occurred_at": "2026-01-15",
      "source_text": "相关报道、评论摘要或完整事实材料",
      "sources": [
        {
          "name": "来源名称",
          "url": "https://example.com/article",
          "published_at": "2026-01-16"
        }
      ],
      "event_process": {
        "trigger": "争议由什么内容或表达触发",
        "timeline": [
          {
            "at": "2026-01-15",
            "stage": "发生",
            "description": "事件经过"
          }
        ],
        "brand_response": "企业采取的回应",
        "outcome": "事件最终结果"
      },
      "consequences": {
        "reputation": "品牌声誉影响",
        "business": "销量、合作或市值等影响",
        "regulatory": "约谈、处罚或整改情况",
        "duration_days": 14,
        "severity_hint": "high"
      },
      "notes": "其他补充信息"
    }
  ]
}
```

必填字段：

- `schema_version`
- `events`
- 每个事件的 `title`
- `source_text`
- `event_process.trigger`
- `event_process.outcome`

`external_id` 建议填写，用于后续增量更新。其余字段允许缺失，由系统标记不确定性并提示管理员补充，不得凭空生成事实。

系统处理后生成并单独保存：

- 风险议题。
- 触发表达或触发模式。
- 可能受影响的群体。
- 传播诱因。
- 规范化行业和平台标签。
- 后果等级。
- 事件摘要。
- 模型置信度。
- 用于语义检索的规范化文本。

系统生成字段必须保留模型版本、生成时间和人工修改记录。

## 9. 平台规则版本管理

每个平台拥有独立规则集和版本。规则版本至少保存：

- 平台名称。
- 版本号。
- 来源名称或链接。
- 发布日期。
- 生效时间。
- 失效时间。
- 导入人、复核人和启用人。
- 规则正文和结构化规则项。
- 状态：草稿、待生效、当前生效、已失效。

导入新版本后，系统展示规则项级别的新增、修改和删除差异。管理员确认生效时间后才能启用。一个平台同一时间只能有一个生效版本。

回滚通过切换生效版本实现，不删除新版本。审核任务开始时固定平台规则版本，即使管理员随后启用新版本，运行中的任务和历史结果仍使用原快照。

## 10. 审核架构

前端继续调用统一的“发起 AI 审核”接口并获得一个审核任务。法律与舆情分析共用现有 DeepSeek 模型服务、认证配置和基础客户端，但作为两个独立内部模块运行。

审核流程：

1. 创建审核任务并固定法律库、行业库、平台规则和舆情库版本。
2. 运行现有法律硬规则匹配。
3. 运行行业与法律语义分析。
4. 运行目标平台的生效规则匹配和语义分析。
5. 舆情引擎执行敏感触发器匹配。
6. 从已发布舆情事件中检索相似案例。
7. 模型结合文案、行业、平台、触发器和相似案例生成舆情解释。
8. 聚合两套独立结果并保存各自状态。

法律与舆情分支应并行执行，避免审核耗时简单翻倍。两者可以分别成功、失败或重试。模型服务相同不意味着两个结果共享状态或相互覆盖。

## 11. 舆情混合引擎

舆情引擎采用三阶段混合方案：

### 11.1 确定性触发

使用管理员复核后的风险议题、触发模式、敏感群体和平台标签进行规则匹配。该阶段必须可解释，并在模型不可用时保留基础提示。

### 11.2 相似事件检索

只检索“已发布”状态的事件。行业和平台作为过滤或加权条件，语义相似度用于排序。每条命中结果必须带事件 ID、标题、相似度、历史后果和资料库版本。

### 11.3 模型解释

模型只基于物料、确定性命中和检索到的事件生成：

- 舆情风险等级：低、中、高、严重。
- 风险议题。
- 潜在受影响群体。
- 传播诱因。
- 相似事件及其后果。
- 修改或复核建议。
- 置信度和不确定性说明。

模型不得虚构未检索到的案例或后果。没有足够依据时必须输出“不确定”而不是低风险。

## 12. 双风险轴结果

法律风险继续使用现有合规分及规则依据。舆情风险使用独立等级，不对法律分加减分。

结果页展示：

- 法律合规分、风险项、法规依据和修改建议。
- 舆情风险等级、风险议题、受影响群体、传播诱因和建议。
- 相似舆情事件、相似度和历史后果。
- 法律库、行业库、平台规则和舆情库版本。
- 两个分析模块各自的完成、失败或不可用状态。

法律低风险但舆情高风险时，系统建议“暂停发布并由品牌负责人复核”，但不替代人工决策。

## 13. 失败、降级和重试

- 任一分析模块失败时，另一模块结果正常展示和保存。
- 失败模块可单独重试，不能重复创建物料或覆盖已完成结果。
- 舆情库没有已发布事件时，舆情模块返回 `knowledge_base_empty`，界面显示资料待补充。
- 模型不可用但规则或相似事件检索成功时，展示基础结果并标记“AI 解释暂不可用”。
- 模型输出无法通过结构校验时，该模块失败，不把原始模型文本伪装成规则。
- 导入失败不能改变当前生效的平台规则或已发布舆情库。
- 版本切换失败时保持原生效版本，并记录失败审计。

## 14. 数据与审计

建议新增的独立领域模型：

- `PublicOpinionEvent`：事件当前业务状态和原始输入。
- `PublicOpinionEventVersion`：结构化内容、系统生成字段和人工修订版本。
- `PublicOpinionLibraryVersion`：一次发布形成的可审核运行快照。
- `PlatformRuleSet`：平台规则集。
- `PlatformRuleVersion`：平台规则内容、状态和生效区间。
- `KnowledgeImportJob`：导入文件、校验结果、处理进度和错误摘要。
- `KnowledgeAuditLog`：发布、归档、恢复、生效和回滚操作。

审核结果保存所使用的知识版本 ID，而不是只保存可变的“当前版本”引用。

## 15. API 边界

现有物料提交与发起审核 API 保持兼容。新增管理员 API 按领域拆分：

- 舆情事件的列表、详情、创建、修改、AI 整理、发布、归档、恢复。
- 舆情 JSON 模板下载、导入预检、确认导入和导入任务状态。
- 平台规则版本的列表、导入、差异、启用和回滚。
- 导入记录和审计记录查询。
- 审核模块的独立重试。

所有写 API 均要求管理员权限。批量导入、AI 整理和索引构建使用后台任务，接口返回任务 ID。

## 16. 安全与隐私

- 上传文件限制大小和扩展名，并使用服务端生成的文件名。
- JSON 解析设置文件大小、事件数量和文本长度上限，防止资源耗尽。
- 来源文本和模型输出在页面展示前进行安全转义。
- 审计日志不得记录访问令牌、模型密钥或完整敏感请求头。
- 归档和发布操作需要二次确认；平台规则回滚明确显示影响平台和目标版本。
- 管理员导入的来源材料按现有企业数据权限处理，不对外公开。

## 17. 测试策略

### 17.1 单元测试

- 舆情 JSON 结构校验、缺失字段和重复 `external_id`。
- 舆情状态转换和禁止物理删除规则。
- 平台规则生效区间、单一当前版本和回滚。
- 风险等级映射、空库状态和模型输出结构校验。
- 资料版本快照和审核结果聚合。

### 17.2 集成测试

- 非管理员无法调用资料中心写 API。
- 自然语言录入、AI 整理、人工修改和发布。
- 批量导入预检、部分错误、确认导入和后台处理。
- 平台规则导入、差异、启用、审核命中和回滚。
- 法律成功/舆情失败、法律失败/舆情成功、两者成功和两者失败。
- 舆情库为空、模型不可用和非法模型输出的降级。

### 17.3 端到端测试

- 普通管理员不使用 JSON 完成舆情案例发布。
- 管理员发布平台规则新版本，市场人员提交对应平台物料并看到版本依据。
- 市场人员从渐进式页面提交物料并查看双轴结果。
- 法务人员查看结果并完成现有复核流程。
- 管理员归档事件后，新审核不再检索该事件，历史审核仍可查看。

## 18. 验收标准

- 提交首屏只保留物料、行业和平台，低频字段可展开。
- 普通管理员不理解 JSON 也能完成舆情事件录入和发布。
- JSON 导入有模板、预检、逐条错误和重复处理，不自动覆盖或发布。
- 平台规则可以查看差异、生效版本、更新时间并回滚。
- 一次前端提交生成法律与舆情双轴结果，并显示使用的资料版本。
- 舆情库为空时不输出虚假低风险结论。
- 两个分析模块可以独立失败和重试。
- 所有资料发布、归档、恢复、启用和回滚操作可审计。
- 自动化测试覆盖权限、导入、版本、降级、聚合和完整业务流程。
- 法律与舆情分支并行运行，新增模块不会让正常审核耗时按两次模型调用简单相加。

## 19. 实施顺序

详细实施计划在本设计审核通过后单独编写。实施时按以下依赖顺序拆分：

1. 数据模型、迁移、状态机和权限。
2. 舆情与平台规则管理 API。
3. 管理员资料中心前端。
4. 平台规则 L4 引擎。
5. 舆情混合引擎与统一审核聚合。
6. 渐进式提交页与双轴结果页。
7. 回归测试、操作文档和 v0.4.2 发布验证。
8. 真实舆情 JSON 导入、校准和 v0.5.0 发布验证。
