# LexAd v0.6 界面、品牌档案与数据库模式修复设计

## 背景

v0.6 完成新界面与品牌库后，测试发现五类问题：登录页没有充分体现参考 HTML 的左右视觉结构；浏览器缩放时三栏布局中的右栏挤压正文；桌面端缺少容易发现的登出入口；品牌档案无法成功加载列表；深色模式中页面与面板颜色过近，信息层级不清。

本次修复维持现有 Vue、UnoCSS、FastAPI、SQLAlchemy 与 Alembic 技术栈，不引入大型 UI 框架，不改变登录页与登录后应用壳层相互隔离的架构。

## 已确认的现状

- `backend/.env` 当前未设置 `DATABASE_URL`，本地后端会使用 `backend/lexad.db`。
- 本地 SQLite 的 Alembic 版本为 `294f05fbf95c`，与当前迁移头一致。
- 本地库已有 3 个演示品牌和 21 个用户，但没有已关联品牌的物料。
- SQLite 与 Neon 是相互独立的数据源，项目不存在本地数据自动同步到 Neon 的机制。
- 品牌列表失败不是由于本地缺少 `brands` 表或演示品牌，而是 `/api/v1/brands` 请求未成功完成。
- 桌面侧边栏只显示版本号；退出登录仅存在于移动端抽屉中。
- 三栏布局从 1024px 起固定使用 280px 左栏和 380px 右栏，缩放后留给中栏的宽度不足。
- 深色样式把多种浅色背景工具类映射为同一个面板色，导致页面、卡片和内嵌区域失去层级。

## 目标

1. 登录页在桌面端准确采用参考 HTML 的左右视觉语言，同时保持移动端可用。
2. 页面缩放或视口变窄时，三栏内容不重叠、不遮挡、不产生不可用的窄正文。
3. 桌面端和移动端都提供明确、可访问的退出登录入口。
4. 品牌列表和档案加载具有明确状态、可诊断错误和重试能力。
5. 深色模式形成清晰的页面、卡片、抬升面板和内嵌区域层级。
6. 日常开发明确使用本地 SQLite；只有显式选择 Neon 时才连接云数据库。

## 非目标

- 不实现 SQLite 与 Neon 的实时或双向同步。
- 不重做整个前端布局系统，也不引入新的组件框架。
- 不将参考 HTML 中登录页与应用内容粘合的结构复制到项目中。
- 不在普通本地启动过程中自动迁移 Neon。
- 不自动将全部测试数据导入 Neon；如有需要，使用单独的一次性导入流程。

## 一、登录页视觉设计

### 桌面端

登录页继续由 `AuthLayout` 独立承载，不渲染 `AppShell`、桌面侧边栏或登录后的顶栏。

左侧品牌区采用参考 HTML 的视觉构成：

- 深蓝到天蓝的 135 度渐变背景；
- LexAd 标志、主标题、副标题和三项能力说明；
- 低透明度的法律符号与天平线稿作为纯装饰元素；
- 装饰元素设置 `pointer-events: none`，并从可访问性树中隐藏；
- 内容保持在装饰图层之上，不因窗口高度变化而被裁切。

右侧表单区宽度约 480px，使用独立面板背景。输入框、按钮和测试账号区域沿用项目令牌，避免复制参考 HTML 的内联样式。桌面高度不足时允许整页纵向滚动。

### 移动端

1024px 以下切换为上下布局。品牌区仅保留标志、标题和副标题，隐藏大型天平装饰与能力列表；表单区占满可用宽度。手机端最小触控高度保持 44px。

## 二、响应式布局设计

桌面侧边栏继续在 1024px 以上固定显示，宽度保持由统一令牌控制。三栏审查布局使用独立的宽屏断点：

- 1280px 及以上：显示三栏。左栏宽度受约束，右栏使用 `clamp` 或等价的最小/理想/最大宽度，中栏使用 `minmax(0, 1fr)`。
- 1024px 至 1279px：三栏转为单列内容流，依次显示摘要、正文和执行报告；取消左右栏固定宽度与独立高度。
- 1024px 以下：继续使用移动导航和单列内容。

浏览器放大会降低 CSS 视口宽度，因此会自然触发中间断点。所有列在单列状态下使用页面滚动，不保留内部固定高度或独立滚动条。

## 三、深色主题设计

深色主题改为语义化表面层级：

| 用途 | 建议颜色 |
| --- | --- |
| 页面底色 | `#070D19` |
| 导航与应用外壳 | `#0B1220` |
| 普通卡片与面板 | `#111B2E` |
| 抬升面板、弹层、选中区 | `#17243A` |
| 内嵌内容区域 | `#0D1627` |
| 边框 | `#2A3A55` |
| 悬停背景 | `#1D2D47` |
| 主文字 | `#E8EEF7` |
| 次要文字 | `#B7C4D6` |
| 弱化文字 | `#7F91AA` |

`bg-white`、`bg-gray-50` 和 `bg-gray-100` 不再统一映射到同一个颜色，而是分别对应普通面板、内嵌区域和次级面板。优先让通用 `.card`、`.input`、状态徽章和布局组件使用语义令牌；仅为尚未迁移的 UnoCSS 工具类保留有限的兼容映射。

蓝、绿、黄、红状态色在深色模式下使用较亮文字和低透明度背景，保证可读性但避免大面积高饱和色块。焦点、悬停、选中和禁用状态分别具有可辨识反馈。登录页左侧品牌渐变保持品牌视觉，右侧表单和测试账号区域使用不同的深色表面层级。

## 四、退出登录设计

桌面顶栏在用户信息旁提供始终可见的“退出登录”图标与文字按钮，不把该操作隐藏在只有头像提示的菜单中。移动端继续在导航抽屉底部显示退出登录。

两个入口调用同一个用户状态方法：清除 Pinia 用户状态与 `access_token`，然后使用路由替换导航到 `/login`，避免用户通过浏览器后退直接回到受保护页面。按钮提供可访问名称和键盘焦点样式。

## 五、品牌档案加载设计

品牌页面使用明确的互斥状态：

- `loading`：首次请求或主动重试正在进行；
- `ready`：请求成功并展示品牌；
- `empty`：请求成功但没有品牌；
- `error`：请求未成功，保留错误类型和用户可理解的消息。

错误分类如下：

- 无 HTTP 响应：提示无法连接本地后端，并显示重试按钮；
- 401：由统一拦截器清理认证状态并返回登录页；
- 403：提示当前账号没有相应权限；
- 5xx：显示后端返回的安全错误信息，允许重试；
- 档案请求失败：品牌列表继续可用，只在右侧档案区显示错误和重试入口。

后端品牌列表端点使用具体的响应模型，而不是无约束的 `list`，确保返回结构稳定。前端不把网络失败误显示为“空品牌库”。搜索期间保留已有列表，失败时不清空成功加载的数据。

## 六、本地 SQLite 与 Neon 模式

### 本地模式

日常一键启动默认选择 `local`：

1. 明确解析到 `sqlite:///./lexad.db`；
2. 启动后端前执行本地 `alembic upgrade head`；
3. 数据库首次创建时，以幂等方式补充测试用户和演示品牌；
4. 迁移或初始化失败时终止启动，并输出明确错误，不继续报告服务可用。

### Neon 模式

Neon 连接必须显式选择 `neon` 并提供连接串。连接串缺失时立即拒绝启动。Neon 迁移使用独立、人工触发的命令执行，不纳入普通本地一键启动。

配置和 Alembic 必须调用同一数据库 URL 解析逻辑，避免应用连接 SQLite、迁移命令却连接 Neon，或反之。生产环境禁止静默回退 SQLite。

### 数据迁移边界

测试期创建的品牌只存在本地 SQLite。准备正式环境时先迁移 Neon 表结构，再根据需要执行一次性数据导入。正式服务切换到 Neon 后，新品牌直接写入 Neon，不经过本地 SQLite。离线同步或冲突合并属于后续独立功能。

## 七、错误处理与可观测性

- 启动输出明确显示当前数据库模式和脱敏后的目标类型，例如 `local SQLite` 或 `Neon PostgreSQL`。
- 禁止在日志、前端错误或脚本输出中展示数据库密码和完整连接串。
- 本地迁移失败、后端健康检查失败和前端启动失败分别报告。
- 品牌页面错误消息面向用户；详细异常保留在后端日志中。
- Neon 连接失败与迁移落后使用不同提示，避免把网络问题误判为数据库结构问题。

## 八、测试与验收

### 前端

- 对登录页分别验证宽屏、1024px 附近和手机宽度，确认登录页与应用壳层分离。
- 在 100%、125%、150% 和 200% 浏览器缩放下验证审查详情页无横向遮挡。
- 验证浅色、深色及跟随系统三种主题，重点检查页面、卡片、内嵌区、弹层、输入框和状态色对比。
- 验证桌面和移动端登出均清除令牌并进入登录页。
- 模拟成功、空数组、无响应、401、403 和 500，验证品牌列表状态与重试行为。
- 执行 `npm run build`。

### 后端与数据库

- 在全新临时 SQLite 上执行完整 Alembic 迁移并运行幂等种子两次。
- 验证本地品牌列表返回演示品牌，品牌档案端点返回稳定结构。
- 验证 `local` 模式不会读取 Neon 连接；`neon` 模式缺少连接串时拒绝启动。
- 使用只读状态检查确认 Neon 当前迁移版本；迁移 Neon 前必须单独确认。
- 执行后端测试集，并补充品牌 API 响应模型与数据库模式测试。

## 九、实施边界与建议顺序

1. 修复登录路由出口并完成登录页视觉与深色令牌。
2. 修复三栏响应式布局。
3. 增加统一登出入口。
4. 完善品牌 API 类型、前端状态与错误处理。
5. 实现显式数据库模式和本地启动迁移流程。
6. 完成构建、后端测试、临时 SQLite 迁移和浏览器验收。

每一步保持小范围变更，不重写现有页面，不覆盖用户当前未提交的脚本与布局修改；实施时先核对工作区差异并在现有修改上增量调整。
