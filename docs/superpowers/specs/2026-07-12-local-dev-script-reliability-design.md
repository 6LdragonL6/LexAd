# LexAd 本地一键启停可靠性修复设计

## 背景与根因

当前 `scripts/start-dev.ps1` 先创建前后端窗口，再执行最长 30 秒的健康检查，最后才写入 `scripts/.dev-pids.json`。因此健康检查期间执行 `stop-dev.bat` 时，关闭脚本无法识别已经创建的进程，只能提示没有已记录的服务。

此外，现有启动脚本即使健康检查失败也继续输出 `Started LexAd dev services`，无法可靠区分“进程窗口已创建”和“应用已可用”。浏览器在后端尚不可用时访问前端，可能呈现空白或不可操作页面。

实际浏览器验证还发现独立的前端白屏原因：`AuthLayout.vue` 是 `/login` 的父路由组件，却使用普通 `<slot />`，导致 Vue Router 匹配到的 `LoginPage.vue` 无处渲染。认证布局应使用 `<router-view />` 承载子路由。

## 目标

- 前后端进程一经创建就立即记录，后续健康检查失败也能通过关闭脚本清理。
- 只有后端和前端均通过健康检查后，才自动打开前端页面。
- 启动失败时返回非零退出状态，并明确指出失败的服务和日志查看位置。
- 关闭脚本只终止本次脚本记录且身份匹配的进程树，避免误伤其他项目。
- 关闭完成后复查 8000 和 5173 端口，并分别报告实际状态。
- `/login` 能渲染登录页内容，不再只显示空的认证布局。

## 方案

保留现有入口和进程模型：根目录 `.bat` 文件调用 `scripts` 下的 PowerShell 脚本，前后端继续运行在两个独立日志窗口中。

### 启动流程

1. 验证项目目录、Python、npm、依赖目录和目标端口。
2. 启动后端与前端 PowerShell 窗口。
3. 立即把两个窗口的 PID、启动时间、服务名和端口写入 `scripts/.dev-pids.json`。
4. 在限定时间内轮询后端 `http://localhost:8000/health` 和前端 `http://localhost:5173`。
5. 两项检查均成功后，使用系统默认浏览器打开 `http://localhost:5173`，并以成功状态结束。
6. 任一检查超时或子进程提前退出时，不打开浏览器；保留 PID 文件，打印对应失败信息和 `stop-dev.bat` 清理提示，并以非零状态结束。

PID 文件写入采用临时文件替换，避免中断时留下半截 JSON。

### 关闭流程

1. 读取 PID 文件；若不存在，明确说明“没有脚本记录”，但不把它等同于“端口未开启”。
2. 对每条记录校验 PID 和进程启动时间，防止 PID 被系统复用后误杀无关进程。
3. 对身份匹配且仍运行的根进程执行进程树终止；已退出的服务单独报告。
4. 等待短暂释放窗口后，复查记录中的端口。
5. 仅当记录进程均已停止且目标端口均已释放时删除 PID 文件；否则保留记录并明确报告残留服务或端口。

关闭脚本不加入“按端口直接强杀”的兜底，以避免终止其他项目。

### 登录页渲染

保持现有嵌套路由结构，将 `AuthLayout.vue` 的 `<slot />` 替换为 `<router-view />`。不调整登录表单、权限守卫或 API 请求。

## 错误处理与输出

- 健康检查代码与注释保持独立行，脚本文件继续使用 UTF-8 编码。
- 健康检查捕获连接错误，但最终必须输出后端、前端各自的成功或失败状态。
- 浏览器打开失败不视为服务启动失败；脚本提示用户手动访问前端地址。
- PID 文件无法解析时，关闭脚本报告文件损坏并保留它供排查，不静默删除运行证据。
- 输出区分“未记录服务”“记录进程已退出”“PID 身份不匹配”“端口仍被占用”。

## 验证

- PowerShell 语法解析无错误，且 `$timeout` 在使用前赋值。
- 正常启动时两个健康检查成功、PID 文件有效、浏览器只打开一次。
- 后端故意启动失败时不打开浏览器，启动命令返回非零，关闭脚本仍能关闭前端窗口。
- 前端故意启动失败时具有对称行为。
- 正常关闭后两个窗口结束、8000/5173 均释放、PID 文件删除。
- 重复关闭时只提示没有脚本记录，不声称端口一定未开启。
- 构造启动时间不匹配的 PID 记录时，关闭脚本跳过该进程且保留 PID 文件。
- 浏览器访问 `/login` 时能看到登录表单，认证布局不再是高度为零的空节点。

## 不在本次范围

- 除认证布局的子路由出口外，不改变前后端应用代码、API 地址或业务逻辑。
- 不自动安装 Python 或 npm 依赖。
- 不强制终止未由本脚本可靠记录的端口占用进程。
