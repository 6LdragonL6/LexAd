# LexAd v0.7.1 品牌创建、启动稳定性与品牌记忆补充设计

日期：2026-07-15

状态：已确认，待实施

## 1. 背景与目标

v0.7.1 发布后发现三个需要补充的问题：

1. 提交页创建新品牌失败时，品牌名称会立即从输入框消失，用户无法理解失败原因或直接重试；
2. Windows 一键启动脚本偶发报告后端健康检查失败，后端窗口有时已经退出，但主脚本只能输出笼统的 healthy 错误；
3. 品牌档案虽然已有指标和审核记录，但缺少与审核界面同源、可直接理解的品牌“现有记忆印象”。

本补充更新的目标是：让品牌创建失败可恢复、让一键启动优先稳定并能说明真实失败状态，以及用确定性历史归纳在审核界面与品牌档案之间建立统一品牌记忆口径。

## 2. 范围与非目标

### 2.1 本次范围

- 修复提交页品牌创建的异步状态和错误呈现；
- 加强 Windows `start-dev.ps1` 的稳定模式、退出检测和健康检查诊断；
- 在现有品牌档案响应中增加确定性的品牌记忆印象；
- 复用品牌记忆组件，并为审核场景和档案场景提供紧凑、完整两种展示；
- 补充后端、前端和脚本验证，并更新 v0.7.1 发布说明。

### 2.2 非目标

- 不新增品牌记忆数据库表或持久化快照；
- 不调用 AI 生成品牌总结；
- 不自动修改品牌资料或审核结论；
- 不改变品牌创建 API 路径和现有响应结构；
- 不把历史品牌记忆计入当前物料的 AI 风险评分；
- 不重构现有 PID 管理、数据库模式或停止脚本。

## 3. 总体方案

采用共享品牌记忆模型方案：后端在读取品牌档案时，根据现有已完成审核和法务决定生成 `memory_impression`；前端共享 `BrandMemoryCard`，在审核、提交和结果页使用紧凑模式，在品牌档案使用完整模式。

品牌创建继续由提交页调用 API，但选择器改为完全受控的异步交互：未收到成功结果前不清空输入或关闭下拉框。启动脚本默认关闭 Uvicorn `--reload`，并为需要热重载的开发者保留显式参数。

## 4. 品牌创建交互

### 4.1 现有问题

`BrandSelector` 当前在触发 `create` 事件后立即清空 `searchText` 并关闭下拉框。Vue 事件不会等待父组件的异步 API 调用，因此即使创建失败，用户输入也已经丢失，错误提示还可能因为下拉框关闭而不可见。

### 4.2 受控异步状态

- `SubmitPage` 持有 `brandCreating` 和 `brandCreateError`；
- `BrandSelector` 接收创建中状态和独立创建错误；
- 点击创建后，选择器只发出事件，不自行清空文字或关闭下拉框；
- 创建过程中禁用重复创建，输入文字保持可见；
- 创建成功后，父页面更新 `modelValue`，选择器监听到品牌已选中后再清空文字并关闭下拉框；
- API 返回 `created=false` 时视为成功复用，直接选中已有品牌；
- 创建失败时保留文字与下拉框，在创建操作附近显示错误，用户可以用原文字直接重试；
- 搜索错误与创建错误分别维护，互不覆盖。

### 4.3 成功标准

- 创建成功的新品牌立即成为当前物料品牌并加载品牌档案；
- 创建失败后名称不消失，错误在当前操作区域可见；
- 连续点击不会产生重复请求；
- 选择已有品牌、清空品牌和本地最近品牌恢复行为保持不变。

## 5. 一键启动稳定性

### 5.1 稳定模式

Windows 一键启动默认使用单 Uvicorn 服务进程，不附加 `--reload`。前端 Vite 仍保留热更新。需要调试后端代码时，开发者可显式执行：

```powershell
.\scripts\start-dev.ps1 -BackendReload
```

`-BackendReload` 是开关参数，默认关闭。现有 `-DatabaseMode local|neon` 行为不变。

### 5.2 健康检查

- 后端与前端检查统一使用 `127.0.0.1`，减少 Windows `localhost` 解析差异；
- 总等待时间由 30 秒延长至 60 秒；
- 保留轮询，不因单次连接拒绝立即失败；
- 保存每个服务最后一次健康请求错误摘要；
- 后端命令退出时写入仅属于本次启动的退出状态标记，包含退出码和时间；
- 主脚本发现后端已经退出时立即结束等待，输出“后端命令已退出”和退出码，并提示查看后端窗口；
- 如果进程仍在但最终超时，输出最后一次请求错误，而不是只有“port not responding”；
- 不自动无限重启后端，避免真实配置、导入或数据库错误被重试噪声掩盖。

退出状态标记在新一轮启动前清理，并使用本次启动标识避免旧状态误判。PID 文件和现有停止脚本仍是进程清理的事实来源。

### 5.3 成功标准

- 普通一键启动不创建 Uvicorn 重载子进程；
- 后端启动较慢时最多等待 60 秒；
- 后端提前退出时，主脚本能区分“已退出”和“仍未健康”；
- 错误输出包含足够信息指导用户查看后端窗口；
- 显式启用后端热重载时仍可正常启动。

## 6. 品牌记忆模型

### 6.1 数据来源

品牌记忆只使用现有数据：

- 未删除且关联该品牌的物料；
- `task_status=completed` 的审核；
- 已提交的法务决定；
- 审核结果中的已验证风险和 `ai_result.suggestions`；
- 物料版本数和管理员确认的品牌行业。

不读取正在执行或失败且无完整结果的审核，不调用模型，也不写回数据库。

### 6.2 响应结构

`BrandProfile` 新增带默认值的 `memory_impression`，至少包含：

- `status`：`collecting`、`stable`、`mixed` 或 `attention`；
- `headline`：整体记忆标签；
- `summary`：固定规则生成的简短解释；
- `completed_review_count` 和 `decided_review_count`：样本依据；
- `recent_decisions`：最近五次有效决定中通过、条件通过、退回的数量；
- `avg_versions`：平均提交轮次；
- `revision_tendency`：修改成本提示；
- `frequent_risks`：最多五项高频风险及次数；
- `common_suggestions`：最多五项常见修改建议及次数；
- `industries`：管理员确认的常用行业。

字段使用明确 schema，不以无约束字典代替。

### 6.3 确定性规则

- 少于 3 次法务决定时，状态固定为 `collecting`，标题为“样本积累中”，不形成稳定评价；
- 3 次及以上，根据整体通过率与最近五次决定归纳：
  - 通过率不低于 80%，且最近五次退回不超过 1 次：`stable`；
  - 通过率低于 50%，或最近五次退回至少 3 次：`attention`；
  - 其他情况：`mixed`；
- 平均提交轮次不高于 1.5 为“修改轮次较少”，1.5 至 2.5 为“通常需要一轮以上调整”，高于 2.5 为“经常需要多轮修改”；
- 高频风险沿用现有计数口径，法务退回审核的风险权重为 2，其他已完成审核权重为 1；
- 常见建议对历史 `ai_result.suggestions` 做去首尾空白、合并连续空白、去重和计数；同一审核内相同建议只计一次；
- 所有列表按次数降序、文字稳定排序，确保相同数据产生相同输出。

### 6.4 表述边界

品牌记忆统一注明“历史参考，不参与本次 AI 风险评分”。摘要不能表述为品牌合法、已获认证或未来必然通过。没有足够样本时明确显示样本不足，不使用中性通过率制造稳定印象。

## 7. 界面设计

### 7.1 共享组件

`BrandMemoryCard` 增加 `mode="compact" | "full"`：

- `compact`：保留审核界面当前体量，展示总体标签、样本量、通过率、常用行业和最多三项高频风险；
- `full`：展示整体摘要、近期决定分布、修改轮次、高频风险、常见建议、行业范围和样本说明。

提交页、结果页和法务详情使用紧凑模式；品牌档案使用完整模式。两种模式读取同一 `memory_impression`，不存在两套归纳逻辑。

### 7.2 品牌档案布局

完整记忆卡位于品牌档案右侧内容区顶部，在管理操作和历史列表之前。现有品牌基础信息、最近审核和通过物料继续保留；原审核指标和高频违规区块合并进完整记忆卡，避免相同数据重复出现。

`collecting` 使用中性色，`stable` 使用低饱和绿色，`mixed` 使用琥珀色，`attention` 使用红色风险语义。明暗模式分别使用现有主题色、边框和表面变量，不引入只适用于浅色背景的固定色块。

## 8. 错误处理与兼容性

- `memory_impression` 提供默认值，旧前端或无历史品牌不会因字段缺失失败；
- 品牌档案归纳单个异常建议时跳过无效项，不影响整个档案响应；
- 创建品牌 API 路径、HTTP 状态和 `BrandCreateResponse` 不变；
- 启动脚本现有数据库预检、迁移、种子、端口验证和 PID 记录逻辑不变；
- 启动失败不自动删除 PID 记录，继续提示使用停止脚本清理；
- 本补充更新不需要 Alembic 迁移。

## 9. 测试与发布验证

### 9.1 后端

- 无历史品牌返回 `collecting` 和零样本；
- 少于三次决定不形成稳定评价；
- 稳定通过、表现波动和重点关注规则分别命中；
- 最近五次决定计数正确；
- 高频风险保留退回权重；
- 常见建议在单次审核内去重，并跨审核累计；
- 品牌档案 schema 可序列化完整记忆结构；
- 完整后端回归通过。

### 9.2 前端

- 创建成功后清空输入并选中新品牌；
- `created=false` 时选中已有品牌；
- 创建失败时保留原文字、下拉框和错误，可再次提交；
- 创建过程中禁止重复请求；
- 紧凑和完整记忆卡均可处理空样本与完整样本；
- TypeScript 检查通过，明暗模式类保持对称。

### 9.3 启动脚本

- PowerShell 语法解析通过；
- 默认后端命令不包含 `--reload`；
- `-BackendReload` 时包含 `--reload`；
- 健康检查使用 `127.0.0.1` 且等待 60 秒；
- 当前启动标识可以区分旧退出标记；
- 后端提前退出和健康超时输出不同诊断。

### 9.4 发布文档

- 版本保持 v0.7.1，作为补充更新记录；
- README 主要展示思路不变，只在质量状态或版本说明中做必要补充；
- 发布说明记录品牌创建、启动稳定性和品牌记忆变化；
- 执行 `git diff --check` 和文档链接检查；
- 尝试生产构建，若执行环境限制 esbuild，则准确记录未验证状态。

4. 扩展共享记忆卡并整理品牌档案重复区块；
