# LexAd v0.7.2 审核进度、双评分与导航一致性设计

日期：2026-07-15

状态：已确认，待实施

## 1. 背景与目标

v0.7.2 解决审核主流程中的四类一致性问题：

1. 用户提交物料后只能看到笼统的等待状态，不知道 AI 审核正在执行哪个真实阶段；
2. 法务面对已经发现风险问题的物料时，可能误点“通过”并直接提交；
3. 工作台与法务审核台进入详情后的返回逻辑不一致，部分状态下缺少返回入口；
4. 法规审核分与舆情审核分的方向和字段命名不一致，`ai_risk_score`、`risk_score` 同时承载不同含义，容易造成业务代码和用户理解混乱。

本次更新通过一次受控的局部重构建立清晰边界：两个评分都采用“越高越好”，审核进度只展示后端真实状态，带风险通过必须二次确认，详情页根据可信入口返回。新增和修改的界面必须同时适配浅色与深色主题。

## 2. 已确认的设计原则

- “有风险”定义为 AI 审核结果中至少存在一项已确认风险问题，不使用分数阈值推断；
- 审核阶段只使用真实后端状态，不使用按时间自动推进的伪进度；
- 法规合规分和舆情安全分均为 0–100，数值越高表示越安全或越合规；
- 工作台进入的详情返回工作台，法务审核台进入的详情返回法务审核台；
- 浏览器原生后退保持可用，不增加全局拦截；
- 前后端在 v0.7.2 同步切换到新评分字段，不保留旧 API 别名；
- 不在页面中重复堆叠状态判断，进度、评分、风险确认和返回逻辑分别由共享边界负责；
- 数据库迁移、接口、前端类型、历史记录和品牌档案必须使用同一评分语义。

## 3. 架构与代码边界

### 3.1 后端边界

后端增加三个明确职责：

1. 审核引擎只生成领域结果：法规引擎输出 `compliance_score`，舆情引擎输出 `safety_score`；
2. 审核服务负责将两个分数写入 `Review` 的正式数据库列，并维护模块状态；
3. 响应组装层根据审核和模块状态生成结构化进度，不由前端猜测后端业务。

评分转换、范围校验和进度映射均实现为可独立测试的纯函数。页面展示需要的进度结构使用明确 schema，不返回无约束字典。

### 3.2 前端边界

前端新增或提取以下共享单元：

- `ReviewProgress`：只负责展示结构化阶段列表和当前状态；
- `RiskApprovalDialog`：只负责带风险通过的二次确认；
- `useReturnNavigation`：只负责解析白名单入口和执行返回；
- 风险问题选择器：从审核结果统一提取已确认风险，结果页和法务页共用；
- 评分展示工具：统一分值范围、颜色和空值语义，不再出现相反方向的局部判断。

页面只组合这些单元并调用 API，不自行复制评分换算、来源校验或阶段推断逻辑。

## 4. 数据库与评分模型重构

### 4.1 正式字段

`reviews` 表调整为：

- `legal_compliance_score`：非空整数，0–100，越高越合规；
- `public_opinion_safety_score`：可空整数，0–100，越高越安全；无法形成可靠自动结论时为 `NULL`。

数据库增加范围约束，阻止写入小于 0 或大于 100 的值。法规与舆情继续作为两个独立维度，不生成混合总分。

### 4.2 字段切换

- 数据库列 `ai_risk_score` 重命名为 `legal_compliance_score`；
- 新增 `public_opinion_safety_score`；
- 法规引擎、服务、schema、队列、历史版本、品牌档案和前端类型全部改用 `legal_compliance_score`；
- 舆情模型提示、校验、服务和界面全部改用 `safety_score` 或 `public_opinion_safety_score`；
- 新 API 响应不再包含 `ai_risk_score` 或数值型 `risk_score`；
- `risk_level` 保留为低、中、高、严重等分类标签，但不再承担数值含义。

只有 v0.7.2 一次性迁移脚本允许引用旧字段名，以便识别和转换历史数据。迁移完成后的活动业务代码不得继续引用旧评分字段。

### 4.3 历史数据迁移

Alembic 迁移在事务中完成以下操作：

1. 重命名法规评分列；
2. 添加舆情安全分列及两个分数的范围约束；
3. 遍历历史审核：只有舆情模块成功、无需人工复核且旧 JSON 分数有效时，按 `100 - 旧风险分` 回填；
4. 舆情失败、不可用、待人工复核、证据不足或旧值无效的记录回填 `NULL`；
5. 从历史 `public_opinion_result` JSON 中移除数值评分键，保留风险等级、证据、解释、置信度和建议；
6. 迁移完成后校验所有非空评分均在 0–100 范围内。

迁移需兼容项目支持的 SQLite 和 PostgreSQL。降级路径根据正式列重建旧字段后再还原列名，确保结构可回退；任何无法可靠还原的人工复核分保持原有“无可靠自动分”语义，不伪造高低分。

### 4.4 新审核写入

法规引擎直接输出合规分。舆情模型和确定性降级逻辑直接输出安全分，不再先生成方向相反的分数再换算。舆情结果 JSON 不保存第二份数值评分，数据库正式列是唯一事实来源。

## 5. 真实审核阶段

提交接口创建审核记录后，前端立即进入结果页的处理中状态。后端响应提供四个有序阶段：

1. `accepted`：物料已接收；
2. `legal_review`：法规审核；
3. `public_opinion_review`：舆情审核；
4. `finalizing`：汇总审查结果。

每个阶段使用统一状态枚举：`pending`、`running`、`completed`、`manual_review`、`failed`。阶段映射规则如下：

- 审核记录存在后，物料接收阶段立即完成；
- 法规与舆情阶段直接映射各自模块状态；
- 两个模块均到达终态且总任务仍在处理时，汇总阶段为进行中；
- 总任务完成后，汇总阶段完成；
- 模块需要人工复核时保留真实的 `manual_review` 状态，不伪装为自动成功；
- 总任务失败时，失败模块和汇总阶段显示失败，并保留重试入口；
- 超过现有长时间阈值时只显示“耗时较长”和返回入口，不自动推进阶段。

`ReviewProgress` 只为 `running` 阶段播放轻量脉冲或流动动画，已完成阶段使用勾选，等待、人工复核和失败使用稳定图标。动画遵守系统减少动态效果设置。

## 6. 带风险通过的二次确认

法务提交决定前统一提取四个法规层级中的已确认 `matched_rules`。当且仅当满足以下条件时打开二次确认：

- 决定为 `approved`；
- 已确认风险问题数大于 0。

弹窗显示风险总数和最高风险等级，主提示明确说明 AI 已发现风险。默认安全操作为“取消并继续复核”，危险操作文案为“确认带风险通过”。第一次点击“提交决定”只打开弹窗，第二次明确确认后才调用 API。

退回、有条件通过和无风险通过不触发弹窗。提交期间同时锁定原按钮和弹窗确认按钮，防止重复请求。提交失败时弹窗保持打开，决定、备注和退回原因均保留，用户可直接重试。

## 7. 返回与浏览器历史

详情入口只传递白名单来源标识：`workbench` 或 `legal-dashboard`，不接受任意 URL。

- 工作台打开审核结果时携带 `workbench`；
- 法务审核台打开审核详情时携带 `legal-dashboard`；
- 新提交或重新提交进入结果页时使用 `workbench`；
- 页面内返回按钮根据可信来源返回对应命名路由；
- 直接访问、刷新或来源缺失时，按角色选择工作台或法务审核台作为安全默认值；
- 结果页和法务详情页在加载、处理中、成功和失败状态下始终提供返回按钮。

正常进入详情继续使用路由 `push`，因此浏览器原生后退仍返回入口页面。页面不注册全局后退拦截。审核轮询在组件卸载时停止，浏览器后退不会留下定时请求。

法务决定提交成功后使用 `replace` 返回法务审核台，避免浏览器后退重新打开已经提交的决定表单。页面内普通返回按钮不改变浏览器自身的后退能力。

## 8. 明暗主题与可用性

本次新增的进度组件、确认弹窗和返回按钮，以及被修改的结果页、法务详情页和审核台，必须同时检查浅色与深色主题：

- 表面、遮罩、边框、正文和次要文字均提供成对主题样式；
- 成功、进行中、警告、人工复核和失败不能只依赖颜色区分，必须同时有图标或文字；
- 弹窗具备标题、说明、明确按钮顺序、键盘焦点和 Escape 取消；
- 打开弹窗后焦点进入弹窗，关闭后返回原提交按钮；
- 移动端按钮保持足够点击区域，长物料名称和风险说明不破坏布局；
- 当前阶段动画在深色背景上不过亮，并支持 `prefers-reduced-motion`。

## 9. 错误处理

- 单个模块失败时展示该模块真实错误，不使用虚构分数填补；
- 舆情需要人工复核时安全分为 `NULL`，界面显示“待人工复核”而不是 0 分；
- 轮询短暂失败继续使用现有自动重试和网络提示，连续失败后仍保留返回入口；
- 评分超出范围时后端拒绝写入并记录可诊断错误；
- 非法返回来源被忽略并使用角色默认页；
- 决定 API 失败不丢失法务输入，也不提前导航；
- 前后端作为同一 v0.7.2 发布单元部署，避免新旧契约长期混用。

## 10. 测试与反复审查门槛

### 10.1 数据库与后端

- SQLite 迁移升级、降级和再次升级烟雾测试；
- PostgreSQL 目标环境迁移验证；
- 旧法规分列正确重命名且数值不变；
- 可靠舆情历史按反向关系正确回填；
- 人工复核、失败、不可用和无效历史分回填为空；
- 历史 JSON 数值评分被清理，证据和说明保持不变；
- 两个评分的数据库范围约束生效；
- 新审核直接写入两个正向分数；
- API、队列、历史版本与品牌档案只返回新字段；
- 进度状态组合矩阵覆盖等待、运行、完成、人工复核和失败。

### 10.2 前端

建立最小 Vitest 测试环境，覆盖：

- 风险问题提取与“有风险通过”触发条件；
- 无风险、退回和有条件通过不触发确认；
- 弹窗确认、取消、API 失败和重复提交锁定；
- 四阶段全部状态及长时间运行提示；
- 工作台、法务审核台、直接访问和非法来源的返回目标；
- 浏览器后退不受拦截，法务提交成功不返回旧表单；
- 两个评分越高越好的颜色和标签；
- 舆情空分显示人工复核，不显示 0 分；
- 关键组件在浅色、深色和移动端的渲染状态。

### 10.3 三轮遗留审查

发布前执行三轮不同目标的审查：

1. **契约审查**：逐项核对数据库模型、Alembic、Pydantic schema、API 响应、TypeScript 类型和组件 props；
2. **遗留引用审查**：全仓扫描 `ai_risk_score`、数值型 `risk_score` 及旧文案，活动代码必须为零命中；只允许迁移脚本和历史发布文档保留必要引用；
3. **用户流程审查**：按营销、法务、管理员角色完整走查提交、轮询、查看、返回、浏览器后退、决定确认和历史记录。

额外执行数据库迁移前后行数与关键字段抽样对比、`git diff --check`、后端完整测试、前端单元测试、TypeScript 检查和生产构建。任何旧字段残留、评分方向不一致、缺失返回按钮或主题不可读都阻止发布。

## 11. 版本发布与回退

- 前端包版本、后端应用版本、健康信息、README 和发布说明统一更新为 0.7.2；
- 新增 `docs/releases/v0.7.2-release-notes.md`；
- 数据库迁移前在目标环境创建可恢复备份；
- 前后端协调发布，避免旧前端请求新 API；
- 应用回退必须与 Alembic 降级配套执行，不能只回退代码；
- 发布后抽查无风险、带风险和人工复核三类历史与新增审核。

## 12. 非目标

- 不生成法规与舆情混合总分；
- 不改变风险规则、法律判定阈值或舆情证据标准；
- 不把阶段动画改成服务器推送或新增任务队列；
- 不重构知识库、品牌创建、权限体系和无关页面；
- 不保留旧评分 API 的长期兼容层。

## 13. 验收标准

1. 提交后能看到与后端一致的四阶段进度，且没有伪进度；
2. 法规合规分与舆情安全分均明确为越高越好，无可靠舆情结论时不显示伪造分数；
3. 活动业务代码、API 和前端类型不再使用含义混乱的旧评分字段；
4. 有至少一项已确认风险时，“通过”必须经过明确二次确认；
5. 每个物料详情状态都有返回按钮，返回目标与入口一致，浏览器后退正常；
6. 法务决定成功后不能通过后退回到可重复提交的旧表单；
7. 新增和修改界面在浅色、深色、桌面和移动端均可正常使用；
8. 数据迁移可验证、可降级，历史记录没有丢失证据或改变决定；
9. 三轮遗留审查和全部自动化验证通过后才允许发布 v0.7.2。
