# Codex 可视化讨论工具链设计

## 1. 目标

为 Codex 中的界面流程讨论建立可重复、可验证的本地可视化流程：

1. 自动启动仅监听 `127.0.0.1` 的临时服务。
2. 通过真实 HTTP 请求确认服务可用。
3. 在 Codex 应用内浏览器中自动打开可见标签页，不修改操作系统默认浏览器。
4. 在展示前验证中文内容未发生编码损坏。
5. 读取用户选择后，自动关闭标签页和临时服务。
6. 任一步骤失败时执行清理并回退到纯文本讨论。

## 2. 已确认的问题

原流程存在两个独立缺陷：

- 后台 Node.js 进程可能在工具调用结束后被回收，但残留的 `server-info` 仍会使调用方误判服务已经启动。
- HTML 文件本身是有效 UTF-8，且模板与响应头声明 UTF-8；因此中文异常必须通过完整链路检查定位，不能只检查源文件或依赖人工观察。

## 3. 方案选择

采用“应用内会话托管”方案。Codex 的持久会话直接持有 HTTP 服务句柄和浏览器标签页引用，不再依赖会被回收的外部后台进程。

不采用以下方案：

- 外部后台进程：无法保证跨工具调用持续存活。
- 直接打开本地 HTML：无法可靠记录交互事件，且受浏览器本地文件安全策略限制。

## 4. 会话状态

每次可视化讨论创建一个独立会话，状态至少包含：

- `sessionId`：随机会话标识。
- `sessionToken`：只用于当前本地会话的随机令牌。
- `serverHandle`：可主动关闭的 HTTP 服务句柄。
- `port`：操作系统分配的空闲端口。
- `baseUrl`：仅使用 `http://127.0.0.1:<port>`。
- `contentDir`：本次讨论的 HTML 内容目录。
- `stateDir`：健康状态和交互事件目录。
- `tabId`：Codex 应用内浏览器标签页标识。
- `status`：`starting`、`ready`、`shown`、`closing` 或 `closed`。

同一 Codex 会话同一时间只保留一个活动可视化讨论会话。启动新会话前必须清理旧会话。

## 5. 启动流程

1. 创建独立内容目录和状态目录。
2. 生成随机 `sessionId` 与 `sessionToken`。
3. 以 UTF-8 无 BOM 写入页面片段，并加入固定中文哨兵文本：
   `中文编码校验：广告舆情风险`。
4. 用严格 UTF-8 解码器回读源文件。遇到非法字节或替换字符立即失败。
5. 在持久会话中启动 HTTP 服务，并让操作系统分配空闲端口。
6. 请求 `GET /health?token=<sessionToken>`，必须同时满足：
   - HTTP 状态码为 200；
   - 返回的 `sessionId` 与当前会话一致；
   - 返回状态为 `ready`。
7. 将 Codex 应用内浏览器设为可见，创建新标签页并导航到 `baseUrl`。
8. 等待 `DOMContentLoaded` 后检查：
   - 当前 URL 与 `baseUrl` 一致；
   - `document.characterSet` 为 `UTF-8`；
   - 页面标题符合预期；
   - 页面可见文本包含完整中文哨兵；
   - 页面不包含 Unicode 替换字符 `U+FFFD`。
9. 全部通过后才向用户宣布页面已经打开。

`server-info` 只作为诊断信息，不作为服务存活依据。

## 6. 交互流程

- 每个可选项使用稳定的 `data-choice` 标识。
- 用户点击后，将事件发送到同源服务；事件包含 `sessionId`、`choice`、显示文本和时间戳。
- 服务只接受携带当前 `sessionToken` 的同源请求。
- 用户回到对话后，Codex读取当前会话事件；终端文字反馈优先，点击事件作为结构化补充。
- 推送新画面前清空上一个画面的事件，并使用新的文件名，避免旧页面和旧选择混淆。

## 7. 关闭流程

以下任一条件触发关闭：

- 用户完成当前视觉选择并回到文字讨论；
- 用户明确取消；
- 启动或展示过程失败；
- 会话超过 30 分钟无活动；
- 新可视化会话准备启动。

关闭顺序：

1. 将状态改为 `closing`，停止接受新事件。
2. 关闭本次流程创建的应用内浏览器标签页；不得关闭用户原有标签页。若应用内浏览器强制保留最后一个内部页，则隐藏浏览器、释放本次标签页引用，并明确记录该限制，不将其报告为“标签页已完全关闭”。
3. 关闭 HTTP 服务并等待关闭完成。
4. 对原端口执行一次连接检查，确认已不可访问。
5. 将状态改为 `closed`。这里的完成标准是本次页面不再对用户可见且本地服务已经关闭，不要求应用内浏览器的内部标签页计数归零。

关闭操作必须可重复调用；标签页或服务已经关闭时仍视为成功。

## 8. 错误处理与降级

任何阶段失败都执行统一清理，不留下可访问端口或空白标签页。

- 服务无法启动：不打开浏览器，直接回退纯文本。
- 健康检查失败：关闭服务，删除陈旧启动状态，回退纯文本。
- 浏览器无法打开：关闭服务，回退纯文本。
- 中文校验失败：保存诊断信息，关闭标签页和服务，回退纯文本。
- 事件提交失败：页面提示选择未保存，同时允许用户在对话中直接回复选项。
- 清理部分失败：报告具体残留项，不宣称已经完整关闭。

## 9. 编码规范

- HTML、CSS、JavaScript 和 JSON 均使用 UTF-8。
- HTML 必须包含 `<meta charset="utf-8">`。
- HTML 响应必须返回 `Content-Type: text/html; charset=utf-8`。
- JSON 响应必须返回 `Content-Type: application/json; charset=utf-8`。
- JSON 序列化保留中文字符，不依赖系统代码页。
- 不使用 PowerShell 默认编码写入页面；内容文件由受控 UTF-8 写入方式生成。
- 每次展示都执行源文件严格解码和浏览器 DOM 中文哨兵校验。

## 10. 安全边界

- 服务仅监听 `127.0.0.1`，不得绑定 `0.0.0.0`。
- 使用随机高位端口和随机会话令牌。
- 只允许读取当前会话内容目录中的文件。
- 拒绝路径穿越和非预期 HTTP 方法。
- 设置合理的请求体大小限制。
- 服务日志不得记录用户输入全文、令牌或敏感信息。

## 11. 验收测试

### 正常路径

1. 启动后健康检查成功。
2. 应用内浏览器自动显示页面。
3. 中文标题、说明和哨兵文本正确显示。
4. 点击选项后能读取正确的 `choice`。
5. 完成后标签页关闭，原端口拒绝连接。

### 故障路径

1. 陈旧 `server-info` 存在但端口未监听时，系统必须判定启动失败。
2. 页面包含非法 UTF-8 时，不得展示给用户。
3. 页面包含 `U+FFFD` 时，不得进入 `shown` 状态。
4. 浏览器导航失败时，服务必须关闭。
5. 重复关闭不得报错或误关用户原有标签页；应用强制保留最后一个内部页时，浏览器必须处于隐藏状态。
6. 已存在旧会话时，启动新会话前必须确认旧端口关闭。

### 系统影响

验证过程中不得更改 Windows 默认浏览器、文件关联、代理配置或其他全局系统设置。

## 12. 使用方法论

以后凡是使用可视化讨论功能，固定执行：

`生成 → 严格 UTF-8 回读 → 启动 → HTTP 健康检查 → 应用内打开 → DOM 中文校验 → 用户交互 → 读取事件 → 关闭标签页 → 关闭服务 → 端口复核`

只有完成“HTTP 健康检查”和“DOM 中文校验”后，才能告诉用户页面可用；只有完成“标签页关闭”和“端口复核”后，才能告诉用户流程已结束。
