# LexAd v0.5.0 Project Polish Design

Date: 2026-07-08

## Goal

After the v0.5.0 feature baseline is complete, perform a conservative quality pass that improves robustness, frontend consistency, and project documentation without changing the intended review workflow.

This pass is a release stabilization task, not a feature expansion or large refactor.

## Scope

In scope:

- Small backend robustness fixes around migration safety, null handling, idempotent import behavior, and user-facing error messages.
- Frontend consistency fixes for Chinese business labels, returned-material flows, empty states, tags, action buttons, and accidental exposure of internal identifiers.
- Documentation cleanup, including a readable README, a v0.5.0 release announcement, and clearer documentation entry points.
- Verification through existing backend tests, frontend type/build checks, and focused smoke checks for v0.5.0 risk areas.

Out of scope:

- Redesigning the review engine architecture.
- Rewriting the frontend component system.
- Moving large directory trees.
- Adding new product features beyond robustness fixes.
- Changing review conclusions unless a clear defect is found.

## Backend Plan

Review the v0.5.0 code paths introduced or changed for:

- Platform rule alias matching.
- L1 absolute-expression detection.
- Returned-material loop, archive, and delete behavior.
- Public-opinion trigger-word ingestion.
- Material version labels and legal return metadata.

Fix only narrow defects or brittle edges, such as:

- Migration transaction handling that may behave differently across PostgreSQL environments.
- Missing defensive handling for optional fields.
- Technical exception messages leaking into business-facing API responses.
- Import scripts that are not safely repeatable.

The backend API contract should remain compatible with the current frontend.

## Frontend Plan

Make the v0.5.0 user experience consistent across market, legal, and admin views:

- Keep business-facing pages in Chinese labels instead of pinyin or internal enum names.
- Show returned-material context consistently where market users naturally review their work.
- Use consistent labels for version, platform, status, return reason, legal notes, and archive actions.
- Keep internal IDs and technical details available only in technical-detail areas.
- Avoid introducing a new UI library or broad style system changes.

## Documentation Plan

Clean up the documentation surface:

- Replace the unreadable README content with a concise UTF-8 Chinese README.
- Update project version references to v0.5.0.
- Add `docs/v0.5.0-release-notes.md` for testers and team members.
- Keep historical design and iteration documents, but make the main entry points easier to follow.

## Verification Plan

Run:

- `python -m pytest app/tests/ -q`
- `npm.cmd run build`
- Targeted smoke checks for:
  - L1 detection of examples such as `最耐用`.
  - Platform display labels such as `抖音` and `小红书`.
  - Returned-material reason and legal-note fields.
  - Public-opinion trigger-word scan output.

Known limitation:

- Live PostgreSQL migration validation may require network access to the configured database and may not be possible inside the sandbox. If blocked, report that explicitly and rely on local/static migration checks.

## Acceptance Criteria

- Existing backend and frontend checks pass.
- README is readable and points to the correct v0.5.0 documents.
- The release announcement clearly explains fixed bugs, new workflow behavior, verification commands, and remaining limitations.
- No ordinary business page exposes platform pinyin identifiers, raw UUIDs, or internal enum names unless placed under a technical-details section.
- Changes remain small and reviewable.
