# LexAd v0.7.1 审核版本、品牌行业与法务工作台设计

## 1. 文档信息

- 适用版本：LexAd v0.7.1
- 设计日期：2026-07-15
- 版本性质：兼容性修复与跨部门协作体验更新
- 核心范围：提交版本一致性、品牌继承、品牌多行业、法务双轴审核、审核台布局与发布文档

## 2. 背景与问题

v0.7.0 已建立资料库约束下的法律、平台与舆情审查能力，但重提与法务协作链路仍存在四类一致性问题：

1. 物料在第二次及以后提交后，工作台识别标题、详情标题、当前文案和历史审核结果可能来自不同版本，导致内容串版。
2. 重提页未恢复原物料品牌，却会向后端发送空品牌，后端据此解除品牌关联，使后续审核不再进入品牌档案。
3. 法务详情页当前把审查引擎放在右栏，核心内容的视觉权重不足，历史版本也没有位于直观的辅助区域。
4. 舆情结果已经由后端生成并在营销结果页展示，但法务详情的审查引擎没有消费该结果，法务无法基于完整双轴信息作出决定。

此外，品牌档案当前只有一个自由文本行业，而物料支持多个标准行业。管理员维护品牌行业时无法复用提交页选项，也无法基于已有物料记录判断是否应为品牌补充行业。

## 3. 目标与非目标

### 3.1 目标

1. 同一次审核所使用、展示和回溯的标题、文案、行业与平台均来自同一不可变提交快照。
2. 工作台和队列保留首次提交标题作为稳定识别名；详情显示所打开版本的实际提交名称与文案。
3. 重提沿用并锁定首次品牌，不允许通过前端或直接请求清空、更换品牌。
4. 一个品牌可关联多个标准行业；新行业由物料记录形成候选，再由管理员确认。
5. 法务详情把本次提交与完整审查引擎放在主中栏，并展示法律、平台和舆情两条风险轴。
6. 舆情状态参与法务决定的可提交条件，同时保留人工复核降级路径。
7. 发布 v0.7.1，并在不改变 README 主要展示思路的前提下更新版本信息与必要说明。

### 3.2 非目标

1. 不把每次提交全面重构为新的 Revision 聚合实体。
2. 不改变 v0.7.0 的法律评分规则、AI 裁决原则或舆情风险计算方法。
3. 不允许重提时迁移品牌，也不设计跨品牌历史记录转移。
4. 不让品牌行业自动限制物料的实际行业选择。
5. 不用当前物料内容补写或伪造缺失的旧审核快照。

## 4. 方案选择

采用“快照一致性修复”方案，而不是仅做前端补丁或完整版本模型重构。

- 前端补丁不能阻止直接 API 请求更换品牌，也无法从根本上避免历史审核读取当前物料。
- 完整 Revision 模型长期更纯粹，但会扩大接口、模型和迁移范围，不适合作为 0.7.1 修复版本。
- 本方案保留现有 Material、MaterialSubmissionSnapshot 和 Review 边界，通过稳定识别标题、原子重提接口、快照驱动审核以及品牌行业关系表消除当前问题。

## 5. 数据模型

### 5.1 稳定物料标题

为 `materials` 新增非空字段 `display_name`：

- 首次提交时，`display_name` 与 `name` 同值。
- `display_name` 是工作台、最近提交、法务队列和删除确认中的稳定识别标题。
- `name` 表示最新工作副本的提交名称，重提时可以修改。
- 迁移时将现有 `name` 回填到 `display_name`。

### 5.2 提交快照

`material_submission_snapshots` 继续保存每次提交的：

- `name`
- `raw_text`
- `industry`
- `platforms`
- `material_type`
- `priority`
- `deadline`

审核详情响应新增 `submission`，直接返回该 Review 绑定的快照。新审核不得在缺少快照时启动。旧审核缺少快照时进入兼容只读模式，明确提示数据不完整。

### 5.3 品牌与行业

新增 `brand_industries`：

- `brand_id`
- `industry`
- `created_by_id`
- `created_at`
- 唯一约束：`brand_id + industry`

新增 `brand_industry_suggestions`：

- `brand_id`
- `industry`
- `status`：`pending`、`accepted`、`ignored`
- `first_material_id`
- `latest_material_id`
- `occurrence_count`
- `first_seen_at`
- `last_seen_at`
- `reviewed_by_id`
- `reviewed_at`
- 唯一约束：`brand_id + industry`

品牌原有单值自由文本 `industry` 在迁移后不再作为业务数据源。其非空值拆分并写入正式品牌行业关联；字段可暂时保留用于迁移兼容，后续版本再移除。

### 5.4 标准行业目录

后端提供统一行业目录接口。提交页、品牌档案页和相关校验均使用同一目录，首批保持现有七类：食品、医疗、教育、汽车、金融、美妆、直播电商。接口返回稳定值与显示名称，前端不再各自硬编码或允许自由输入。

## 6. 提交与审核数据流

### 6.1 首次提交

1. 用户选择品牌、实际行业、平台并提交物料。
2. 后端写入 Material，固定 `display_name`，保留品牌关联。
3. 对每个物料行业检查品牌正式行业：未关联时按“品牌 + 行业”幂等创建或更新待确认候选。
4. 创建第 1 次提交快照与 Review。
5. 法律/平台分支与舆情分支均读取该快照，而不是重新读取可变 Material 内容。

### 6.2 重新提交

新增原子化重提接口，由一次请求完成：

1. 校验当前用户拥有操作权限，且物料状态为 `returned`。
2. 校验请求未清空或更换首次品牌；重提界面不提供品牌编辑控件。
3. 校验行业与平台输入。
4. 无条件将版本号增加一次；显式重新提交即代表一个新版本，即使只修改名称。
5. 更新最新工作副本，但保持 `display_name` 和 `brand_id` 不变。
6. 写入新版本快照并创建 Review。
7. 更新品牌行业候选。
8. 提交事务并调度审核任务。

版本、快照、候选或 Review 任一步失败时全部回滚，不留下空版本或已修改但未进入审核的物料。

### 6.3 审核执行

审核任务通过 Review 的 `submission_snapshot_id` 读取输入。法律、平台和舆情分支使用相同的 `raw_text`、`industry` 与 `platforms`。法务详情和营销结果页也显示这一快照，从而保证“被审内容、结果依据和可见内容”一致。

## 7. 品牌行业交互

### 7.1 提交页

- 选择品牌后，将品牌已有行业置顶并标记“品牌常用”。
- 不自动选中品牌全部行业，也不限制用户只能选择品牌行业。
- 用户仍按本次物料实际情况选择一个或多个行业。
- 超出品牌已关联范围的行业不阻断提交，而是形成管理员候选。
- 重提页显示已锁定品牌，不能清空或更换。

### 7.2 管理员品牌档案

- 已关联行业使用与提交页相同的标准多选控件，不再使用自由文本框。
- 管理员可直接添加或移除正式行业。
- “物料记录建议”展示待确认行业、出现次数、最近物料与最近时间。
- “添加到品牌”建立正式关联并将候选标为 `accepted`。
- “忽略”将候选标为 `ignored`；后续同类物料继续累计依据，但不反复作为新候选提示。
- 被忽略候选可以恢复为 `pending`。
- 移除正式品牌行业不改写历史物料；后续物料再次使用时重新形成待确认候选。
- 只有管理员可以接受、忽略、恢复或直接维护品牌行业；其他角色只读。

### 7.3 历史数据迁移

迁移先把品牌旧行业转成正式关联，再扫描所有仍有效且已关联品牌的历史物料。历史物料中未成为正式关联的行业生成 `pending` 候选，并按品牌与行业累计出现次数。迁移必须幂等。

## 8. 法务审核台布局

### 8.1 宽屏

页面采用主次明确的三栏布局：

- 左栏约 220–240px：评分、确认问题数量与问题快速定位。
- 中栏至少 640px，并占据剩余空间：本次提交与完整审查引擎，是主要阅读区域。
- 右栏约 300–320px：历史版本、法务决定和折叠式品牌信息。

不沿用三个栏位等权的视觉关系。中栏使用主表面背景和更完整的卡片层级，两侧使用弱化背景。

### 8.2 中栏内容

1. 顶部提交卡显示快照名称、版本、行业、平台和完整文案。
2. “审查引擎·执行报告”标题居中。
3. 先显示法律与舆情双轴摘要。
4. 法律/平台报告默认展开，包含四层结果、核验事项、引用和建议。
5. 舆情报告默认展开，包含状态、风险等级、解释、风险议题、受影响群体、传播诱因、原文证据、相似事件及建议。
6. 左栏问题可定位到中栏对应证据。

法律使用克制的蓝色和橙色语义；舆情使用紫色语义；失败和人工复核统一使用琥珀色。颜色仅用于辅助层级，所有状态必须同时有文字。

所有新增背景、边框、正文、次要文字、标签和状态色优先复用现有主题变量，并为明暗模式分别定义协调层级。暗色模式不得简单反相或沿用浅色固定色值；两种模式都要保证正文、辅助信息、禁用状态和风险提示具有足够对比度，且法律、舆情与人工复核的语义保持一致。

### 8.3 右栏内容

- 历史版本置顶，每条显示版本、提交时间和法务状态。
- 法务决定保持清晰可见。
- 品牌记忆默认显示摘要，可展开查看。
- 点击历史版本打开抽屉，展示该版本提交快照、法律报告、舆情报告和法务结论。

### 8.4 响应式

- 超宽屏使用完整三栏。
- 中等屏幕使用“中栏主内容 + 右侧操作栏”，左栏压缩为中栏顶部风险摘要。
- 手机和平板使用单栏，顺序为：本次内容、审查引擎、风险导航、版本与决定。
- 中栏内容始终优先出现并获得最大可用宽度。

## 9. 舆情审核与法务决定

法务详情复用营销结果页的舆情报告组件，避免字段映射和文案分叉。

- `pending` 或 `running`：禁用法务决定，并说明舆情仍在处理。
- `succeeded`：允许提交决定。
- `failed` 或 `unavailable`：显示“待人工复核”，法务必须勾选“已完成人工舆情复核”后才能提交决定。
- 舆情风险等级用于辅助判断，不自动改变法律合规分，也不直接替代法务决定。
- 后端决定接口重复校验模块状态和人工复核确认，不能只依赖前端禁用按钮。

法务决定请求增加人工舆情复核确认字段。仅在舆情失败或不可用时要求为真；其他状态下忽略该字段。

## 10. 接口调整

### 10.1 物料与审核

- Material 输出增加 `display_name`。
- Review 输出增加 `submission` 快照。
- 新增原子重提接口，返回新 Review。
- 旧的通用 Material 更新接口不再承担重新提交职责，也不得在退回重提场景修改品牌。
- 队列的 `material_name` 改为稳定 `display_name`，其他版本化展示字段来自最新 Review 快照。

### 10.2 品牌与行业

- Brand 输出增加 `industries: string[]`。
- 品牌资料输出增加行业候选与统计。
- 新增行业目录读取接口。
- 新增管理员接受、忽略和恢复品牌行业候选的接口。
- 品牌更新接口以标准行业列表替代自由文本行业。

### 10.3 法务决定

- 请求增加人工舆情复核确认字段。
- 后端拒绝舆情仍处于等待或运行状态的决定请求。
- 舆情失败或不可用且未确认人工复核时，后端拒绝决定请求。

## 11. 错误处理与权限

- 非退回状态重提返回业务冲突错误。
- 重提时品牌不一致返回明确的品牌锁定错误。
- 品牌已归档或删除时，历史物料仍可读，但不得新建首次品牌关联；既有物料重提保持原品牌并由管理员先恢复品牌后继续。
- 非标准行业输入返回校验错误。
- 行业候选管理接口只允许管理员调用。
- 历史快照缺失时展示兼容提示，不回退到当前 Material 内容。
- 舆情失败不会抹除已完成的法律结果，但会触发人工复核门槛。

## 12. 测试设计

### 12.1 后端

1. 首次提交同时创建稳定标题、快照和品牌行业候选。
2. 每次显式重提版本只增加一次，包括只修改名称的情况。
3. 重提事务失败时 Material、版本、快照和 Review 全部回滚。
4. 重提不能清空或更换品牌，直接 API 请求同样被拒绝。
5. 法律和舆情引擎均读取快照；创建审核后修改工作副本不影响执行输入。
6. Review 与版本接口返回正确快照，旧快照缺失时不串用当前内容。
7. 品牌行业候选幂等累计，接受、忽略、恢复和移除后的再候选行为正确。
8. 历史数据迁移在重复执行时不重复创建关联或候选。
9. 舆情等待、运行、成功、失败和不可用状态对应正确的决定权限。
10. 管理员与非管理员的品牌行业管理权限正确。

### 12.2 前端

1. 工作台和队列显示稳定首次标题。
2. 当前与历史详情显示各自快照名称、文案、行业和平台。
3. 重提页正确显示锁定品牌，不出现品牌清空路径。
4. 品牌已有行业置顶但不自动选择，新行业可正常提交。
5. 品牌档案使用标准多选，并正确处理候选状态。
6. 法务中栏完整显示法律、平台和舆情报告。
7. 舆情状态正确控制决定按钮和人工复核确认。
8. 超宽屏、中屏、手机、长文案以及明暗双模式下布局可用，主题色调协调且信息对比度清晰。
9. 前端类型检查与生产构建通过。

### 12.3 回归

- 现有登录、资料库、平台规则、回收站、管理员知识库和审核测试继续通过。
- SQLite 本地开发与 PostgreSQL 部署迁移均验证升级路径。
- 0.6.x 与 0.7.0 历史审核继续可读。

## 15. 验收标准

1. 第二次及以后提交不会造成列表标题、详情文案和审核结果串版。
2. 列表始终使用首次稳定标题；详情按审核版本显示提交名称与文案。
3. 重提无法丢失、更换或清空品牌，后续审核持续进入同一品牌档案。
4. 品牌支持多个标准行业，管理员可审核由历史和新物料产生的行业候选。
5. 法务审核台中栏是明确主内容区，审查引擎居中且完整展示舆情信息。
6. 舆情未完成时不能提交决定；失败时完成显式人工复核后可以提交。
7. 历史版本可查看当次提交、法律报告、舆情报告和法务结论。
8. 所有新增接口有后端权限与状态校验，不能依赖前端约束。
9. 数据迁移不丢失现有品牌、物料、审核和版本记录。
10. 后端完整测试、前端类型检查和生产构建通过。
11. 应用版本统一为 0.7.1，发布说明完整，README 主要展示思路保持不变。
12. 新增页面元素在明暗双模式下色调协调、语义一致，无低对比度文字、突兀固定底色或不可辨识禁用状态。
