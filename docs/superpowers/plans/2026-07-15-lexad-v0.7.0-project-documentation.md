# LexAd v0.7.0 项目文档体系实施计划

> 依据：`docs/superpowers/specs/2026-07-15-lexad-v0.7.0-project-documentation-design.md`

## 目标

把现有按版本堆叠、部分过期的说明文档，重构为兼顾项目评估、企业应用和工程维护的 v0.7.0 文档体系。重点提升整体叙事、企业效率价值和技术可信度，不以具体实际案例作为主线，也不修改审查业务逻辑。

## 实施步骤

### 1. 建立当前事实清单

- 核对版本号、技术栈、启动方式、数据库模式和模型配置。
- 核对 v0.7.0 审查流水线、结果状态、资料库与人工复核行为。
- 核对当前 API 路由、管理员能力、回收站与安全约束。
- 记录可验证测试命令和当前通过数量。

### 2. 收敛历史文档

- 建立 `docs/archive/releases/` 并整合主要版本演进摘要。
- 移动 v0.5.x 至 v0.6.x 发布说明，保留版本更新事实。
- 把旧指南和迭代报告中的有效内容并入当前指南与版本演进后清理重复文件。
- 为保留的归档文档增加统一历史提示和当前文档入口。
- 清理已被大版本主设计吸收的单点过程文档，保留每个大版本的主设计和关键实施计划。
- 更新所有受影响链接。

### 3. 建立工程事实文档

- 新建 `docs/architecture/review-methodology.md`。
- 将 `docs/technical-reference.md` 重构到 `docs/architecture/technical-reference.md`。
- 将 `docs/local-development.md` 更新到 `docs/guides/local-development.md`。
- 编写 `docs/guides/user-guide.md` 与 `docs/guides/admin-guide.md`。

### 4. 建立发布与项目文档

- 编写 `docs/releases/v0.7.0-release-notes.md`。
- 编写 `docs/project/project-overview.md`。
- 编写 `docs/project/demo-evaluation-guide.md`。
- 使用可维护的 Mermaid 图表达审查机制和企业协作流程。

### 5. 重写统一入口

- 重写根 `README.md`，突出企业问题、创新机制、实际作用与可信边界。
- 重写 `docs/README.md`，按项目评估人员、用户、管理员和开发者提供阅读路径。
- 避免在入口文档重复技术参考的全部细节。

### 6. 验证与修订

- 执行后端完整测试和前端生产构建。
- 检查 Markdown 链接、版本号、过期架构描述和英文风险等级标题。
- 检查文档中的命令、环境变量、接口路径和测试数量。
- 检查敏感信息、夸大宣传、占位符和文档间冲突。
- 执行 `git diff --check` 并复核最终目录。

## 完成定义

- 根 README 能快速说明作品的企业价值、核心创新和使用入口。
- 项目概览、用户、管理、开发和历史材料边界清楚。
- 审查方法论与 v0.7.0 实现一致，关键词与硬规则不再被描述为直接结论。
- 当前文档不存在 v0.6.x 现状表述、固定权重融合或本地关键词直接展示逻辑。
- 所有关键链接和命令有效，验证结果真实可追溯。
