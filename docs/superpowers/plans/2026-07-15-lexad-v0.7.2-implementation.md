# LexAd v0.7.2 审核进度、双评分与导航一致性实施计划

日期：2026-07-15

依据：[v0.7.2 审核进度、双评分与导航一致性设计](../specs/2026-07-15-lexad-v0.7.2-review-flow-score-navigation-design.md)

## 目标

以可迁移、可测试、可回退的方式完成评分模型重构，并用共享组件统一真实审核进度、带风险通过确认和详情返回逻辑。活动业务代码与新 API 不保留旧评分字段，发布前执行契约、遗留引用和用户流程三轮审查。

## 任务 1：先建立测试护栏

涉及文件：

- `backend/app/tests/test_v072_review_contract.py`
- `backend/app/tests/test_v072_score_migration.py`
- `frontend/package.json`
- `frontend/package-lock.json`
- `frontend/vite.config.ts`
- `frontend/src/**/*.spec.ts`

步骤：

1. 为现有法规评分、舆情结果、审核队列和历史版本写特征测试，固定迁移前有效行为；
2. 新增 v0.7.2 契约测试，明确只允许 `legal_compliance_score` 和 `public_opinion_safety_score`；
3. 建立最小 Vitest + Vue Test Utils 环境，先为待提取纯函数写失败测试；
4. 添加审核阶段组合、风险提取、返回来源和评分展示语义测试；
5. 保持测试数据独立，不依赖开发数据库。

步骤：

1. 重命名 `reviews.ai_risk_score` 为 `legal_compliance_score`；
2. 新增可空 `public_opinion_safety_score` 和两个 0–100 范围约束；
3. 可靠历史舆情按 `100 - 旧值` 回填，人工复核、失败、不可用和无效数据回填空；
4. 清理历史 `public_opinion_result` 中的数值评分键，保留证据、说明和建议；
5. 实现可回退 downgrade，并兼容 SQLite batch migration 与 PostgreSQL；
6. 更新 ORM 字段并运行升级、降级、再升级测试；
7. 对迁移前后审核总数、法规分和舆情证据做一致性断言。

## 任务 7：三轮审查与发布验证

### 第一轮：契约审查

逐项比对 Alembic、ORM、Pydantic、服务返回、TypeScript、组件 props 和文档，确保字段名、可空性、范围与方向完全一致。

### 第二轮：遗留引用审查

全仓扫描旧字段及旧评分文案。活动后端、前端、脚本和当前文档必须零残留；只允许一次性迁移脚本、历史迁移和历史发布文档出现旧名。

### 第三轮：用户流程审查

按营销、法务、管理员角色走查新提交、重新提交、处理轮询、结果查看、历史查看、工作台返回、法务台返回、浏览器后退、带风险通过、提交失败和人工复核。

最终命令：

1. Alembic 全新升级到 head；
2. Alembic v0.7.1 → v0.7.2 → v0.7.1 → v0.7.2 循环；
3. 后端完整 `pytest`；
4. 前端 `npm test`；
5. 前端 `npm run build`；
6. 全仓遗留字段扫描；
7. Markdown 链接检查；
8. `git diff --check` 与工作区范围审查。

## 完成标准

- 数据迁移可升级、可降级、历史证据不丢失；
- 两个正式分数均为越高越好，人工复核不伪造 0 分；
- 活动代码和新 API 不含旧评分字段；
- 审核进度完全来源于真实状态；
- 带风险通过必须二次确认且不能重复提交；
- 所有物料详情状态都有来源一致的返回入口，浏览器后退保持正常；
- 浅色、深色、桌面、移动端和减少动画设置可用；
- 三轮审查、完整测试、构建和迁移验证全部通过；
- v0.7.2 版本与发布文档一致。
