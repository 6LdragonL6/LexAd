# LexAd v0.4.2 / v0.5.0 实施计划

日期：2026-07-07

来源设计：`docs/superpowers/specs/2026-07-07-lexad-v0.4.2-v0.5.0-design.md`

## 1. 交付边界

### v0.4.2

v0.4.2 交付可运行框架和完整业务闭环：

- 优化物料提交页为单页渐进式界面。
- 新增管理员资料中心。
- 新增舆情事件自然语言录入、AI 整理、人工复核、发布、归档、恢复。
- 新增舆情 JSON 批量导入能力，但放在“更多工具”中，不作为普通管理员默认入口。
- 新增平台规则版本管理、差异展示、生效、回滚。
- 将 L4 平台规则接入正式审核。
- 在统一审核任务中并行输出法律风险与舆情风险双轴结果。
- 支持空舆情库、模型失败、单模块失败的明确降级与重试。
- 补充权限、审计、测试和管理员操作说明。

### v0.5.0

v0.5.0 等用户补充真实舆情 JSON 后再做：

- 导入真实舆情事件。
- 校验事件结构和标签质量。
- 校准触发规则、相似度阈值、风险等级和模型提示词。
- 固化舆情回归测试集。
- 完成最终发布验证并升级版本号。

## 2. 总体顺序

按依赖关系分 9 个阶段推进。每个阶段都应能独立提交和验证。

1. 数据模型、迁移、枚举与权限基础。
2. 管理员资料中心后端 API。
3. 舆情导入、AI 整理任务与审计。
4. 平台规则版本管理与 L4 引擎。
5. 舆情混合引擎与统一审核聚合。
6. 管理员资料中心前端。
7. 物料提交页和结果页改造。
8. 自动化测试、文档和 v0.4.2 发布验证。
9. 真实 JSON 导入、校准和 v0.5.0 发布验证。

## 3. 阶段 1：数据模型、迁移、枚举与权限基础

目标：先建立稳定的数据边界，避免后续 API 和引擎直接依赖临时结构。

### 3.1 后端模型

新增或扩展后端模型：

- `PublicOpinionEvent`
  - 保存事件主记录、状态、原始自然语言输入、创建人、更新时间。
- `PublicOpinionEventVersion`
  - 保存结构化内容、系统生成字段、人工修改记录、模型版本、置信度。
- `PublicOpinionLibraryVersion`
  - 保存一次发布形成的舆情资料库快照。
- `PlatformRuleSet`
  - 保存平台维度的规则集合。
- `PlatformRuleVersion`
  - 保存规则正文、结构化规则项、生效区间、状态、来源和版本号。
- `KnowledgeImportJob`
  - 保存导入任务、预检结果、错误摘要、处理进度和状态。
- `KnowledgeAuditLog`
  - 保存发布、归档、恢复、启用、回滚、导入确认等管理员写操作。
- 审核任务相关模型扩展
  - 保存本次审核使用的法律库、行业库、平台规则、舆情库版本 ID。
  - 保存法律分析与舆情分析各自的状态、错误、重试次数和完成时间。

### 3.2 枚举与状态机

新增状态枚举：

- 舆情事件状态：`draft`、`ai_processing`、`pending_review`、`published`、`archived`。
- 平台规则状态：`draft`、`pending_effective`、`active`、`expired`。
- 导入任务状态：`uploaded`、`validated`、`validation_failed`、`importing`、`completed`、`failed`、`cancelled`。
- 审核模块状态：`pending`、`running`、`succeeded`、`failed`、`unavailable`。

状态机约束：

- 已发布舆情事件不能物理删除，只能归档。
- 未发布草稿可以删除。
- 一个平台同一时间只能有一个当前生效版本。
- 审核任务启动后固定资料版本快照，后续规则或舆情库更新不影响历史结果。

### 3.3 权限基础

后端必须校验管理员权限：

- 所有资料中心写 API 必须要求管理员角色。
- 普通市场人员和法务人员不能访问资料中心写 API。
- 前端隐藏入口只是辅助，不能替代后端权限。

### 3.4 验收

- 数据迁移可执行、可回滚到开发环境上一版本。
- 单元测试覆盖状态机合法/非法转换。
- 单元测试覆盖非管理员访问资料中心写 API 被拒绝。
- 审核结果模型能保存双模块状态和知识版本快照。

## 4. 阶段 2：管理员资料中心后端 API

目标：为前端资料中心提供稳定 API，不依赖真实舆情 JSON 才能开发。

### 4.1 舆情事件 API

新增管理员 API：

- 列表：按状态、行业、平台、关键词、更新时间筛选。
- 详情：返回原始输入、结构化版本、系统生成字段、人工修改记录。
- 新建草稿：支持自然语言字段，不要求 JSON。
- 更新草稿或待复核事件。
- 发起 AI 整理。
- 发布事件。
- 归档事件。
- 恢复事件。
- 删除未发布草稿。

关键约束：

- 发布前必须通过必填字段校验。
- 发布后形成资料库版本或更新当前资料库快照。
- 归档后新审核不再检索该事件，历史审核仍可查看快照引用。

### 4.2 平台规则 API

新增管理员 API：

- 平台列表与当前生效版本。
- 新建或导入规则版本。
- 查看规则版本详情。
- 查看规则项差异。
- 设置生效时间。
- 启用版本。
- 回滚到指定历史版本。

关键约束：

- 启用新版本必须记录审计日志。
- 回滚是切换生效版本，不删除新版本。
- 差异结果至少区分新增、修改、删除。

### 4.3 导入记录与审计 API

新增管理员 API：

- 舆情 JSON 模板下载。
- 舆情 JSON 预检。
- 确认导入。
- 导入任务详情与错误列表。
- 审计日志列表。

### 4.4 验收

- API schema 明确区分普通自然语言录入和 JSON 批量导入。
- 导入预检不会发布数据。
- 失败导入不改变已发布资料库。
- 所有写操作有审计日志。

## 5. 阶段 3：舆情导入、AI 整理任务与审计

目标：让不懂计算机的管理员能录入案例，同时保留后续 JSON 批量导入能力。

### 5.1 自然语言录入

默认流程：

1. 管理员填写“这件事发生了什么”。
2. 管理员填写“造成了什么后果”。
3. 管理员可选填写来源、链接、大致时间。
4. 点击“AI 帮我整理事件”。
5. 系统生成结构化草稿。
6. 管理员在自然语言表单中复核和修改。
7. 管理员发布。

实现要点：

- 结构化结果必须通过后端 schema 校验。
- 模型不得凭空补充事实；不确定字段标记为不确定。
- AI 整理失败时保留原始输入，并允许重试。
- 草稿自动保存。

### 5.2 JSON 批量导入

JSON 导入放在“更多工具”：

- 提供下载模板。
- 上传后先预检。
- 按事件编号和字段返回错误。
- 合格事件保存为草稿。
- 不合格事件不保存。
- `external_id` 重复时由管理员选择新增、更新或跳过。
- 批量 AI 整理仍需人工复核后发布。

### 5.3 审计

记录以下行为：

- 新建、更新、删除草稿。
- AI 整理发起、成功、失败。
- JSON 预检和确认导入。
- 发布、归档、恢复。
- 平台规则启用和回滚。

### 5.4 验收

- 管理员不接触 JSON 也能完成案例发布。
- JSON 导入错误信息能被非技术人员理解。
- 中断或失败导入不会产生半发布数据。
- 审计日志能追溯关键操作人、时间、对象和结果。

## 6. 阶段 4：平台规则版本管理与 L4 引擎

目标：解决“平台规则不时更新，工具是否同步”的核心问题。

### 6.1 平台规则版本管理

实现：

- 平台规则版本存储。
- 当前生效版本查询。
- 生效时间与失效时间。
- 规则项级差异。
- 回滚。

规则版本来源：

- v0.4.2 不自动抓取平台规则。
- 管理员导入或录入规则后启用。
- 系统展示规则来源、版本、更新时间和生效时间。

### 6.2 L4 平台审核引擎

将现有审核流水线中的 L4 占位逻辑替换为正式执行：

- 根据投放平台查找当前生效版本。
- 执行确定性规则命中。
- 必要时调用同一模型服务做平台规则语义分析。
- 输出平台规则风险项、依据、建议和版本 ID。

### 6.3 验收

- 没有生效平台规则时输出明确不可用状态，不假装通过。
- 平台规则命中可追溯到规则版本。
- 回滚后新审核使用回滚版本，历史审核保持原快照。
- L4 结果进入法律风险轴，但保留平台规则依据标识。

## 7. 阶段 5：舆情混合引擎与统一审核聚合

目标：在同一个审核任务中并行运行法律/平台规则审核与舆情风险分析。

### 7.1 舆情引擎

舆情引擎分三层：

1. 确定性触发器
   - 使用人工复核后的风险议题、触发表达、敏感群体、平台标签匹配。
2. 相似事件检索
   - 只检索已发布且未归档事件。
   - 行业和平台作为过滤或加权条件。
   - 返回事件 ID、标题、相似度、历史后果和资料库版本。
3. 模型解释
   - 使用同一模型服务与统一 API 配置。
   - 输入限制为物料、触发器命中、相似事件和后果。
   - 输出风险等级、议题、群体、传播诱因、相似案例、建议、置信度。

### 7.2 空库与失败处理

必须实现：

- 舆情库为空：返回 `knowledge_base_empty`，界面显示“资料库待补充”，不输出低风险。
- 模型不可用：保留规则命中和相似案例，标记 AI 解释暂不可用。
- 舆情模块失败：法律结果正常展示，可单独重试舆情模块。
- 法律模块失败：舆情结果正常展示，可单独重试法律模块。

### 7.3 统一审核聚合

前端仍调用统一审核入口：

- 后端创建一个审核任务。
- 固定知识版本快照。
- 法律/平台规则分支与舆情分支并行运行。
- 聚合结果保存到同一个审核结果。
- 结果页按双风险轴展示。

### 7.4 验收

- 一次提交产生法律和舆情两套独立状态。
- 法律低风险但舆情高风险时，系统建议暂停发布并由品牌负责人复核。
- 单模块失败不覆盖另一模块结果。
- 正常审核耗时不因新增舆情模块简单翻倍。

## 8. 阶段 6：管理员资料中心前端

目标：让非技术管理员能维护舆情和平台规则。

### 8.1 路由和导航

新增管理员可见入口：

- 资料中心。
- 舆情事件。
- 平台规则。
- 导入记录。
- 审计日志。

非管理员：

- 不显示入口。
- 直接访问路由也被拦截。

### 8.2 舆情事件界面

页面能力：

- 列表、筛选、状态标签。
- “新增舆情案例”默认进入自然语言表单。
- “AI 帮我整理事件”按钮。
- AI 整理结果复核页。
- 不确定字段高亮。
- 发布、归档、恢复、删除草稿。
- 已发布事件详情展示版本和引用状态。

文案要求：

- 不把 JSON、schema、字段名作为默认说明。
- 错误提示使用普通语言。
- 高级导入能力放到“更多工具”。

### 8.3 JSON 导入界面

页面能力：

- 下载模板和示例。
- 上传 JSON。
- 预检结果。
- 按条目展示错误。
- 重复 `external_id` 处理选择。
- 确认导入为草稿。
- 导入任务进度和结果。

### 8.4 平台规则界面

页面能力：

- 平台列表和当前生效版本。
- 新建或导入版本。
- 差异展示。
- 设置生效时间。
- 启用和回滚。
- 版本历史。

### 8.5 验收

- 普通管理员能按自然语言流程完成一个案例发布。
- JSON 导入入口不干扰日常录入。
- 平台规则更新流程能看见“当前版本、目标版本、差异、影响平台”。
- 所有危险操作有二次确认。

## 9. 阶段 7：物料提交页和结果页改造

目标：解决节点三“提交物料是否繁琐”和新增双轴结果展示。

### 9.1 提交页

改为单页渐进式：

首屏只保留：

- 物料内容：上传文件或粘贴文本。
- 行业。
- 投放平台。
- 开始风险审查按钮。

折叠到“更多设置”：

- 物料名称。
- 物料类型。
- 优先级。
- 截止时间。

保留：

- 文件提取预览。
- 文本编辑。
- 异步任务轮询。

### 9.2 结果页

新增双风险轴：

- 法律合规风险。
- 舆情风险。

结果页展示：

- 各自状态：完成、失败、不可用、重试中。
- 法律法规、行业规则、平台规则依据。
- 舆情风险等级、相似事件、历史后果、传播诱因、建议。
- 本次审核使用的知识版本。
- 单模块重试入口。

### 9.3 验收

- 首屏字段数量明显减少。
- 新用户能在不展开“更多设置”的情况下完成提交。
- 结果页能解释为什么舆情为空库或不可用。
- 重试单个模块不会重复创建物料。

### 10.5 v0.4.2 发布条件

- L4 平台规则不再是占位实现。
- 普通管理员可完成自然语言录入闭环。
- 统一审核 API 保持兼容。
- 自动化测试通过。

## 11. 阶段 9：真实 JSON 导入、校准和 v0.5.0 发布验证

目标：等用户补充真实舆情 JSON 后，把框架变成可正式判断的舆情风险能力。

### 11.1 数据接收

接收文件时检查：

- 是否符合 `schema_version: "1.0"`。
- `events` 是否为数组。
- 必填字段是否完整。
- 事件是否有可追溯来源。
- 后果描述是否足以判断严重度。
- 是否存在明显重复事件。

### 11.2 导入与复核

流程：

1. 使用 v0.4.2 JSON 预检。
2. 修复格式错误。
3. 导入为草稿。
4. 批量 AI 整理。
5. 抽检系统生成字段。
6. 管理员发布。
7. 形成舆情资料库版本。

### 11.3 校准

校准内容：

- 风险议题归一化。
- 触发表达归一化。
- 敏感群体标签。
- 行业标签。
- 平台标签。
- 相似度阈值。
- 风险等级映射。
- 模型提示词输出稳定性。

### 11.4 回归测试集

从真实事件中抽取：

- 高风险样例。
- 中风险样例。
- 低风险或不确定样例。
- 法律低风险但舆情高风险样例。
- 平台规则命中但舆情不明显样例。
- 舆情触发但缺少足够案例依据样例。

### 11.5 v0.5.0 发布条件

满足以下条件才标记 v0.5.0：

- 真实 JSON 全部导入或明确列出无法导入原因。
- 风险标签和后果等级完成抽检。
- 回归测试集建立并通过。
- 阈值和提示词经样例校准。
- 结果页能展示真实相似案例。
- 舆情风险输出不依赖编造案例。

## 12. 建议提交拆分

建议按以下提交粒度推进：

1. `backend: add knowledge data models and migrations`
2. `backend: add admin knowledge APIs and permissions`
3. `backend: add public opinion import and structuring jobs`
4. `backend: implement platform rule versions and L4 review`
5. `backend: add public opinion review engine and aggregation`
6. `frontend: add admin knowledge center`

8. `test: cover knowledge center and dual-risk review`

每个提交应尽量保持可运行。数据库迁移、模型、API、前端和测试不要混成一个无法验证的大提交。

## 13. 风险与控制

### 13.1 最大工程风险：范围过大

控制方式：

- v0.4.2 先交付框架和空库降级，不等待真实数据。
- v0.5.0 再做数据导入和阈值校准。
- 管理员自然语言录入优先，JSON 导入作为高级工具。

### 13.2 最大产品风险：舆情结论被误解为确定预测

控制方式：

- 舆情风险与法律风险分轴展示。
- 输出置信度和不确定性。
- 空库或依据不足时明确显示不可判断。
- 高风险建议为“人工复核”，不是自动阻断。

### 13.3 最大技术风险：模型输出不稳定

控制方式：

- 所有模型输出必须过 schema 校验。
- 确定性触发和相似事件检索不依赖模型。
- 模型失败时保留基础结果。
- v0.5.0 用真实案例建立回归测试。

### 13.4 最大运维风险：平台规则更新破坏历史审核

控制方式：

- 审核启动时固定版本快照。
- 平台规则回滚不删除版本。
- 历史结果保存版本 ID。
- 启用和回滚都有审计日志。
