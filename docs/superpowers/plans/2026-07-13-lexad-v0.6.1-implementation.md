# LexAd v0.6.1 体验与审查质量加固实施计划

来源设计：[LexAd v0.6.1 体验与审查质量加固设计](../specs/2026-07-13-lexad-v0.6.1-design.md)。

## 实施约束

- 在现有 `Material → Review → Result` 流程上增量修改，不重建认证、品牌库或审批状态机。
- 优先复用现有 `PlatformRuleSet`、`PlatformRuleVersion`、`PublicOpinionEvent`、`PublicOpinionLibraryVersion`、`KnowledgeImportJob` 和提交快照，不为可由现有 JSON/元数据承载的信息新增表。
- 内置基线只补充缺失平台或空资料库；管理员已有启用版本永不被覆盖。
- 混合检索必须有不依赖模型服务的确定性回退，不以向量数据库作为本版前提。
- 历史内容只读，旧记录缺少快照时明确降级，不用当前内容补算。
- 每阶段完成后执行与风险相称的测试；不回退用户的无关修改。

## 阶段 4：角色化统计与合规提醒池

主要文件：

- `data/compliance_tips.json`（新增）

- `frontend/src/pages/HomePage.vue`

工作项：

1. 建立结构化提醒数据，字段固定为 `id`、`title`、`body`、`source`、`source_path`、`roles`、`tags` 和 `enabled`。
2. 从现有法规和平台资料整理完整短句，不在运行时随机截断原文；至少覆盖市场、法务和管理员三个角色。
3. 增加纯函数：按角色筛选、排除上次编号、随机选择、资源异常时返回内置兜底提醒。
4. 工作台每次加载执行提醒选择，并通过 `localStorage` 记录上次编号；候选只有一条时允许重复。
5. 将统计卡片改为角色配置：市场端为待处理/需修改/已通过，法务端为待审核/已通过/已退回，管理员为全局待审核/已通过/已退回。
6. 明确每个卡片的状态集合，避免把法务“已退回”解释成法务需要修改。
7. 根据现有权限隐藏无权使用的快捷入口。

验收：法务账号不显示“需修改”；三个角色统计口径符合设计；重复加载工作台时提醒轮换且资源异常不阻塞页面。

## 阶段 5：实现内置知识基线解析与幂等接入

主要文件：

- `backend/app/services/knowledge_bootstrap.py`（新增）
- `backend/app/services/knowledge_parsers.py`（新增）
- `backend/app/db/seed.py`
- `backend/app/models/knowledge.py`
- `backend/app/tests/test_knowledge_bootstrap.py`（新增）
- `backend/app/tests/test_parsers.py`

工作项：

1. 将平台目录、L3 案例目录和结构化 JSON 的读取、文本切分、规则生成分别封装为无数据库依赖的解析函数。
2. 平台规则解析按标题、段落和中文句子切分，优先抽取包含“不得、禁止、严禁、应当、违规、虚假”等规范性提示的完整条款；每条规则保留来源相对路径和来源哈希。
3. 生成稳定规则标识，结构至少包含 `rule_id`、`title`、`text`、`keywords`、`risk_level`、`source_path` 和 `source_hash`。
4. L3 案例使用稳定 `external_id=builtin:<relative-path>`，原文和解析字段写入现有事件及版本模型；发布后生成新的舆情资料库版本。
5. 平台基线使用稳定版本标签和现有 `source_name/raw_text/structured_rules` 字段；来源哈希及导入摘要写入现有 `KnowledgeImportJob.options/error_summary`。
6. 选择种子管理员作为内置导入操作者。用户种子完成后执行基线检查，使已有 v0.6.0 数据库也能补齐空知识表。
7. 如果平台已有启用版本则跳过；同一来源哈希已经完成则跳过；单个文件失败只记录该来源并继续其他来源。
8. 输出发现文件、生成条目、启用平台、跳过平台和失败来源的结构化摘要。

验收：空临时 SQLite 初始化后存在已启用拼多多基线和非空舆情资料库；连续运行两次不增加重复版本；预置管理员版本不会被改变。

## 阶段 6：平台别名与混合文本相似性

主要文件：

- `backend/app/engine/text_similarity.py`（新增）
- `backend/app/engine/layer4_platform.py`
- `backend/app/engine/public_opinion.py`
- `backend/app/schemas/review.py`
- `backend/app/tests/test_platform_review.py`
- `backend/app/tests/test_public_opinion_review.py`

工作项：

1. 将平台规范化和别名表提取为可测试逻辑，保证 `拼多多/pdd/pinduoduo` 映射到同一候选集合。
2. 实现不依赖外部服务的文本规范化、字符短语或中文词元集合、Jaccard/余弦式相似分和阈值过滤。
3. L4 先保留现有关键词精确命中，再对未精确命中的结构化规则执行相似性召回；精确命中优先、候选去重、Top-K 有上限。
4. 舆情案例把行业、平台、触发模式和案例文本相似性合并为可解释分数，不再只依赖短词原样包含。
5. 为命中项增加向后兼容的可选字段：`matched_text`、`match_method`、`score`、`source_id` 和 `suggestion`。
6. 模型服务仅接收经过阈值过滤的候选用于排序和说明；模型错误继续走确定性结果。
7. 测试精确命中、近义改写、低相关噪声、空资料、无启用版本和模型不可用。

验收：拼多多典型规则能够命中；近义改写得到高于阈值的候选；无关文本不会因平台或行业相同而产生高相似命中。

## 最终验证命令

在仓库根目录执行：

1. `cd backend; python -m pytest app/tests -q`

## 完成判定

只有在三级侧栏和所有主要页面通过连续窗口验收、拼多多 L4 真实命中、混合检索在模型不可用时仍可产出来源明确的结果、命中优先报告正确折叠无命中模块、法务工作台移除“需修改”、历史第一轮报告可回看、提醒轮换正常，且版本与文档统一为 0.6.1 时，实施才可结束。
