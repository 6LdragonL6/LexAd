# LexAd v0.7.1 管理员指南

管理员负责 AI 服务、舆情案例、平台规则、回收站和审计记录。资料质量直接影响审查质量；系统不会把缺失或过期资料默认为低风险。

## 1. 管理职责

- 保证 DeepSeek API 配置可用且密钥安全；
- 维护有来源、可核验的舆情事件；
- 维护平台规则的来源、版本、有效期和结构化内容；
- 处理回收站恢复与永久清理；
- 通过导入记录和操作日志追踪管理行为；
- 维护品牌的标准行业，并处理物料产生的行业候选；
- 定期复核资料覆盖、有效期和异常人工复核原因。

## 2. AI 配置

进入“系统管理 → AI 配置”：

1. 输入 DeepSeek API Key；
2. 点击验证并保存；
3. 确认页面显示固定网关和模型；
4. 如需轮换，直接验证并保存新 Key；
5. 如需停用数据库配置，执行清除。

当前网关固定使用：

- Base URL：`https://api.deepseek.com`
- 模型：`deepseek-v4-flash`

页面和接口只显示密钥掩码，无法读取已保存明文。数据库配置优先于服务器环境变量；清除后如果仍存在 `DEEPSEEK_API_KEY`，系统会回退到环境变量。

连接测试只验证最小请求是否成功，不替代法律、平台和舆情业务回归测试。

### 密钥安全

- 生产环境必须设置随机、稳定的 `SECRET_KEY`；
- 使用默认 `SECRET_KEY` 时，生产环境拒绝保存 API Key；
- `SECRET_KEY` 用于派生加密密钥，改变后旧配置无法解密；
- 轮换 `SECRET_KEY` 前先清除 AI 配置，变更后重新保存；
- 不在截图、日志、对外材料或仓库中披露真实 Key。

## 3. 舆情案例管理

### 单条录入

进入“资料中心 → 舆情案例”：

1. 填写标题、事件材料、后果和来源信息；
2. 保存为草稿；
3. 使用“AI 整理”生成结构化版本；
4. 人工检查行业、平台、风险议题、触发机制、受影响群体、传播诱因和后果；
5. 确认来源与表述后发布。

只有已发布事件进入最新舆情资料库版本并参与候选召回。草稿、待复核和已归档事件不应作为当前审查依据。

### 状态管理

| 状态 | 作用 |
| --- | --- |
| 草稿 | 可继续编辑或删除，不参与审查 |
| AI 整理中 | 正在生成结构化版本 |
| 待复核 | 等待管理员确认，不参与审查 |
| 已发布 | 进入资料库版本，可供 AI 参考 |
| 已归档 | 保留历史，不参与当前审查 |

AI 整理用于提高录入效率，不替代管理员核验。不得把来源不明、后果不清或未经证实的网络传言直接发布为已核验案例。

### JSON 批量导入

推荐流程：

1. 获取系统提供的 `schema_version: "1.0"` 模板；
2. 按模板整理事件数组；
3. 执行预检，查看总数、有效数、错误和重复外部编号；
4. 选择重复项的跳过或更新策略；
5. 确认导入；
6. 按需执行 AI 结构化，再由管理员复核发布。

建议每条事件至少具有标题、事件材料、触发过程、结果和可追溯来源。`external_id` 应保持稳定，用于幂等导入和重复识别。

## 4. 平台规则管理

### 建立规则集

为每个平台建立一个稳定规则集，填写内部平台标识、展示名称和说明。平台别名由系统适配，但规则集名称仍应清楚、唯一。

### 新建版本

每个版本应包含：

- 唯一版本标签；
- 来源名称与链接；
- 发布时间、生效时间和失效时间；
- 原始规则正文；
- 结构化规则、风险类型和建议；
- 与上一版本的差异摘要。

结构化规则是平台 AI 裁决的候选资料。仅保存原始正文但没有结构化规则时，该版本无法提供可靠自动裁决，系统会要求人工复核。

### 启用与回滚

启用前检查来源、有效期、结构化内容和差异。审查只使用当前时点处于启用状态且在有效期内的版本。

新版本出现问题时可以回滚到旧版本。历史审查记录保存当次平台规则版本标识，不会因回滚或新版本启用而被覆盖。

## 5. 品牌行业管理

品牌档案使用与物料提交页相同的标准行业目录，不再依赖自由文本。一个品牌可以包含多个行业。

当关联品牌的物料使用了尚未收录的行业时，系统会累计一个行业候选，并记录出现次数和最近物料。管理员可以：

- **接受**：把行业加入该品牌的正式行业列表；
- **忽略**：保留记录但不加入正式列表；后续再次出现仍会累计次数；
- **恢复**：把已忽略候选恢复为待确认状态。

候选不会阻止提交，也不会自动成为品牌正式行业。管理员应结合真实业务范围和历史物料判断，避免仅因一次误选扩大品牌行业范围。

## 6. 资料质量标准

### 法规与平台资料

- 优先使用官方、现行、可定位的来源；
- 标明标题、版本、生效时间和适用平台或行业；
- 区分强制规定、平台政策、解释材料和案例；
- 过期资料及时停用或归档，不直接覆盖历史版本；
- 结构化规则保留适用条件，避免只抽取孤立关键词。

### 舆情资料

- 有清晰事件经过、触发机制和后果；
- 有可核验来源，避免仅凭转述；
- 能代表可复用的风险机制，而不是堆积相似新闻；
- 不把共同数字、公共短语或单个词当作案例相似依据；
- 发布前复核 AI 生成字段，删除推测和编造内容。

## 7. 人工复核异常排查

当大量任务进入人工复核时，依次检查：

1. AI 配置是否存在、能否通过连接测试；
2. 法规和行业资料目录是否可读；
3. 所选平台是否存在当前生效规则版本；
4. 平台版本是否包含结构化规则；
5. 舆情资料库是否存在已发布版本；
6. 后端日志是否出现认证、限流、超时或结构化解析错误；
7. 模型输出是否因原文证据或依据标识无效而被过滤。

不要通过放宽引用验证或重新展示原始关键词来掩盖资料问题。正确处理方式是修复服务、补齐资料或安排人工审查。

## 8. 回收站

管理员可以把物料、品牌、舆情事件和平台规则集移入回收站：

- 默认保留 15 天；
- 软删除对象不会参与普通列表、统计或新审查；
- 到期后自动永久清理；
- 到期前可以恢复，或经二次确认立即永久清理。

永久清理不可恢复：

- 物料清理会同时删除相关审查和提交快照；
- 品牌清理只解除历史物料关联，不删除物料；
- 舆情事件清理会删除其版本并从资料库版本集合移除；
- 平台规则集清理会删除所属规则版本。

导入记录和审计日志不属于可删除业务对象。

## 9. 审计与日常维护

建议维护节奏：

- 每次规则更新：记录来源、创建新版本、复核差异、再启用；
- 每周：检查人工复核原因、失败请求和临近失效的平台版本；
- 每月：抽查已发布舆情事件的来源与结构化质量；
- 每次模型或提示词调整：执行完整业务回归，而不只做连接测试；
- 每次发布：检查版本号、迁移状态、备份、默认账号和敏感配置。

审计日志应记录谁在何时对哪个对象执行了什么操作，但不保存完整敏感物料或明文 API Key。
