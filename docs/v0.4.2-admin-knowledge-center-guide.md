# LexAd v0.4.2 管理员资料中心使用说明

日期：2026-07-07

## 适用范围

本说明面向不具备计算机基础的管理员。v0.4.2 的目标是让管理员可以持续维护两类资料：

1. 舆情案例资料库：用于识别广告素材可能引发的声誉、传播和公众反感风险。
2. 平台规则资料库：用于同步抖音、小红书、微博等平台规则的版本变化。

v0.4.2 不内置真实舆情案例库。后续补入真实 JSON 数据后，再进入 v0.5.0。

## 入口

登录管理员账号后，打开左上角菜单，进入“资料中心”。

资料中心包含：

- 舆情案例
- 平台规则
- 导入记录
- 操作日志

## 添加单条舆情案例

推荐普通管理员优先使用“单条录入”，不要直接从数据库或代码层面修改资料。

操作流程：

1. 在“舆情案例”区域填写事件标题。
2. 在“舆情文本/事件经过”中粘贴报道、评论摘录、事件经过或内部整理材料。
3. 在“事件后果”中填写品牌声誉、销量、监管、平台、合作关系等影响。
4. 点击“保存草稿”。
5. 点击“AI 整理”，系统会调用统一模型服务，把自然语言材料整理为结构化标签、触发点、传播因素和严重程度提示。
6. 管理员人工复核结果。
7. 确认无误后点击“发布”。

发布后，该案例才会进入舆情资料库，并参与后续广告审核。发布前的草稿不会影响审核结果。

## 批量导入 JSON

批量导入放在“更多工具”中，适合后续一次性补入多条案例。管理员只需要复制 JSON 内容并粘贴，不需要操作数据库。

推荐流程：

1. 点击“下载/查看模板”。
2. 按模板整理 JSON。
3. 复制完整 JSON 到导入框。
4. 点击“预览导入”。
5. 系统会显示总条数、有效条数、错误条数和重复编号。
6. 如果有错误，按提示修改后重新预览。
7. 对重复 `external_id` 选择跳过或更新。
8. 点击“确认导入”。
9. 如勾选“导入后自动整理”，系统会继续调用统一模型服务生成结构化版本。

最小可用 JSON 示例：

```json
{
  "schema_version": "1.0",
  "events": [
    {
      "external_id": "case-001",
      "title": "某品牌广告表达引发争议",
      "industry": ["食品"],
      "platforms": ["微博", "抖音"],
      "occurred_at": "2026-01-15",
      "source_text": "这里填写报道、评论摘录、事件经过或内部整理材料。",
      "sources": [
        {
          "name": "来源名称",
          "url": "https://example.com/article",
          "published_at": "2026-01-16"
        }
      ],
      "event_process": {
        "trigger": "争议由什么内容、表达或场景触发",
        "timeline": [
          {
            "at": "2026-01-15",
            "stage": "发生",
            "description": "事件经过"
          }
        ],
        "brand_response": "企业采取的回应",
        "outcome": "事件最终结果"
      },
      "consequences": {
        "reputation": "品牌声誉影响",
        "business": "销量、合作或市值等影响",
        "regulatory": "约谈、处罚或整改情况",
        "duration_days": 14,
        "severity_hint": "high"
      },
      "notes": "其他补充信息"
    }
  ]
}
```

必填字段：

- `schema_version`：固定填写 `"1.0"`。
- `events`：案例数组。
- `title`：事件标题。
- `source_text`：事件材料。
- `event_process.trigger`：触发原因。
- `event_process.outcome`：事件结果。

建议填写字段：

- `external_id`：外部编号，用于避免重复导入。
- `industry`：行业标签。
- `platforms`：传播平台。
- `sources`：来源名称、链接和发布时间。
- `consequences`：声誉、商业、监管影响。

## 舆情案例状态

- 草稿：可编辑、可删除，不参与审核。
- 待复核：AI 已整理，等待管理员确认。
- 已发布：参与审核。
- 已归档：不再参与审核，但保留历史记录。

已发布案例不建议物理删除。需要停止使用时，使用“归档”。草稿可以直接删除。

## 平台规则维护

平台规则用于解决“平台规则不时更新，审核工具是否同步”的问题。v0.4.2 改为管理员维护版本。

操作流程：

1. 新建平台规则集，例如“抖音”“小红书”“微博”。
2. 新建规则版本，填写版本号、来源名称、来源链接、生效时间和规则正文。
3. 保存后系统会和上一版本比较，展示新增、修改、删除数量。
4. 管理员确认后点击“启用”。
5. 如新版本有问题，可回滚到旧版本。

审核结果会记录当次命中的平台规则版本 ID，便于之后追溯“当时按哪个规则版本审核”。

## 审核结果如何理解

v0.4.2 将结果拆为两条风险轴：

- 法律/广告合规风险：继续使用原有 L1-L5 合规审查体系。
- 舆情风险：使用已发布舆情案例库进行触发词、相似案例和模型解释聚合。

如果舆情资料库为空，结果页会明确提示“资料库待补充”。系统不会伪装成低风险。

## 数据质量要求

舆情案例不是越多越好。优先补充这类材料：

- 有清晰触发点，例如冒犯群体、灾难营销、低俗表达、虚假价值观、地域歧视等。
- 有完整事件经过和后果。
- 有明确来源，便于复核。
- 能代表某类重复出现的传播风险。

管理员发布前应复核 AI 整理结果。模型整理用于提高录入效率，不替代人工判断。
